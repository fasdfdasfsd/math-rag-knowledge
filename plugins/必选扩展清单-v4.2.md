## 一、选型方法论

以"Claude Code 成为通用软件产品全生命周期生产工厂"为目标，从两个维度扫描能力缺口：

```
维度 A：生命周期轴 — 需求→设计→开发→测试→安全→文档→CI/CD→运维→维护 (9 阶段)
维度 B：基础能力域 — 数据库/搜索/浏览/知识/编排/通知/文件/命令/媒体 (9 域)

筛选规则：
  1. v4.1 已覆盖的能力 → 不选
  2. Claude Code 内置能力已覆盖的 → 不选
  3. 项目当前阶段不需要的 → 不选
  4. 剩下的是必选
```

**分析结果**：18 个能力域中，11 个已被 v4.1 + Claude Code 内置能力覆盖。**真实缺口 7 个**，全部由第三方 MCP Server 填补。

## 三、逐项详解

---

### 1. E2B — 安全沙箱执行

| 属性 | 内容 |
|------|------|
| **仓库** | [github.com/e2b-dev/mcp-server](https://github.com/e2b-dev/mcp-server) |
| **安装命令** | `npx -y @e2b/mcp-server` |
| **免费额度** | 100 沙箱时/月（无需信用卡） |
| **备用方案** | Docker 本地沙箱 |

**为什么必选**：
- v4.1 CLAUDE.md §安全红线明确写"禁止执行不可信代码（必须用 E2B 沙箱）"— 但没有 E2B，这只是纸面声明
- 安装 E2B 后，这条红线从"文档声明"升级为"机制保证"
- pytest 测试执行、pip 依赖安装、用户提交代码执行全部在隔离云沙箱中完成

**在四层架构中的位置**：L1 全局安全层 — 所有代码执行不经本地，统一经沙箱

**安全注意事项**：
- 沙箱自动超时销毁，无持久化风险
- 不访问本地文件系统
- 每次执行在全新 VM 中

### 2. GitHub MCP — 代码协作中枢

| 属性 | 内容 |
|------|------|
| **仓库** | [github.com/modelcontextprotocol/servers](https://github.com/modelcontextprotocol/servers) |
| **安装命令** | `claude mcp add github npx -y @modelcontextprotocol/server-github` |
| **免费额度** | 无限（GitHub API rate limit 适用） |
| **备用方案** | GitLab MCP（如果用 GitLab） |

**为什么必选**：
- v4.1 有 `Bash(git *)` Allow + Deny 保护主分支，但只能操作本地 git
- GitHub MCP 填补远程协作能力：创建 PR、查看 CI 状态、回复 Review、管理 Issue
- 与 v4.1 现有规则协同：Hook 阻止危险 git 操作，MCP 提供正常的 GitHub 工作流

**在四层架构中的位置**：L2 项目层 — 与 `.gitignore`、Deny 规则、PreToolUse Hook 协同

**提供的工具**：
```
create_pull_request, list_issues, add_issue_comment,
get_workflow_run, list_commits, search_repositories, ...
```

### 3. Context7 — 实时库文档

| 属性 | 内容 |
|------|------|
| **仓库** | [github.com/upstash/context7](https://github.com/upstash/context7) |
| **安装命令** | `npx -y @upstash/context7-mcp` |
| **免费额度** | 1,000 API 调用/月（超限后每天额外 20 次） |
| **备用方案** | 直接使用 WebFetch + 官方文档站 |

**为什么必选**：
- v4.1 `knowledge/04-Python生态链/` 有 5 个静态规范讲**最佳实践**，但不记录具体 API 签名
- Claude 经常生成过时的 FastAPI/SQLAlchemy 调用
- Context7 自动拉取**最新版本**的库文档注入上下文，消除 API 幻觉
- 与 `knowledge/` 分工明确：knowledge/ 管"怎么写是对的"，Context7 管"这个函数的参数是什么"

**在四层架构中的位置**：L2 项目层 — 与 `knowledge/` 静态规范互补

**日耗估算**：
```
个人日常开发：~10-15 次/天 → 300-450 次/月 → 免费额度充足
团队高频使用：~40 次/天 → 1,200 次/月 → 需 Pro ($10/月)
```

### 4. Snyk Scanner — 安全自动扫描

| 属性 | 内容 |
|------|------|
| **仓库** | [github.com/snyk/cli](https://github.com/snyk/cli) |
| **安装命令** | `uvx mcp-scan@latest --skills` |
| **免费额度** | 无限（开源 CLI，本地运行） |
| **备用方案** | `uv lock --check`（仅检查 Python 依赖，不扫描 Skill） |

**为什么必选**：
- v4.1 安全规则说"第三方依赖使用前检查已知漏洞"— 没有自动化机制
- 社区 Skill 生态 **36% 有安全漏洞**，安装前必须扫描
- 两条扫描线：`mcp-scan --skills`（扫 Skill）+ `uv lock --check`（扫 Python 依赖）

**在四层架构中的位置**：不入层 — 独立守护进程，安装 Skill 前触发 + 定期扫描

**使用时机**：
```bash
# 安装新 Skill/Plugin 之前
uvx mcp-scan@latest --skills /path/to/SKILL.md

# 定期检查项目依赖
uv lock --check
```

### 5. PostgreSQL MCP — 数据库操作能力

| 属性 | 内容 |
|------|------|
| **仓库** | [github.com/modelcontextprotocol/servers](https://github.com/modelcontextprotocol/servers) (官方) |
| **安装命令** | `claude mcp add postgres npx -y @modelcontextprotocol/server-postgres "postgresql://localhost:5432/mydb"` |
| **免费额度** | 无限（本地连接） |
| **备用方案** | Sequelae MCP（读写支持更好） |

**为什么必选**：
- v4.1 有完善的数据库**规范**（PostgreSQL 规范、向量数据库规范、SQL 注入防护规则）但没有数据库**操作能力**
- 就像工厂有全套施工图纸但没有机床 — 知道怎么做，做不了
- PostgreSQL MCP 提供：查询表结构、验证迁移结果、EXPLAIN ANALYZE 检查执行计划、查看索引使用情况

**在四层架构中的位置**：L2 项目层 — 与 `.claude/rules/database.md` 协同（规则约束怎么写，MCP 提供操作通道）

**安全前置要求**：
```sql
-- ⚠️ 必须创建只读用户，禁止用超级用户连接 MCP
CREATE ROLE claude_readonly WITH LOGIN PASSWORD 'strong_password';
GRANT CONNECT ON DATABASE your_db TO claude_readonly;
GRANT USAGE ON SCHEMA public TO claude_readonly;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO claude_readonly;
```

**提供的工具**：`query`（执行只读 SQL）、`list_tables`、`describe_table` 等

### 6. Brave Search MCP — Web 技术搜索

| 属性 | 内容 |
|------|------|
| **仓库** | [github.com/modelcontextprotocol/servers](https://github.com/modelcontextprotocol/servers) (官方) |
| **安装命令** | `claude mcp add brave-search -s user --env BRAVE_API_KEY=YOUR_KEY npx -y @modelcontextprotocol/server-brave-search` |

| **备用方案** | Google Custom Search（100 次/天，2027 年停用 — 不推荐） |

- 与 Context7 分工：Context7 查库文档，Brave 查技术方案、调试错误、最新实践

**在四层架构中的位置**：L2 项目层 — 与 Context7 互补（库文档 vs 通用技术搜索）

**日耗估算**：
```
个人日常开发：~20 次/天 → 600 次/月 → 免费额度充足
```

### 7. Playwright MCP — 浏览器自动化

| 属性 | 内容 |
|------|------|
| **仓库** | [github.com/microsoft/playwright-mcp](https://github.com/microsoft/playwright-mcp) (微软官方) |
| **安装命令** | `claude mcp add playwright npx @playwright/mcp@latest` |
| **免费额度** | 无限（本地运行） |
| **备用方案** | 内置 `webapp-testing` Skill（功能子集，无截屏） |

**为什么必选**：
- v4.1 `WebFetch` 只能抓取静态 HTML，无法处理 SPA 渲染的页面
- 大量现代技术文档（React/Vue 官方文档）是 SPA 渲染的，WebFetch 拿不到实际内容
- Playwright MCP 提供：SPA 页面内容获取、E2E 测试截屏验证、网页交互操作

**在四层架构中的位置**：L2 项目层 — 与内置 `webapp-testing` Skill 互补

**系统依赖**（Ubuntu）：
```bash
npx playwright install chromium
sudo npx playwright install-deps
```

## 五、安装顺序与命令

### 第一阶段：安全基础（必须先装）

```bash
# 1. E2B — 沙箱执行（L1 安全层基础）
npx -y @e2b/mcp-server

# 2. Snyk Scanner — 安全扫描工具（后续安装都先扫描）
uvx mcp-scan@latest --skills
```

### 第二阶段：开发基础设施

```bash
# 3. GitHub MCP — 代码协作中枢
claude mcp add github npx -y @modelcontextprotocol/server-github

# 4. Context7 — 实时库文档
npx -y @upstash/context7-mcp

# 5. PostgreSQL MCP — 数据库操作
claude mcp add postgres npx -y @modelcontextprotocol/server-postgres \
  "postgresql://claude_readonly:YOUR_PASS@localhost:5432/python_rag"
```

### 第三阶段：搜索与浏览器

```bash
# 6. Brave Search MCP — 技术搜索（需先获取 API Key）
#    访问 https://api.search.brave.com/app/keys 注册获取 key
claude mcp add brave-search -s user \
  --env BRAVE_API_KEY=YOUR_BRAVE_API_KEY \
  npx -y @modelcontextprotocol/server-brave-search

# 7. Playwright MCP — 浏览器自动化
claude mcp add playwright npx @playwright/mcp@latest
npx playwright install chromium
```

## 八、安装前备注

1. **Node.js v18+ 必需** — 7 个 MCP 中 6 个基于 Node.js/npx
2. **API Key 需提前注册** — Brave Search 需要免费注册获取 key
3. **PostgreSQL 只读用户必须创建** — 数据库连接前先执行安全 SQL
4. **Chromium 需安装** — Playwright 需要下载浏览器二进制
5. **安装后必须重新运行 v4.1 审计脚本** — 确保四层配置未被破坏
6. **所有 MCP 均为本地/stdio 模式运行** — 不暴露网络端口，无外部攻击面
