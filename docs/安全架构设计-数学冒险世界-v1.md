## 1. STRIDE 威胁建模

### 1.1 威胁矩阵总览

| 组件 | Spoofing (S) | Tampering (T) | Repudiation (R) | Info Disclosure (I) | DoS (D) | Elevation of Privilege (E) |
|:---|:---:|:---:|:---:|:---:|:---:|:---:|
| **API Gateway** | 🔴 高 | 🔴 高 | 🟡 中 | 🔴 高 | 🔴 高 | 🟡 中 |
| **Decision Engine** | 🟢 低 | 🔴 高 | 🟢 低 | 🟡 中 | 🟡 中 | 🔴 高 |
| **RAG Constraint** | 🟢 低 | 🔴 高 | 🟢 低 | 🟡 中 | 🟡 中 | 🟢 低 |
| **LLM Generation** | 🟡 中 | 🔴 高 | 🟢 低 | 🟡 中 | 🔴 高 | 🟢 低 |
| **PostgreSQL** | 🟡 中 | 🔴 高 | 🟡 中 | 🔴 高 | 🟡 中 | 🟡 中 |
| **关卡预生成池** | 🟢 低 | 🔴 高 | 🟢 低 | 🟡 中 | 🟡 中 | 🟢 低 |

### 1.2 高风险项详细分析与缓解措施

#### 1.2.1 API Gateway

| 威胁 | 风险描述 | 缓解措施 |
|------|----------|----------|
| **S - Spoofing** | 攻击者伪造 JWT Token（HS256 alg 混淆攻击）或重放过期 Token 获取未授权访问 | (1) 强制 RS256 非对称签名，JWKS 从服务端下发，`alg` 不从 Token header 信任；(2) 验证 `iss`, `aud`, `exp`, `iat` 四字段；(3) access_token TTL ≤ 15 分钟，refresh_token ≤ 7 天；(4) Token 绑定设备指纹 + IP 段 |
| **T - Tampering** | 攻击者篡改 HTTP 请求体/参数，绕过校验逻辑（如修改 user_id 访问他人数据） | (1) Anti-BOLA：所有资源查询按 JWT sub 过滤，禁止从请求体绑定 user_id；(2) 请求参数 Pydantic schema 强校验；(3) API 签名验证（HMAC-256 request body signing）|
| **R - Repudiation** | 用户否认发起过敏感操作（如数据删除请求） | (1) 所有写操作生成不可篡改审计日志（HMAC 链式哈希 + 独立 Audit DB）；(2) 家长敏感操作要求邮箱验证码 + 时间戳签名 |
| **I - Info Disclosure** | 攻击者通过响应头/错误信息获取内部架构信息 | (1) 统一异常处理中间件，生产环境不暴露堆栈；(2) 响应头移除 Server/X-Powered-By；(3) 所有 API 响应 Content-Type 固定，禁用目录遍历 |
| **D - DoS** | 攻击者通过大量请求耗尽 API 资源，或利用 LLM 长响应阻塞线程池 | (1) 三层限流：IP 级 (100 req/min) → 用户级 (20 次/关) → 全局级 (500 并发)；(2) LLM 调用独立隔离池，不占用 API worker 线程；(3) 关卡预生成池缓冲 LLM 峰值 |
| **E - Elevation** | 攻击者通过参数篡改提升权限（如修改 role 为 admin） | (1) 权限字段不来自请求体——从 JWT claims 读取；(2) RABC 矩阵与 JWT role 绑定，API 路由级授权中间件 |

#### 1.2.2 Decision Engine

| 威胁 | 风险描述 | 缓解措施 |
|------|----------|----------|
| **T - Tampering** | 攻击者修改难度算法参数/掌握度矩阵，使系统过分拔高或压低难度 | (1) 算法参数由配置文件控制，禁止运行时修改；(2) 难度参数签名验证，篡改则降级为默认参数集；(3) 核心算法输出哈希校验 |
| **E - Elevation** | 攻击者覆盖决策引擎的知识点选择逻辑（如强制生成超纲内容） | (1) 决策引擎只接受结构化输入（知识点ID/难度值/模式类型），不暴露自由度参数；(2) 所有决策输出经 RAG 约束层二次校验 |
| **D - DoS** | 决策引擎被大量无效请求耗尽计算资源 | (1) 决策引擎非阻塞异步调用；(2) 单连接并发上限 = 3；(3) 决策结果缓存（Redis, TTL 5 分钟）|

#### 1.2.3 RAG Constraint

| 威胁 | 风险描述 | 缓解措施 |
|------|----------|----------|
| **T - Tampering** | 攻击者污染向量数据库，插入恶意文档使 LLM 生成不当内容（数据投毒） | (1) 知识库写入权限仅限 Admin 角色 API，且要求请求签名；(2) 写入内容经过：MinerU 解析 → 安全审核 → 标注源 → 入库，无法直接写入 Milvus；(3) 周期性向量完整性校验（checksum 比对） |
| **I - Info Disclosure** | RAG 检索返回不属于本用户的私有数据（跨用户数据泄露） | (1) 公共库与私有库物理隔离（两个 Milvus collection）；(2) 私有库检索附加 tenant_id 过滤（从 JWT sub 派生，不从请求体读取）；(3) 检索结果脱敏校验 |

#### 1.2.4 LLM Generation

| 威胁 | 风险描述 | 缓解措施 |
|------|----------|----------|
| **T - Tampering** | Prompt 注入攻击——学生输入突破 System Prompt 约束，使 LLM 输出不当内容 | (1) Pre-LLM 注入检测（种子库 ≥ 50 条儿童场景）；(2) 学生输入作为 `data` 而非 `instruction` 注入；(3) System Prompt 固化红线约束，不可被对话覆盖；(4) Post-LLM 输出审核兜底 |
| **D - DoS** | LLM API 请求堆积导致队列溢出，或单个请求超时阻塞 | (1) LLM 调用超时 60s，超时降级为模板内容；(2) 请求队列最大 50，满队列触发降级模式；(3) 限流：单学生 10 次/小时（关卡生成），100 次/小时（对话） |
| **I - Info Disclosure** | LLM 输出中包含其他用户的个人信息（训练数据泄露） | (1) 禁止将儿童数据用于模型训练（COPPA 2025 强制）；(2) Prompt 中的用户数据仅包含"薄弱知识点标签"等非个人标识信息 |

#### 1.2.5 PostgreSQL

| 威胁 | 风险描述 | 缓解措施 |
|------|----------|----------|
| **S - Spoofing** | 攻击者伪造 PostgreSQL 连接，绕过应用层直接访问数据库 | (1) 数据库仅监听 127.0.0.1 (Docker 内部网络)，不暴露外部端口；(2) TLS 双向认证（客户端证书）；(3) 连接池用户名/密码从环境变量读取，禁止硬编码 |
| **T - Tampering** | SQL 注入攻击篡改查询 | (1) 所有查询使用 asyncpg 参数化查询（禁止字符串拼接）；(2) ORM 层（SQLAlchemy）自动参数化；(3) 数据库用户权限最小化：应用用户仅 SELECT/INSERT/UPDATE，无 DDL 权限 |
| **R - Repudiation** | 管理员否认执行过数据修改操作 | (1) PostgreSQL audit log（pgAudit 扩展）记录所有 DDL/DML 操作；(2) 审计日志独立存储，应用用户无删除审计日志权限 |
| **I - Info Disclosure** | SQL 注入或配置错误导致儿童数据泄露 | (1) 敏感字段加密存储（姓名/邮箱/家长联系方式使用 AES-256-GCM）；(2) 全量备份加密（gpg symmetric encryption）；(3) 查询结果脱敏中间件——日志中不输出原始数据 |

#### 1.2.6 关卡预生成池

| 威胁 | 风险描述 | 缓解措施 |
|------|----------|----------|
| **T - Tampering** | 攻击者篡改预生成关卡内容缓存，使未审核内容直接展示给学生 | (1) 预生成池存储内容为经过 Post-LLM 审核通过的关卡；(2) 内容哈希签名验证（HMAC-SHA256），读取时校验完整性；(3) 预生成池与 Redis 隔离写权限，仅后台 worker 可写入 |
| **I - Info Disclosure** | 学生 A 看到为学生 B 预生成的关卡（包含 B 的学情数据） | (1) 预生成池按 user_id 分片存储；(2) 读取时验证请求者 user_id 与缓存 key 的 user_id 一致 |

### 2.2 Layer 1: Policy as Code 策略引擎

**技术选型**：Casbin（Python）— 轻量，原生 Python 生态，无需独立服务进程

#### 策略模型 (Model)

```
[request_definition]
r = sub, obj, act

[policy_definition]
p = sub, obj, act, eft

[role_definition]
g = _, _
g2 = _, _

[matchers]
// 管理员
m = g(r.sub, "admin") && r.act != "delete_system"
// 家长：仅操作自己关联的儿童数据
m = g(r.sub, "parent") && g2(r.obj, r.sub) && r.act in ["read_child", "delete_child", "export_child"]
// 儿童：仅读取自己的关卡和基础信息
m = g(r.sub, "child") && g2(r.obj, r.sub) && r.act in ["play_level", "read_profile"]
// 教师：仅读取班级视图（脱敏）
m = g(r.sub, "teacher") && r.act in ["read_class_summary", "read_class_progress"]
// 系统服务：仅内部 API 调用
m = g(r.sub, "system") && r.act in ["internal_write", "internal_read"]
```

#### 策略规则示例

```
# 角色继承
g, child_123, child
g, parent_456, parent
g, parent_456, parent_of_child_123
g2, child_123, parent_456

# 资源 - 角色绑定
g2, level_789, child_123
g2, report_parent, parent_456

# API 路由策略
p, child, "/api/v1/levels/*", "GET", allow
p, child, "/api/v1/profile", "GET", allow
p, child, "/api/v1/*", "POST", deny  # 儿童只能 GET，不能 POST
p, parent, "/api/v1/children/*/data", "DELETE", allow
p, parent, "/api/v1/children/*/data", "GET", allow
p, admin, "/api/v1/admin/*", "*", allow
```

#### 限流策略 (Rate Limit)

| 角色 | 端点 | 限流值 | 超出处理 |
|------|------|--------|----------|
| 儿童 | 关卡生成 | 1 次/5 分钟 | 返回预生成池缓存 |
| 儿童 | 博士对话 | 20 次/关 | 返回"休息一下"提示 |
| 儿童 | 提交答题 | 50 次/关 | 阻断 + "慢慢来"提示 |
| 家长 | 查看报告 | 10 次/小时 | 返回 429 |
| 全局 | 所有 API | 500 并发 | 队列等待或 503 |

### 2.3 Layer 2: DLP 数据防泄漏

#### 2.3.1 PII 检测与脱敏规则

| PII 类型 | 检测方式 | 脱敏策略 |
|----------|----------|----------|
| 中文姓名（2-7 字） | 正则 + NER 模型 | "张小明" → "张同学" |
| 手机号 | 正则 `1[3-9]\d{9}` | 保留前 3 后 4：`138****4567` |
| 家庭地址 | 正则 + NER | "北京市朝阳区..." → "你所在的城市" |
| 学校名称 | 提示词库 + NER | "实验小学" → "你的学校" |
| 身份证号 | 正则 `\d{17}[\dXx]` | 完全脱敏 → "[已过滤]" |
| 家长联系方式 | 正则 `[\w.+-]+@[\w-]+\.[\w.-]+` | 完全脱敏 → "[已过滤]" |
| 生日/年龄 | 正则 | "我今年 8 岁" → 保留范围："嗯，你在这个年龄段" |

#### 2.3.2 脱敏执行流程

```
用户输入
  │
  ├── Pre-LLM PII 脱敏模块
  │   ├─ Step 1: 正则快速匹配（高频 PII 类型）—— O(1) 级
  │   ├─ Step 2: NER 模型检测（中低频，中文姓名/地址）—— ~50ms
  │   └─ Step 3: 脱敏替换 → 输出脱敏文本 + 脱敏日志
  │
  ├──→ LLM 处理（脱敏后文本，不接触原始 PII）
  │
  └──→ Post-LLM PII 回填检查
      ├─ 检查 LLM 输出是否意外嵌入了原始 PII（LLM 记忆泄露）
      └─ 检测到 → 段落级替换为"该内容无法显示"
```

#### 2.3.3 家长数据删除管道

```
家长发起删除请求
  │
  ├── Layer 1 身份验证：JWT + 邮箱验证码二次确认
  │
  ├── Layer 2 请求记录：生成删除请求流水号 → 写入 Audit Log
  │
  ├── Layer 3 异步执行队列：
  │   ├─ 任务 1: PostgreSQL — 用户行标记 deleted_at + 软删除
  │   ├─ 任务 2: Milvus — 用户私有 collection 整体 drop
  │   ├─ 任务 3: Redis — 用户缓存逐出（TTL 设置为 0）
  │   ├─ 任务 4: 审计日志 — 保留删除记录，删除具体数据
  │   └─ 任务 5: 关卡预生成池 — 清理该用户的缓存内容
  │
  └── Layer 4 删除确认：
      ├─ 邮件通知家长"删除已完成"
      └─ 任务完成状态写入删除日志（证据链归档）

SLA: 删除请求 → 确认完成 ≤ 3 个工作日（COPPA 2025 要求）
     设计目标：异步队列全量执行 ≤ 30 分钟
```

### 2.4 Layer 3: 防篡改审计日志

#### 2.4.1 审计范围与粒度

| 事件类型 | 记录内容 | 保留期限 |
|----------|----------|----------|
| **LLM 生成** | 关卡 ID + 输入 Prompt 摘要(SHA256) + 输出全文 + 校验结果 + 耗时 | 180 天 |
| **用户交互** | 用户 ID + 操作类型 + 时间戳 + IP + User-Agent | 180 天 |
| **脱敏事件** | 用户 ID + 脱敏 PII 类型 + 时间戳（**不记录原始 PII**） | 90 天 |
| **安全告警** | 事件等级 + 类型 + 触发规则 + 操作员 | 365 天 |
| **管理操作** | 管理员 ID + 操作 + 参数 + 结果 + 来源 IP | 365 天 |
| **数据删除** | 删除请求 ID + 用户 ID + 删除范围 + 完成状态 | 永久 |

#### 2.4.2 链式哈希结构

```
┌──────────────────────────────────────────────────────────┐
│                HMAC 链式哈希审计日志                        │
│                                                          │
│  Block 1 (prev_hash = 0000...)                           │
│  ┌──────────────────────────────┐                        │
│  │ timestamp: 2026-06-01T10:00Z │  H(prev_hash + data)  │
│  │ event_type: LLM_GENERATE     │       =               │
│  │ event_data: { level_id, ... }│   hash_1 = 0x7A3B...  │
│  │ hash: hash_1                 │                        │
│  └──────────────────────────────┘                        │
│                               ↓                          │
│  Block 2 (prev_hash = hash_1)                            │
│  ┌──────────────────────────────┐                        │
│  │ timestamp: 2026-06-01T10:01Z │  H(hash_1 + data)      │
│  │ event_type: INPUT_SANITIZE   │       =               │
│  │ event_data: { pii_type, ...} │   hash_2 = 0xE1F4...  │
│  │ hash: hash_2                 │                        │
│  └──────────────────────────────┘                        │
│                               ↓                          │
│  Block 3 (prev_hash = hash_2)                            │
│  ...                                                     │
│                                                          │
│  完整性验证方式：                                          │
│  1. 从 Block 1 开始，顺序计算每一块的 hash                 │
│  2. 如果某块被篡改，其后所有块的 hash 均不匹配              │
│  3. 日终归档时记录"最终哈希"到独立的"见证存储"(PostgreSQL)   │
└──────────────────────────────────────────────────────────┘
```

#### 2.4.3 审计日志架构

```
审计日志生产者
  │
  ├── 同步通道（关键操作：数据删除/管理操作/安全告警）
  │   └── 直接写入 Audit DB（PostgreSQL 独立实例，最小权限）
  │
  └── 异步通道（非关键操作：LLM 生成日志/常规用户交互）
      └── Redis 队列 → Worker 批量写入（缓冲保护，峰值 1000 条/秒）

日终归档：
  1. 计算当日链式哈希最终值
  2. 最终哈希写入"见证表"（PostgreSQL，仅 APPEND）
  3. 全量日志归档到 S3/本地磁盘（gzip 压缩）
  4. 在线 Audit DB 保留 90 天热数据
```

### 2.5 Layer 4: 产品治理强制执行

#### 2.5.1 70+ 条规则到代码层的映射策略

| 规则类别 | 条数 | 执行方式 | 代码层形态 |
|----------|:---:|----------|-----------|

```python
        # 从独立 JSON 配置文件加载，不硬编码

        """Pre-LLM 输入消毒：检测红线规则，命中则拦截"""
        violations = []
        for keyword in self.POLITICAL_KEYWORDS + self.RELIGIOUS_KEYWORDS:
            if keyword in text:
                violations.append(f"redline_political_religious:{keyword}")
        for keyword in self.VIOLENCE_KEYWORDS:
            if keyword in text:
                violations.append(f"redline_violence:{keyword}")

        """Post-LLM 输出审核：红线阻断 + 强约束检查"""
        violations = []

        for keyword in self.POLITICAL_KEYWORDS + self.RELIGIOUS_KEYWORDS:
            if keyword in text:
                violations.append(f"blocked_redline:{keyword}")

        # 暴力内容检查
        for keyword in self.VIOLENCE_KEYWORDS:
            if keyword in text:
                violations.append(f"blocked_violence:{keyword}")

        # AI 身份披露检查
        if not any(variant in text for variant in self.AI_IDENTITY_VARIANTS):
            violations.append("missing_ai_identity_disclosure")

        # 禁止表述检查 (REQ-VLD-001, REQ-NRT-001)
        forbidden_phrases = ["请计算", "正确答案是", "你做错了", "再试一次"]
        for phrase in forbidden_phrases:
            if phrase in text:
                violations.append(f"forbidden_phrase:{phrase}")

        return len(violations) == 0, violations
```

## 3. 儿童身份与同意管理

### 3.1 年龄验证方案（最小 PII 收集）

#### 设计方案

采用**分级验证模型**——不要求所有用户上传出生证明等强 PII，而是根据操作风险逐级验证：

| 验证级别 | 验证方式 | 收集的 PII | 适用场景 |
|----------|----------|-----------|----------|
| **L0 自声明** | 选择出生年份范围（非精确日期） | 年龄区间（6-8 / 9-11 / 12-14） | 首次注册、开始游戏 |
| **L1 家长邮箱验证** | 填写家长邮箱 → 发送验证码 → 验证通过 | 家长邮箱（仅用于验证 + 通知） | 数据导出、解锁关卡限制 |
| **L2 家长身份确认** | 邮箱验证 + 支付验证（小额扣款 ￥0.01 后返还） | 家长邮箱 + 支付渠道 | 删除儿童账户、导出完整数据 |
| **L3 家长正式认证** | 邮箱验证 + 身份证号后 4 位 + 人脸比对（可选） | 家长身份证部分信息 | 数据迁移、法律争议 |

#### 年龄验证流程

```
用户注册
  │
  ├── Step 1: 选择年龄区间（不要求精确出生日期）
  │   └── 6-8 岁 → 低年级模式
  │   └── 9-11 岁 → 中年级模式
  │   └── 12-14 岁 → 高年级模式（可放宽部分限制）
  │
  ├── Step 2: 填写家长邮箱（即可开始使用，不阻塞首次体验）
  │   └── 系统发送"家长通知邮件"：孩子已注册 + 管理链接（7 日有效）
  │   └── 7 日内家长未点击链接 → 第二次提醒 + 功能降级（限制每日关卡数）
  │
  └── Step 3: 家长点击链接后可选操作
      ├── "确认同意"（VPC 完成，全功能可用）
      ├── "我想了解更多"（转至家长说明页面）
      └── "这不是我的孩子"（发起账户删除流程，3 日内自动删除）
```

### 3.3 儿童模式 vs 家长模式的会话隔离

#### 3.3.1 会话类型定义

| 属性 | 儿童模式 | 家长模式 |
|------|----------|----------|
| 登录方式 | 无密码（会话 Token 绑定设备） | 邮箱 + 密码 + TOTP（可选） |
| Session TTL | 24 小时（连续使用刷新） | 2 小时（无操作超时） |
| 可访问 API | GET only（关卡/对话/收藏） | GET/POST/DELETE（管理操作） |
| 数据可见范围 | 仅自己的关卡和收藏 | 自己关联的所有儿童数据 |
| 操作审计 | 标准事件日志 | 全量审计（含管理操作指纹） |

#### 3.3.2 会话隔离机制

```
                  ┌──────────────────────────────┐
                  │       JWT Token 识别          │
                  │                              │
                  │  Child Token:                 │
                  │  { sub: "child_789",          │
                  │    role: "child",             │
                  │    parent_id: "parent_456",   │
                  │    device_id: "device_abc",   │
                  │    exp: 24h,                  │
                  │    scope: "play_only" }       │
                  │                              │
                  │  Parent Token:                │
                  │  { sub: "parent_456",         │
                  │    role: "parent",            │
                  │    child_ids: [789, 790],     │
                  │    exp: 2h,                   │
                  │    scope: "manage_children",  │
                  │    mfa_verified: true }       │
                  └──────────────────────────────┘

安全规则：
1. 儿童 Token 无法查看任何"管理/设置"类页面
2. 儿童 Token 发起的 DELETE 请求 → 强制返回 403
3. 家长 Token 无法进入游戏界面（关卡/对话）
4. 设备切换上下文时强制注销原 Token
5. 同一设备不允许同时持有儿童+家长活跃 Token
```

#### 3.3.3 家长敏感操作二次认证

```
敏感操作（数据删除/导出/更改密码）
  │
  ├── Step 1: 验证 JWT Token（家长角色 + 有效期内）
  │
  ├── Step 2: 发送二次验证码到注册邮箱
  │   └── 验证码有效期 5 分钟，单次使用
  │
  ├── Step 3: 校验验证码 + 记录审计日志
  │   └── 写入 Block: { operation, timestamp, ip, mfa_method }
  │
  └── Step 4: 执行操作
      └── 操作结果写入审计日志 (链式哈希)
```

### 4.3 Red Team 测试执行脚本设计

```python
# scripts/redteam_runner.py
# CI Pipeline 红队测试执行器

import json
import sys
from pathlib import Path
from typing import Dict, List, Tuple

sys.path.insert(0, str(Path(__file__).parent.parent / "src"))
from tests.redteam.seed_library import RedTeamTestCase, RedTeamSeedLibrary

class RedTeamRunner:
    """红队测试执行器"""

    def __init__(self, break_rate_threshold: float = 0.01):
        self.library = RedTeamSeedLibrary()
        self.break_rate_threshold = break_rate_threshold
        self.results: List[Dict] = []

    def run_pre_llm_tests(self) -> Tuple[int, int]:
        """测试 Pre-LLM 消毒模块"""
        from core.governance.red_line_rules import RedLineRuleEngine
        engine = RedLineRuleEngine()

        passed = 0
        blocked = 0

        for tc in self.library.test_cases:
            _, violations = engine.pre_llm_sanitize(tc.input_text)

            # 判断期望结果
            if tc.expected_result == "blocked":
                expected_blocked = True
            else:
                expected_blocked = False

            actual_blocked = len(violations) > 0

            if expected_blocked == actual_blocked:
                passed += 1
                status = "PASS"
            else:
                blocked += 1
                status = "FAIL"

            self.results.append({
                "id": tc.id,
                "category": tc.category,
                "stage": "pre_llm",
                "expected": tc.expected_result,
                "input": tc.input_text[:50],
                "status": status,
                "violations": violations,
            })

        return passed, blocked

    def run_post_llm_tests(self) -> Tuple[int, int]:
        """测试 Post-LLM 审核模块"""
        from core.governance.red_line_rules import RedLineRuleEngine
        engine = RedLineRuleEngine()

        passed = 0
        blocked = 0

        for tc in self.library.test_cases:
            if tc.expected_result != "blocked":
                continue  # 只测试需要拦截的用例

            # 模拟 LLM 输出了被拦截的内容
            simulated_llm_output = f"博士回答说：{tc.input_text}"
            is_clean, violations = engine.post_llm_audit(simulated_llm_output)

            if not is_clean:
                passed += 1
                status = "PASS"
            else:
                blocked += 1
                status = "FAIL"

            self.results.append({
                "id": tc.id,
                "category": tc.category,
                "stage": "post_llm",
                "expected": "blocked",
                "status": status,
                "violations": violations,
            })

        return passed, blocked

    def generate_report(self) -> Dict:
        """生成安全测试报告"""
        total = len(self.library.test_cases)
        pre_passed, pre_blocked = self.run_pre_llm_tests()
        post_passed, post_blocked = self.run_post_llm_tests()

        total_tests = pre_passed + pre_blocked + post_passed + post_blocked
        total_passed = pre_passed + post_passed
        total_failed = pre_blocked + post_blocked
        break_rate = total_failed / total if total > 0 else 0

        report = {
            "timestamp": "2026-06-01T00:00:00Z",
            "total_test_cases": total,
            "pre_llm": {
                "passed": pre_passed,
                "blocked": pre_blocked,
                "pass_rate": pre_passed / (pre_passed + pre_blocked) if (pre_passed + pre_blocked) > 0 else 0,
            },
            "post_llm": {
                "passed": post_passed,
                "blocked": post_blocked,
                "pass_rate": post_passed / (post_passed + post_blocked) if (post_passed + post_blocked) > 0 else 0,
            },
            "overall": {
                "passed": total_passed,
                "failed": total_failed,
                "break_rate": break_rate,
                "threshold": self.break_rate_threshold,
                "verdict": "PASS" if break_rate < self.break_rate_threshold else "FAIL",
            },
            "by_category": self._results_by_category(),
            "details": self.results,
        }

        return report

    def _results_by_category(self) -> Dict:
        categories = {}
        for r in self.results:
            cat = r["category"]
            if cat not in categories:
                categories[cat] = {"total": 0, "pass": 0, "fail": 0}
            categories[cat]["total"] += 1
            if r["status"] == "PASS":
                categories[cat]["pass"] += 1
            else:
                categories[cat]["fail"] += 1
        return categories

    def write_report(self, report: Dict, path: str = "redteam-report.json"):
        with open(path, "w", encoding="utf-8") as f:
            json.dump(report, f, ensure_ascii=False, indent=2)
        print(f"Red Team report written to {path}")
        print(f"Overall verdict: {report['overall']['verdict']}")
        print(f"Break rate: {report['overall']['break_rate']:.4%} (threshold: {self.break_rate_threshold:.1%})")


if __name__ == "__main__":
    runner = RedTeamRunner(break_rate_threshold=0.01)  # < 1%
    report = runner.generate_report()
    runner.write_report(report)

    if report["overall"]["verdict"] == "FAIL":
        print("ERROR: Red team tests failed — break rate exceeds 1% threshold!")
        sys.exit(1)
    else:
        print("SUCCESS: Red team tests passed.")
        sys.exit(0)
```

### 4.4 突破率 < 1% 的验证标准

| 指标 | 目标值 | 测量方式 | 持续验证 |
|------|--------|----------|----------|
| **Pre-LLM 拦截率** | ≥ 99% | 对种子库 54 条 + 变异 100 条的测试 | 每次 CI 构建 |
| **Post-LLM 拦截率** | ≥ 99% | 对种子库中 expected=blocked 的测试 | 每次 CI 构建 |
| **全链路突破率** | < 1% | 端到端注入测试 (种子库变异版) | 每日定时 CI |
| **PII 脱敏覆盖率** | 100% (已知 PII 类型) | 正则 + NER 测试覆盖 | 每次 CI 构建 |
| **安全回归测试** | 0 回归 | 新注入向量加入种子库后全量重跑 | 种子库更新时 |
| **红队人工测试** | 月度 | 外部安全研究员/自动化工具 | 月度周期 |

#### 种子库动态维护策略

```
种子库维护规则：
1. 新注入向量发现 → 24h 内加入种子库
2. 注入向量变异 → 每季度自动生成 50 条变异用例（fuzzing 策略）
3. 种子库淘汰 → 每半年审查，移除不再有效的注入向量
4. 种子库版本号 → semver 管理，用于 VPC 记录 consent_version

变异生成策略（fuzzing）：
  - 同义词替换（"忽略"→"无视"/"跳过"/"不管"）
  - 句式变换（陈述句→疑问句→祈使句）
  - 编码绕过（base64/unicode/url 编码变体）
  - 多轮注入（第一轮建立信任 → 第二轮执行注入）
  - 上下文污染（先正常对话 → 对话中突然插入注入）
```

### 5.2 Pre-LLM 输入消毒通道

#### 5.2.1 规则引擎执行流程

```
输入文本
  │
  ├── Step 1: 长度检查
  │   └── 单次输入 ≤ 500 中文字符（超出截断并告警）
  │
  ├── Step 2: 红线规则引擎
  │   ├── 政治敏感关键词表（~200 条，JSON 配置）—— 命中即阻断
  │   ├── 宗教内容关键词表（~50 条）—— 命中即阻断
  │   ├── 武器/暴力关键词表（~30 条）—— 命中即阻断
  │   └── 不当内容关键词表（~100 条）—— 命中即阻断
  │   └── 阻断 → 返回友善提示："我不太明白，能换个说法吗？"
  │
  ├── Step 3: Prompt 注入检测
  │   ├── 正则模式匹配（"忽略指令"/"假装你是"/"忘记规则"等 20 种模式）
  │   ├── 注入种子库词频分析（字符级 n-gram + 阈值检测）
  │   ├── 上下文一致性检查（当前输入与历史记录的语义突变检测）
  │   └── 检测到注入 → 阻断 + 安全审计日志
  │
  ├── Step 4: PII 脱敏
  │   ├── 正则匹配（手机/身份证/邮箱/地址等）
  │   ├── NER 模型检测（中文姓名/学校名称等）
  │   ├── 替换为占位符 + 记录到脱敏日志
  │   └── 输出脱敏后文本
  │
  └── → 进入 LLM（脱敏后文本，带安全标记 header）
```

#### 5.2.2 注入检测模型参数

```
注入检测模型规格：

模型类型：轻量级分类器（非 LLM，避免循环依赖）
推理方式：正则 + 关键词词频 + 语义相似度（MiniLM 中文版 embedding）
部署形态：内存模型 + ONNX Runtime（< 50MB 内存, < 10ms 推理）

检测特征维度：
  1. 指令覆盖模式（"忽略"、"忘记"、"跳过"、"无视"）—— 加权权重 0.4
  2. 角色切换模式（"假装"、"现在你是"、"你不是"）—— 加权权重 0.3
  3. 权限声明模式（"我是大人"、"我是老师"、"我妈妈说了"）—— 加权权重 0.2
  4. 上下文突变（余弦相似度 < 0.3 vs 历史对话均值）—— 加权权重 0.1

决策阈值：
  综合得分 ≥ 0.7 → 阻断（高置信度注入）
  综合得分 ≥ 0.4 → 标记可疑 + 脱敏后放行 + 加入人工抽检
  综合得分 < 0.4 → 放行
```

#### 5.3.1 审核分类器规格


```python
# src/core/audit/sampling_queue.py

from dataclasses import dataclass
from enum import Enum
from datetime import datetime
from typing import Optional

class ReviewPriority(Enum):
    P1_HIGH = "p1_high"        # 2 小时内处理
    P2_MEDIUM = "p2_medium"    # 24 小时内处理
    P3_LOW = "p3_low"          # 72 小时内处理

class ReviewStatus(Enum):
    PENDING = "pending"
    IN_REVIEW = "in_review"
    APPROVED = "approved"
    REJECTED = "rejected"
    ESCALATED = "escalated"

@dataclass
class ReviewQueueItem:
    """抽检队列项"""
    review_id: str                     # 审核唯一标识
    priority: ReviewPriority           # 优先级
    status: ReviewStatus               # 审核状态
    created_at: datetime               # 入队时间
    sla_deadline: datetime             # SLA 截止时间

    # 关联数据（脱敏后引用）
    level_id: str                      # 关卡 ID
    child_age_group: str               # 年龄段
    knowledge_point: str               # 知识点
    content_snippet: str               # 内容摘要（前 100 字）

    # 自动审核结果
    auto_audit_result: str             # "passed" | "flagged" | "failed"
    auto_audit_details: list           # 触发规则详情
    post_llm_risk_score: float         # 风险评分 (0-1)

    # 人工审核结果
    reviewer: Optional[str] = None     # 审核人
    reviewed_at: Optional[datetime] = None
    review_decision: Optional[str] = None  # "approve" | "reject" | "escalate"
    review_notes: Optional[str] = None     # 审核备注


class SamplingQueueManager:
    """人工抽检队列管理器"""

    # 每日抽检率
    DAILY_SAMPLE_RATE = 0.05  # 5%

    def should_sample(self, level_id: str, risk_score: float) -> bool:
        """判断是否需要抽检"""
        # 高风险内容强制入队
        if risk_score >= 0.7:
            return True
        # 随机 5% 抽检
        import hashlib
        hash_val = int(hashlib.md5(level_id.encode()).hexdigest(), 16)
        return (hash_val % 100) / 100 < self.DAILY_SAMPLE_RATE

    def calculate_priority(self, level_id: str, risk_score: float,
                           is_user_reported: bool,
                           is_first_generation: bool) -> ReviewPriority:
        """计算审核优先级"""
        if is_user_reported or risk_score >= 0.7:
            return ReviewPriority.P1_HIGH
        if is_first_generation or risk_score >= 0.4:
            return ReviewPriority.P2_MEDIUM
        return ReviewPriority.P3_LOW

    def escalate_sop(self, level_id: str, reason: str):
        """触发升级 SOP"""
        # 确认数学错误/内容安全事件
        # 暂停该知识点的 LLM 生成
        # 24h 内修复并人工验证
        # 写入安全告警审计日志
        pass
```

### 6.3 安全架构决策记录

```
ADR-SEC-001: Pre/Post-LLM 双端审核架构
- 决策：学生输入进 LLM 前消毒，LLM 输出到用户前审核
- 理由：单端无法覆盖所有风险（输入绕过 + 输出幻觉）
- 替代方案：仅 Post-LLM 审核（不充分，注入攻击无法防范）

ADR-SEC-002: 治理规则引擎 vs 纯 LLM 审核
- 决策：确定性规则引擎（关键词+正则+N分类器）为主，LLM 审核为辅
- 理由：规则引擎确定性高、延迟低（<10ms）、无循环依赖
- 限制：LLM 审核仅用于"指导原则"级别的人工抽检辅助

ADR-SEC-003: 审计日志链式哈希
- 决策：采用链式哈希而非简单日志
- 理由：满足合规审计的"不可篡改"要求
- 替代方案：区块链（太重，不需要去中心化）

ADR-SEC-004: 年龄验证最小 PII
- 决策：分 4 级验证，从自声明到正式认证渐进升级
- 理由：不阻塞首次体验，同时满足 COPPA 2025 要求
- 设计约束：6-8 岁最低保护等级，12-14 岁可适当放宽

ADR-SEC-005: 种子库 + Fuzzing 双保险
- 决策：手动种子库（54 条）+ 自动变异生成（每季度 50+ 条）
- 理由：手工确保覆盖已知攻击面，自动变异覆盖未知变体
- 验证标准：突破率 < 1%

ADR-SEC-006: 安全审核独立部署
- 决策：内容安全审核服务独立部署（不与应用服务共享进程）
- 理由：安全审计的隔离性和可靠性要求（一个进程崩溃不影响审核）
- 实施：审核服务 = 独立 Docker container，独立资源限制
```
