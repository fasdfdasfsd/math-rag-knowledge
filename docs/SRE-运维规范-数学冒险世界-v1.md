## 1. SLI / SLO 定义

### 1.1 总体承诺

| 指标 | SLO 目标 | SLA 承诺 | 衡量周期 | 说明 |
|------|----------|----------|----------|------|
| 可用性 | >= 99.5% | >= 99.0% | 月度 | 核心 API 健康检查成功率 |
| 延迟 P95 | < 2s | < 3s | 月度 | 所有用户请求（含 LLM 生成） |
| 延迟 P99 | < 5s | < 8s | 月度 | 含 LLM 流式生成的尾延迟 |
| 错误率 | < 1% | < 2% | 月度 | HTTP 5xx / 4xx 占比（不含 429） |

### 1.2 按服务维度 SLI

| 服务 | SLI 指标 | 衡量方法 | SLO |
|------|----------|----------|-----|
| **FastAPI 主应用** | API 健康检查成功率 | `/health` 端点每 15s 探测 | >= 99.5% |
| | API 响应延迟 P95 | Prometheus `http_request_duration_seconds` | < 500ms（非 LLM） |
| | API 错误率 | 5xx 响应占比 | < 0.5% |
| **PostgreSQL** | 连接成功率 | `pg_isready` | >= 99.9% |
| | 查询延迟 P95 | `pg_stat_activity` 慢查询 | < 200ms |
| | 连接池饱和度 | 活跃连接数 / max_connections | < 80% |
| **Redis** | 缓存命中率 | `keyspace_hits / (hits + misses)` | >= 85% |
| | 响应延迟 P99 | `INFO LATENCY` | < 5ms |
| | 内存使用率 | `used_memory / maxmemory` | < 80% |
| **Milvus** | 向量搜索延迟 P95 | Milvus metrics API | < 300ms |
| | 集合加载状态 | `LoadState` | 100% 加载成功 |
| | 索引构建成功率 | 索引任务完成率 | >= 99% |
| **LLM 生成** | 生成成功率 | 非流式 2xx 响应 | >= 95% |
| | 生成延迟 P95 | 从请求到首 token / 全文完成 | < 10s |
| | Token 配额使用率 | 已用 / 月配额 | < 90% |
| **内容校验服务** | 校验请求延迟 P95 | HTTP 请求时长 | < 1s |
| | 校验错误率 | 校验服务自身 5xx | < 0.5% |
| **预生成 Worker** | 关卡缓存命中率 | 用户请求命中预生成池 | >= 60% |
| | 预生成成功率 | 预生成任务完成率 | >= 95% |

### 1.3 用户感知 SLI

| 指标 | SLO | 说明 |
|------|-----|------|
| 登录成功率 | >= 99.5% | JWT 认证全链路 |
| 关卡加载延迟 P95 | < 3s | 从点击冒险到关卡显示完毕 |
| NPC 对话响应 P95 | < 5s | 含 LLM 生成时间 |
| 知识点检索延迟 P95 | < 1s | Milvus 语义搜索 + 结果排序 |
| 家长看板数据刷新 | < 2s | PostgreSQL 聚合查询 |

## 2. 告警规则设计

### 2.1 优先级定义

| 级别 | 定义 | 首次响应 | 进场时间 | 通知方式 |
|------|------|----------|----------|----------|
| **P0** | 服务完全不可用 / 核心功能中断 | 5 分钟 | 持续至恢复 | 电话 + 即时消息（企业微信/飞书）+ PagerDuty |
| **P1** | 服务严重受损 / 大批用户受影响 | 15 分钟 | 4 小时内 | 即时消息 + 电话（值班人） |
| **P2** | 部分功能异常 / 小范围影响 | 2 小时 | 24 小时内 | 即时消息 |
| **P3** | 预警通知 / 无需立即处理 | Next Sprint | 下个迭代 | 邮件 / 工单（Jira） |

### 2.2 P0 告警（立即响应）

| 告警名称 | 触发条件 | 持续时间 | 影响 | 建议 Runbook |
|----------|----------|----------|------|-------------|
| **主 API 不可用** | `/health` 连续 3 次探测失败 | > 1 分钟 | 全体用户无法访问 | 检查 Docker 容器状态 → 查看日志 → 回滚最近变更 |
| **PostgreSQL 不可用** | `pg_isready` 失败 | > 30 秒 | 所有数据库操作失败 | 检查容器 → 检查磁盘 → 检查 WAL 归档 → 从备份恢复 |
| **Milvus 不可用** | Milvus health check 失败 | > 2 分钟 | 语义检索失败，RAG 降级 | 检查 etcd / MinIO → 重启 Milvus → 重建索引 |
| **Redis 不可用** | ping 失败 | > 30 秒 | 会话 / 缓存 / Embedding 缓存失效 | 检查容器 → 检查 RDB/AOF → 检查 OOM |
| **LLM 调用连续失败** | 连续 10 次 LLM API 返回 5xx | > 2 分钟 | 关卡生成 / NPC 对话不可用 | 检查 API Key → 检查配额 → 切换备用 Provider |
| **错误预算耗尽** | 当月错误预算使用率 >= 100% | 触发即告警 | 稳定性危机，需冻结发布 | 触发事故流程 → 定位源头 → 冻结新功能发布 |

### 2.3 P1 告警（快速响应）

| 告警名称 | 触发条件 | 持续时间 | 影响 |
|----------|----------|----------|------|
| **API 延迟 P95 > SLO** | P95 延迟 > 2s | > 5 分钟 | 用户体验下降 |
| **错误率突增** | 错误率 > 3%（基线 2 倍） | > 5 分钟 | 部分请求失败 |
| **PostgreSQL 连接池耗尽** | 活跃连接 > max * 80% | > 2 分钟 | 新连接排队 |
| **Redis 内存 > 85%** | `used_memory / maxmemory` > 85% | > 5 分钟 | 缓存逐出加剧 |
| **Milvus 搜索延迟 P95 > 500ms** | 搜索延迟 > 500ms | > 5 分钟 | 检索响应变慢 |
| **磁盘使用率 > 85%** | 数据盘使用率 > 85% | > 10 分钟 | 写入失败风险 |
| **预生成命中率 < 40%** | 命中率低于 40% | > 30 分钟 | 用户等待 LLM 实时生成 |
| **内容校验失败率 > 5%** | 校验接口错误率 > 5% | > 5 分钟 | 生成内容可能未经校验 |
| **证书将在 7 天内过期** | TLS 证书剩余天数 < 7 | — | HTTPS 中断风险 |

### 2.4 P2 告警（计划内处理）

| 告警名称 | 触发条件 | 持续时间 |
|----------|----------|----------|
| CPU 使用率 > 80% | 主机 CPU > 80% | > 10 分钟 |
| 内存使用率 > 85% | 主机内存 > 85% | > 10 分钟 |
| 慢查询数量突增 | 每秒慢查询数 > 基线 3 倍 | > 15 分钟 |
| Milvus 分段数过多 | 分段数 > 100 | > 1 小时 |
| 日志增长速度异常 | 日日志增量 > 基线 3 倍 | > 1 小时 |
| 审计日志写入延迟 | 审计日志写入耗时 > 500ms | > 10 分钟 |
| Docker 容器异常重启 | 容器 24h 内重启 > 3 次 | — |

### 2.5 P3 告警（通知级别）

| 告警名称 | 触发条件 |
|----------|----------|
| 磁盘使用率 > 75%（预警） | 数据盘 > 75% |
| 证书将在 30 天内过期 | TLS 证书剩余天数 < 30 |
| Docker 镜像版本落后 | 运行版本 != 最新稳定版 |
| 依赖服务（上游 API）延迟升高 | 第三方 API P95 > 1s |
| 错误率轻微上升 | 错误率 > 1% 但 < 2% |

### 2.6 告警去重与抑制规则

| 规则 | 说明 |
|------|------|
| **抑制** | PostgreSQL 不可用 → 抑制所有依赖 PG 的服务告警（Milvus 查询、登录等） |
| **去重** | 同类型告警 15 分钟内不重复触发（Flapping 保护） |
| **静默** | 夜间 00:00-07:00 P2/P3 告警静默，次日 09:00 汇总 |
| **聚合** | 同容器同指标的多实例告警聚合成一条，附实例列表 |

### 3.2 延迟（Latency）

#### 监控目标
- API 端点 P50 / P95 / P99 响应时间
- LLM 流式生成的 TTFT（Time To First Token）和完成时间
- 数据库查询延迟
- 外部服务调用延迟（Milvus、Redis、LLM Provider）
- 组件间链路耗时（API → Service → LLM → Response）

#### 监控实现

| 组件 | 指标名 | Prometheus 采集方式 | 告警阈值 |
|------|--------|-------------------|----------|
| FastAPI | `http_request_duration_seconds`（分 endpoint、method、status） | FastAPI Prometheus middleware | P95 >= 2s → P1 |
| PostgreSQL | `pg_stat_activity` 查询延迟 | postgres_exporter | 慢查询 > 200ms |
| Redis | `redis_latency_*` | redis_exporter | P99 > 10ms |
| Milvus | `milvus_search_latency` | Milvus metrics port (9091) | P95 > 500ms |
| LLM Provider | `llm_request_duration_seconds` | 自定义 exporter（从应用代码上报） | P95 > 10s |
| Content Validation | `validation_request_duration_seconds` | FastAPI middleware | P95 > 1s |

#### 关键 Dashboard 面板
- **API 延迟热力图**：按 endpoint 分组，展示时间序列
- **LLM 生成耗时分解**：TTFT 和总耗时分开，标注模型版本
- **数据库查询延迟**：区分读/写，标注慢查询指纹
- **端到端用户请求链路**：从 HTTP 入口到最终响应分段展示

### 3.3 流量（Traffic）

#### 监控目标
- 每秒请求数（RPS）及按 endpoint / HTTP method 分布
- 每秒活跃用户数（DAU 实时估算）
- 关卡生成请求量（区分预生成命中 vs 实时 LLM 生成）
- LLM API 每分钟调用次数和 Token 消耗量
- Milvus 每秒查询次数（QPS）

#### 监控实现

| 组件 | 指标名 | 说明 |
|------|--------|------|
| FastAPI | `http_requests_total` | 总请求数，分 endpoint/status/method |
| FastAPI | `active_users_gauge` | 最近 5 分钟活跃用户数（自定义 metric） |
| Milvus | `milvus_qps` | Milvus 查询 QPS |
| LLM | `llm_token_total` | 输入/输出 Token 计数（LLM Provider 维度） |
| LLM | `llm_requests_total` | LLM API 调用次数 |
| Pregen | `pregen_hit_total` vs `pregen_miss_total` | 预生成命中/未命中计数器 |
| PostgreSQL | `pg_queries_per_sec` | 每秒查询数 |

#### 关键 Dashboard 面板
- **流量总览**：RPS 曲线，标注部署事件（流量突降 = 可能的故障）
- **LLM Token 消耗看板**：日/周/月 Token 使用趋势，按模型和功能拆分
- **预生成命中/未命中比率**：时间序列 + 当前命中率百分比
- **端点流量排名**：Top 10 最繁忙的 API 端点

### 3.4 错误率（Errors）

#### 监控目标
- HTTP 4xx / 5xx 错误率及分布
- 业务错误率（如 LLM 生成失败、内容校验拒绝、Milvus 搜索失败）
- 业务逻辑异常（如关卡生成数学正确性校验失败）
- 安全相关错误（登录失败、Token 过期、鉴权拒绝）
- 依赖服务调用失败率

#### 监控实现

| 组件 | 指标名 | 说明 |
|------|--------|------|
| FastAPI | `http_errors_total` | 分 status_code / endpoint |
| FastAPI | `http_error_ratio` | 错误率 = 5xx / 总请求数 |
| LLM | `llm_error_total` | LLM 调用失败（分超时/限流/服务端错误） |
| Milvus | `milvus_search_error_total` | 搜索失败次数 |
| Pregen | `pregen_failure_total` | 预生成任务失败次数 |
| Validation | `validation_rejected_total` | 内容校验拒绝次数（分安全/正确性/违规） |
| Auth | `auth_failure_total` | 登录/Token 验证失败（分原因） |
| Audit | `audit_chaining_error_total` | 审计日志链式哈希不匹配次数（安全事件） |

#### 关键 Dashboard 面板
- **错误率看板**：5xx / 4xx 分开，标注 P0/P1 告警阈值红线
- **错误按端点分布**：各端点的错误数量和比例
- **LLM 错误诊断**：分错误类型（超时/配额/模型不可用）、分 Provider
- **安全事件视图**：认证失败、内容校验拒绝、审计异常的趋势

### 3.5 饱和度（Saturation）

#### 监控目标
- CPU 使用率（整体 + 每个容器）
- 内存使用量及使用率
- PostgreSQL 连接池饱和度
- Redis 内存使用率 + key 数量 + OOM killer 风险
- Milvus 内存 + 磁盘 + 分段数 + 索引构建队列
- 磁盘 I/O 和 Inode 使用率
- 网络带宽使用率
- 容器 OOM 次数

#### 监控实现

| 组件 | 指标名 | 临界阈值 | 预警阈值 |
|------|--------|----------|----------|
| 主机 CPU | `node_cpu_seconds_total` | > 90% 持续 10 min | > 75% |
| 主机内存 | `node_memory_MemAvailable_bytes` | < 10% 可用 | < 20% 可用 |
| 主机磁盘 | `node_filesystem_avail_bytes` | < 15% 可用 | < 25% 可用 |
| Docker 容器 | `container_cpu_usage_seconds_total` | CPU limit 90% | CPU limit 75% |
| PostgreSQL 连接 | `pg_stat_activity.count` / max_connections | > 85% | > 70% |
| Redis 内存 | `redis_memory_used_bytes / maxmemory` | > 90% | > 80% |
| Redis keys | `redis_db_keys` | — | 日增量 > 10%（预警） |
| Milvus 分段数 | `milvus_segment_num` | — | > 100（需 compaction） |
| Milvus 索引队列 | `milvus_index_queue_length` | > 50 | > 20 |
| 磁盘 I/O | `node_disk_io_time_seconds_total` | IO util > 80% | IO util > 60% |
| Docker OOM | container 重启次数 | 3 次 / 24h | 1 次 / 24h |

#### 关键 Dashboard 面板
- **资源饱和度总览**：主机 + 所有容器 CPU / 内存 / 磁盘，按使用率排名
- **数据库连接池看板**：活跃 / 空闲 / 等待连接数
- **Redis 内存健康**：内存使用率 + key 数量 + 逐出策略效果
- **Milvus 分段管理**：分段数 + 索引状态 + 占用空间趋势
- **Docker 资源看板**：每个容器的资源限制和实际使用对比

## 4. 日常运维 Checklist

### 4.1 每日检查

```markdown
## 每日运维检查 (UTC+8 09:00)

### 服务健康
- [ ] Grafana 总览 Dashboard 所有服务健康状态 = GREEN
- [ ] P0/P1 告警列表为空（或已在处理中）
- [ ] 所有 Docker 容器正常运行（`docker ps --filter "status=exited"` 为 0）
- [ ] Milvus 所有集合/分区已加载（Milvus Dashboard 或 API）

### 资源水位
- [ ] 各主机磁盘使用率 < 80%
- [ ] PostgreSQL 连接池饱和度 < 70%
- [ ] Redis 内存使用率 < 80%
- [ ] 主机 CPU 和内存使用率 < 75%

### 数据与备份
- [ ] 昨夜 PostgreSQL 自动备份成功（检查备份日志）
- [ ] Redis RDB 最新快照时间在 24h 内
- [ ] 审计日志链式哈希完整性检查通过（每日自动校验脚本）

### 外部依赖
- [ ] LLM API（DeepSeek / 通义）服务状态正常
- [ ] CDN / 静态资源可正常访问
- [ ] 无 SSL 证书将在 14 天内过期

### 安全
- [ ] 最近 24h 认证失败次数正常（无密码爆破迹象）
- [ ] 内容校验服务拒绝率无异常突增
- [ ] Rate Limiter 未被大量触发
```

### 4.2 每周检查

```markdown
## 每周运维检查 (每周一 UTC+8 10:00)

### 趋势分析
- [ ] 周错误率趋势 -> 对比上周是否上升（上升 > 20% 需分析）
- [ ] 周 API 延迟 P95/P99 趋势 -> 对比上周
- [ ] 周 LLM Token 消耗趋势 -> 是否接近配额上限
- [ ] 周活跃用户趋势 -> 是否需容量规划

### 数据库
- [ ] PostgreSQL 慢查询日志分析（Top 10 慢查询）
- [ ] 检查 PostgreSQL VACUUM 和 ANALYZE 执行情况
- [ ] 检查 Milvus 分段数，是否需要手动 compaction
- [ ] Redis 大 Key 扫描（> 10MB 的 key）
- [ ] 检查 PostgreSQL 表膨胀情况

### 缓存与预生成
- [ ] Embedding 缓存命中率 -> 低于 80% 排查缓存策略
- [ ] 预生成池水位和命中率 -> 低于 50% 调整预生成策略
- [ ] Redis 缓存逐出统计（evicted_keys）-> 逐出过多需扩容

### 依赖与变更
- [ ] Docker 镜像版本检查 -> 基础镜像是否有安全更新
- [ ] 第三方依赖 CVE 检查（`uv lock --check`）
- [ ] 检查上周发布的变更是否引入新告警

### 可观测性自身
- [ ] Prometheus 数据保留和磁盘使用
```

### 5.2 事故响应角色与职责

| 角色 | 职责 | 备注 |
|------|------|------|
| **Incident Commander (IC)** | 全局协调，决策"做什么"，不参与具体排查 | 事故期间唯一决策者 |
| **Scribe** | 记录事故时间线和所有操作，确保不遗漏 | 使用共享文档实时记录 |
| **SME（主题专家）** | 在 IC 指挥下执行具体排查和修复 | API / DB / ML 各领域专家 |
| **Liaison** | 对外沟通，向管理者和用户通报进展 | 非技术沟通，避免 IC 被打扰 |

### 5.3 事故处理通用规则

| 规则 | 内容 |
|------|------|
| **止血优先** | 第一时间恢复服务（回滚/重启/降级），根本原因可后续排查 |
| **降级策略** | LLM 不可用 -> 返回缓存关卡；Milvus 不可用 -> PostgreSQL 关键词回退 |
| **操作记录** | 所有排查操作记录到事故频道，禁止口头沟通不留痕 |
| **更新频率** | P0 每 15 分钟更新一次状态；P1 每 30 分钟 |
| **升级机制** | 15 分钟无进展 -> IC 升级给 SRE Lead；30 分钟无进展 -> 通知技术负责人 |
| **事后复盘** | 事故关闭后 48 小时内完成 Postmortem 报告 |

### 5.4 常见事故降级策略

| 故障场景 | 降级方案 | 触发条件 |
|----------|----------|----------|
| LLM API 不可用 | 返回预生成缓存池中已有的关卡 | 连续 5 次 LLM 调用失败 |
| Milvus 不可用 | PostgreSQL 使用 `pgvector` 本地向量检索（精度降级） | Milvus health check 失败 |
| Redis 不可用 | 跳过缓存层，直接查询 PostgreSQL，限制并发 | Redis ping 失败 |
| PostgreSQL 不可用 | 返回只读模式的错误提示，拒绝写请求 | 主库不可用 |
| Content Validation 不可用 | 跳过自动校验，标记 `need_review=true`，人工事后审查 | 连续 3 次校验超时 |

## 6. 错误预算策略

### 6.1 错误预算定义

| SLO | 月度错误预算 | 计算公式 |
|-----|-------------|----------|
| 可用性 99.5% | 216 分钟 / 月 | 43200 min × (1 - 0.995) |
| 延迟 P95 < 2s | 按比例计算 | 请求总数 × 5% × 超时所占比例 |
| 错误率 < 1% | 按比例计算 | 总请求数 × 1% 可失败 |

**主要追踪**: 可用性错误预算（216 分钟 / 月）。

### 6.2 错误预算消耗速率计算

```
错误预算消耗率 = (当月已消耗分钟数 / 总预算分钟数) × 100%

每月可用总时间 = 30 × 24 × 60 = 43,200 分钟
每月错误预算 = 43,200 × (1 - 0.995) = 216 分钟 = 3.6 小时

日均预算 = 216 / 30 = 7.2 分钟/天
```

### 6.3 错误预算阈值与行动

| 消耗程度 | 状态 | 行动 |
|----------|------|------|
| **0% - 50%**（0 - 108 min） | 健康 | 正常发布，正常迭代 |
| **50% - 80%**（108 - 172.8 min） | 注意 | 提高告警敏感度，控制风险变更 |
| **80% - 100%**（172.8 - 216 min） | 危险 | 冻结高风险变更（LLM 模型切换、数据库迁移、架构重构） |
| **>= 100%** | 耗尽 | **冻结所有新功能发布**，仅允许稳定性修复，需 VP Engineering 审批 |

### 6.5 错误预算消耗事件类型归属

| 不可用类型 | 是否计入错误预算 | 说明 |
|-----------|-----------------|------|
| 计划内维护窗口 | 不计入 | 需提前 48h 通知用户，凌晨 02:00-05:00 |
| 基础设施故障 | 计入 | 硬件故障、云服务商故障 |
| 软件缺陷 | 计入 | 代码缺陷导致的故障 |
| 外部依赖（LLM API） | 计入 | 需要容错设计，但外部中断也影响可用性 |
| 人为失误 | 计入 | 配置错误、误操作 |
| 安全攻击 | 暂不计入 | 需区分是否可防御，事后评估 |
| 第三方服务商完全中断 | 协商计入 | 如 DeepSeek / 阿里云整体宕机 |

### 6.6 错误预算恢复策略

| 策略 | 说明 |
|------|------|
| **滚动恢复** | 每月 1 日重置错误预算 |
| **紧急追加** | 特殊情况下可由 VP Engineering 审批紧急追加（需记录原因） |
| **错误预算透支池** | 预留 20%（43.2 min）备用金，用于紧急安全修复的发布 |
| **错误预算快速恢复** | P0 事故修复 + 增加冗余 + 确认根因修复 = 返还 50% 透支（需审批） |

### 6.7 错误预算耗尽后的发布冻结规则

| 发布类型 | 是否允许 | 审批路径 |
|----------|----------|----------|
| 稳定性修复（bug fix） | 允许 | SRE Lead 审批 |
| 安全补丁 | 允许 | VP Engineering 审批 |
| 性能优化 | 允许（需论证可降低错误预算消耗） | SRE Lead + 架构师 |
| 新功能 | 禁止 | — |
| 模型升级/切换 | 禁止 | — |
| 数据库迁移 | 禁止 | — |
| 架构重构 | 禁止 | — |
| 配置变更（风险类） | 禁止 | — |
