### 1.1 系统角色定义

| 角色 | 描述 | 核心使用场景 | 技术交互方式 |
|------|------|-------------|-------------|
| **学生 (Student)** | 6-14 岁儿童，分初/中/高级三个阶段 | 每日冒险闯关、与 NPC 对话、收集角色卡 | HTTPS / SSE |
| **家长 (Parent)** | 学生监护人 | 查看学习报告、管理使用时间、处理数据请求 | HTTPS (仪表盘) |
| **教师 (Teacher)** | 班级管理者 | 查看班级整体进度、共性薄弱点 | HTTPS (仪表盘) |

### 1.2 外部系统依赖

| 外部系统 | 用途 | 协议 | 依赖等级 | 备选方案 |
|----------|------|------|---------|---------|
| **DeepSeek Chat API** | 关卡剧情生成、博士对话、引导话术 | HTTPS / REST (OpenAI 兼容) | **高** — 核心 AI 能力 | 通义千问（热切换） |
| **DeepSeek Embedding API** | 文本向量化（公共库+私有学情） | HTTPS / REST (OpenAI 兼容) | **高** — RAG 检索基础 | 本地 BGE-M3 模型 |
| **通义千问 API (备选)** | DeepSeek 不可用时的 LLM 备选 | HTTPS / REST (OpenAI 兼容) | **中** — 降级使用 | 无，只能切换 |
| **Web Speech API** | 叙事段落 TTS 语音播报 | 浏览器原生 API | **低** — MVP 辅助体验 | 云端 TTS (Azure/讯飞) |
| **MinerU** | 小学数学教材/教辅 PDF 解析入库 | 本地进程调用 | **低** — 知识库初始化阶段 | 人工录入 |
| **浏览器 Push API** | 家长通知（学习报告就绪/审核结果） | 浏览器 Push | **低** — 辅助功能 | 邮件通知 |

### 1.3 系统上下文边界规则

| 规则 | 说明 |
|------|------|
| 所有学生输入经 Pre-LLM 消毒后才进入 LLM Prompt | REQ-SEC-001 |
| 所有 LLM 输出经 Post-LLM 安全审核才推送前端 | REQ-SEC-002 |
| LLM API 超时上限 60s，超时降级为模板化内容 | REQ-NFR-P01 |
| 系统不存储学生原始输入中的 PII（脱敏后存储） | COPPA / PIPL 合规 |

### 2.2 容器间数据流

```
关卡生成请求流（同步）：
  Frontend → Gateway (JWT验证) → Decision Engine (选知识点+算难度)
    → RAG Constraint (KG查询+向量检索 → ConstraintPackage)
    → LLM Generation (Prompt组装+流式生成)
    → 逐段落: Content Validation (校验通过) → SSE Push → Frontend 渲染

关卡重生成路径：
  Content Validation (校验失败) → LLM Generation (重生成该段落, 最多1次)
    → 再次校验失败 → 模板化段落替换 → SSE Push

背景任务流（异步）：
  Background Worker → 关卡预生成池 → 预生成关卡 → PostgreSQL 关卡缓存表
     → 用户请求时缓存命中 → 直接返回(避过 LLM 调用)

数据回写流（同步/异步混合）：
  答题结果 → API Gateway → Decision Engine (掌握度更新)
    → PostgreSQL (画像更新) → Redis (热缓存刷新)
```

#### 3.1.1 决策引擎核心算法

```
知识点选择算法：
  候选集 = 掌握度矩阵[知识点] < 0.8          // 未掌握的知识点
         ∪ 间隔复习队列[知识点]                // 到复习时间的旧知识点
  优先级排序 = 掌握度_升序 × 遗忘系数_升序 × 进度依赖_权重
  输出 = 候选集[0]                            // 最高优先级

  排重规则：输出知识点的标签哈希 ∉ 最近 5 关标签集

难度计算算法：
  difficulty = mastery * 0.4 + forgot * 0.2 + emotion * 0.2 + progress * 0.2
  其中:
    mastery   = 1.0 - 该知识点正确率 (0~1)
    forgot    = min(1.0, days_since_last_review / 30)
    emotion   = max(0, 1.0 - recent_error_rate * 2)   // 连续错误→降难度
    progress  = min(1.0, current_stage / total_stages)

  目标: difficulty 调节使得 P(correct) ∈ [0.5, 0.7]
  调节方式: 难度分档(1-5), 每档对应预设的题目数值范围和推理步数

模式轮换 FSM：
  [英雄模式] --3次--> [导师模式] --1次--> [探索模式] --1次--> [英雄模式] (循环)
  例外: 回顾关卡固定为"重返故地"模式(英雄模式变体,剧情轻量化)
```

```typescript
  version: "2.0";

  /** ── 目标知识点 ── */
```

### 4.2 LLMProvider 抽象接口 (Python)

支持 DeepSeek ↔ 通义千问 热切换的抽象基类。

```python
"""
LLM Provider 抽象接口
支持 DeepSeek Chat ↔ 通义千问 运行时热切换
"""
from abc import ABC, abstractmethod
from dataclasses import dataclass, field
from typing import AsyncGenerator, Optional


@dataclass
class LLMMessage:
    """对话消息"""
    role: str  # "system" | "user" | "assistant"
    content: str


@dataclass
class LLMConfig:
    """LLM 调用配置"""
    model: str  # 模型名称 (如 "deepseek-chat", "qwen-turbo")
    temperature: float = 0.7
    max_tokens: int = 4096
    top_p: float = 0.95
    frequency_penalty: float = 0.0
    presence_penalty: float = 0.0
    stop: Optional[list[str]] = None

    # Prompt Cache 相关
    enable_prompt_caching: bool = True
    cache_seed: Optional[int] = None  # 确定性缓存种子


@dataclass
class LLMResponse:
    """LLM 同步响应"""
    content: str
    model: str
    usage: dict  # {"prompt_tokens": N, "completion_tokens": N}
    cached: bool = False  # 是否命中 Prompt Cache


@dataclass
class LLMStreamChunk:
    """LLM 流式响应片段"""
    content: str
    is_final: bool = False
    usage: Optional[dict] = None  # 仅在 is_final=True 时有值


class LLMProvider(ABC):
    """
    LLM 提供者抽象基类

    所有 LLM API 调用通过此抽象层，支持运行时热切换。

    使用示例:
        provider: LLMProvider = DeepSeekProvider(api_key=...)
        # 运行时切换:
        provider = TongyiProvider(api_key=...)
    """

    @abstractmethod
    async def generate(
        self,
        messages: list[LLMMessage],
        config: LLMConfig,
        **kwargs,
    ) -> LLMResponse:
        """
        同步生成完整响应。

        Args:
            messages: 消息列表 (system + user + assistant 轮次)
            config: 生成配置

        Returns:
            LLMResponse: 包含生成内容和用量统计

        Raises:
            LLMProviderError: API 调用失败
            LLMTimeoutError: 超时
            LLMRateLimitError: 限流
        """
        ...

    @abstractmethod
    async def generate_stream(
        self,
        messages: list[LLMMessage],
        config: LLMConfig,
        **kwargs,
    ) -> AsyncGenerator[LLMStreamChunk, None]:
        """
        流式生成响应片段。

        Yields:
            LLMStreamChunk: 内容片段，is_final=True 时机包含用量统计

        Raises:
            LLMProviderError: API 调用失败
            LLMTimeoutError: 超时
            LLMRateLimitError: 限流
        """
        ...

    @abstractmethod
    async def embed(
        self,
        texts: list[str],
        model: Optional[str] = None,
        **kwargs,
    ) -> list[list[float]]:
        """
        文本向量化。

        Args:
            texts: 待向量化的文本列表
            model: 向量模型名称 (默认为 provider 默认模型)

        Returns:
            list[list[float]]: 向量列表，每个向量维度与模型有关
        """
        ...

    @abstractmethod
    async def count_tokens(
        self,
        text: str,
        model: Optional[str] = None,
    ) -> int:
        """
        估算文本的 Token 数量（用于 Prompt Cache 管理和成本估算）。

        Args:
            text: 待估算文本
            model: 模型名称 (不同模型 Tokenizer 不同)

        Returns:
            int: 估算的 Token 数
        """
        ...

    # --- 运行时管理 ---

    @abstractmethod
    async def health_check(self) -> bool:
        """
        健康检查 - 确认 API 是否可用。

        Returns:
            bool: True 表示可用
        """
        ...

    @abstractmethod
    def get_model_list(self) -> list[str]:
        """
        获取当前 provider 支持的模型列表。
        """
        ...

    @property
    @abstractmethod
    def provider_name(self) -> str:
        """Provider 名称, 如 "deepseek", "tongyi" """
        ...


class LLMProviderError(Exception):
    """LLM Provider 调用异常基类"""
    def __init__(self, message: str, provider: str, status_code: Optional[int] = None):
        self.provider = provider
        self.status_code = status_code
        super().__init__(f"[{provider}] {message}")


class LLMTimeoutError(LLMProviderError):
    """LLM 调用超时"""
    pass


class LLMRateLimitError(LLMProviderError):
    """LLM 限流"""
    pass
```

```
    │                          │                       │                    │                      │                       │                    │
```

### 5.2 时序约束

| 阶段 | 组件 | 最大耗时 | 说明 |
|------|------|---------|------|
| ① 决策引擎 | Decision Engine | 50ms | 纯算法计算，无外部 I/O |
| ② KG + 向量检索 | RAG Constraint | 500ms (P99) | 含 PostgreSQL + Milvus + Redis 调用 |
| ③ LLM 首次生成 | LLM Generation | 20s (P95) | 完整 6 段，含 Prompt Cache 命中 |
| ④ 逐段校验(×6) | Content Validation | 500ms × 6 = 3s | 每段独立校验 |
| ⑤ 首段到达前端 | SSE Push | 3s (P95) | LLM 首 token + 首段校验 |
| ⑥ 全部完成 | 端到端 | 60s (P95) | 含 1 次重生成预算 |

### 5.3 降级策略

| 故障场景 | 降级行为 | 用户体验 |
|----------|---------|---------|
| LLM API 超时(>60s) | 使用 PostgreSQL 缓存的模板化关卡 | 关卡内容较平淡但可用 |
| LLM API 不可用 | 从预生成池取关卡 | 无感知（延迟略增） |
| 单段落校验失败(1次) | 重生成该段落 | 该段落延迟增加 10-15s |
| 单段落校验失败(2次) | 安全模板库替换 | 该段内容模板化，其余 5 段正常 |
| 连续 3 段替换 | 整关降级为模板 | 降低预期，保证可用 |
| Embedding API 不可用 | 仅 KG 检索（无向量语义），降低召回率 | RAG 效果下降但不中断 |

## 6. 关卡预生成池架构

### 6.1 问题定义

500 并发用户的 LLM API 瓶颈：
- 关卡生成需要 1 次 DeepSeek Chat API 调用（非流式 ~15s, 流式 ~20s）
- DeepSeek API 并发上限受限（单账号 ~10-30 RPM）
- 高峰期（放学后 17:00-19:00）请求密度最大

### 6.3 预生成参数

| 参数 | 值 | 说明 |
|------|-----|------|
| 池大小目标 | 50-100 个 ready 关卡 | 缓冲峰值需求 |

### 6.4 预生成策略

```
预生成知识点选择：
  1. 从课标中选出当前活跃年级（根据登录用户分布动态调整）
  2. 每个活跃年级选 3-5 个知识点
  3. 知识点优先级 = 访问频率 * 0.6 + 生成耗时 * 0.2 + 最近生成时间 * 0.2

预生成模拟画像：
  使用"典型画像模板"（而非真实用户数据）：
    Level 1: 中等掌握度(0.5), 零错题, 英雄模式
    Level 2: 薄弱掌握度(0.3), 有错题, 导师模式
    Level 3: 高掌握度(0.8), 无错题, 探索模式
  每种知识点 × 每种典型画像 = 预生成的组合

池水位管理：
  目标水位 = 50 (ready 关卡数)
  低水位(<20): 紧急补充, 每 1 分钟 Batch
  正常水位(20-80): 常规补充, 每 2 分钟 Batch
  高水位(>80): 暂停预生成
```

### 6.5 与 500 并发的匹配关系

```
用户请求高峰模型:
  高峰期: 17:00-19:00 (放学后)
  假设 500 DAU, 高峰期 200 人同时在线
  每人每 30 分钟请求 1 次关卡生成
  高峰期请求率 ≈ 200 / 1800s ≈ 0.11 req/s → 约 6.7 req/min

预生成池供给能力:
  10 Workers × 每 15 分钟 1 个关卡 = 10 / 15 = 0.67 关卡/分钟
  高峰期需求 ≈ 6.7 关卡/分钟
  命中率 60% → 需要实时生成 6.7 × 40% = 2.7 关卡/分钟
  实时生成能力: 1 LLM 调用 ≈ 20s → 1 通道可生成 3 关卡/分钟
  → 实时生成 2.7 关卡/分钟 < 3 关卡/分钟 ✅

结论: 预生成池 + 实时生成组合可满足 500 并发需求
      关键约束在 DeepSeek API 的 RPM 限制, 而非计算能力
```

## 7. 架构决策记录 ADR-008~015

### ADR-008: 架构模式选择

- **决策**: 模块化单体（Modular Monolith） + CQRS 逻辑分离
- **依据**: 1 人团队 + AI 协作，微服务运维成本过高；CQRS 逻辑分离读写模型
- **来源**: `knowledge/10-软件工程基础/03-架构模式.md` 决策树
- **反面考虑**: 微服务拆分 -> 评估后否决（团队规模 < 3 人，无独立扩展需求）

### ADR-009: 三层解耦架构

- **决策**: Decision Engine (算法) → RAG Constraint (检索) → LLM Generation (生成)
- **依据**: SRS v2.1 架构约束 + 需求优化方案 §4 核心架构优化
- **反面考虑**: LLM 全链路决策 → 否决（不可靠、不可解释、成本高）

### ADR-010: 知识图谱存储

- **决策**: PostgreSQL 关系表 + 递归 CTE 查询
- **依据**: 小学数学 200-400 节点规模，远未达到专用图数据库需求
- **正面**: 零额外运维，自动化备份，ORM 统一管理

### ADR-011: LLM Provider 抽象接口

- **决策**: 定义 LLMProvider ABC，支持 DeepSeek ↔ 通义千问 运行时热切换
- **依据**: 降低 DeepSeek API 不可用风险（R4）
- **实现**: 工厂模式注册 Provider，配置驱动切换，运行时优雅迁移

### ADR-012: 段落级流式输出

- **决策**: LLM 一次生成完整 6 段 JSON → 逐段提取 → 逐段校验 → 逐段 SSE 推送
- **依据**: REQ-COM-003 + REQ-NFR-P06（首段 < 3s）
- **技术选择**: SSE 而非 WebSocket（MVP 单向推送足够，WebSocket v3.0 多人模式引入）
- **替代方案**: 每次只生成 1 段 → 6 次 LLM 调用 → 否决（成本 6x, 且丧失剧情连贯性）

### ADR-013: 关卡预生成池

- **决策**: Background Worker + PostgreSQL 关卡缓存表 + apscheduler 调度
- **依据**: REQ-NFR-P05 (500 并发) + DeepSeek API RPM 限制
- **设计要点**: constraint_hash 匹配预生成，典型画像模板而非真实用户数据

### ADR-014: 内容校验重生成限制

- **决策**: 校验失败最多重生成 **1 次**（非 SRS v2.0 的 3 次）
- **依据**: P95 < 60s 性能目标要求，3 次重生成 → 单关时间可能突破 60s
- **降级**: 第 2 次失败 → 段落级模板替换（预审核安全模板库）
- **变更记录**: 修正 SRS v2.1 §4.8 REQ-VLD-001 中的"最多重生成 3 次"为 1 次

### ADR-015: 前端 SSE Client 架构

- **决策**: 浏览器原生 EventSource API + 自定义重连逻辑 + 段落级状态机
- **依据**: EventSource 支持自动重连（Last-Event-ID），无需额外库
- **UI 映射**: 每个段落 3 态（generating → validating → rendering → done），动画驱动

## 附录 B: LLMProvider 实现架构

```python
"""
LLM Provider 工厂与注册模式
"""
from enum import Enum
from typing import Protocol


class ProviderType(str, Enum):
    DEEPSEEK = "deepseek"
    TONGYI = "tongyi"  # 通义千问


class LLMProviderRegistry:
    """
    Provider 注册中心
    支持运行中热切换
    """

    _providers: dict[ProviderType, type[LLMProvider]] = {}

    @classmethod
    def register(cls, provider_type: ProviderType, provider_class: type[LLMProvider]):
        cls._providers[provider_type] = provider_class

    @classmethod
    def create(cls, provider_type: ProviderType, **kwargs) -> LLMProvider:
        if provider_type not in cls._providers:
            raise ValueError(f"Unknown provider: {provider_type}")
        return cls._providers[provider_type](**kwargs)


# Provider 路由配置
LLM_ROUTING_CONFIG = {
    "level_generation": {    # 关卡生成 — 需要最高质量
        "primary": ProviderType.DEEPSEEK,
        "primary_model": "deepseek-chat",
        "fallback": ProviderType.TONGYI,
        "fallback_model": "qwen-turbo",
        "timeout": 60.0,
    },
    "dialogue": {            # 博士对话引导 — 低延迟优先

        "fallback": None,    # 无备选，降级为模板

    },
    "embedding": {           # 向量化 — 仅 DeepSeek Embedding

}
```
