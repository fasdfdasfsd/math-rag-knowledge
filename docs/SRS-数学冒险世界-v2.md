## 1. 引言

### 1.1 目的

本文档定义"数学冒险世界"（Math Adventure World）的软件需求规格。产品定位为：**面向 6-14 岁儿童的 AI 驱动游戏化数学学习平台**，通过"数学概念拟人化 + 冒险叙事 + 自适应难度"三大支柱，将数学从"需要完成的任务"转化为"孩子想做的事"。

本文档用于：
- 指导后续架构设计和开发实现
- 作为测试用例编写的需求基线
- 作为验收交付的需求依据

### 1.2 文档约定

| 约定 | 说明 |

| 需求标识 | `REQ-{模块代码}-{三位序号}`，如 `REQ-NRT-001` |

### 1.4 项目范围

**In Scope**：
- 沉浸式数学冒险关卡生成与交互系统
- 三层解耦架构（决策引擎 + RAG 约束层 + LLM 生成层）
- 数学概念拟人化 NPC 角色系统
- 自适应难度调节引擎
- 三种闯关模式（英雄/导师/探索）轮换
- 知识图谱增强 RAG 检索
- 用户学情追踪与长期记忆
- 家长/教师仪表盘

**Out of Scope（v2.0）**：
- 多人在线协作/竞争模式
- 自建语音识别（ASR）
- 第三方 LMS 深度集成（如 Canvas/Google Classroom）
- 移动原生 App（v2.0 为 Web-first，PWA 支持）

#### 2.1.2 竞品格局中的位置

基于 23 款产品全景分析，数学冒险世界定位在 **第一梯队（游戏化深度融合）+ 第三梯队（叙事情感化）** 的交汇蓝海：

- 市面上"AI讲题"已成红海（好未来/猿辅导/作业帮均主打）
- **没有任何产品**做到了"每个孩子专属的、永不结局的、RAG+LLM驱动的数学冒险世界"
- 叙事+游戏化的第一梯队产品（Prodigy/Mathseeds）依赖预设内容，无法做到千人千面

### 4.2 REQ-NRT：叙事引擎

#### ⭐ REQ-NRT-001：六段式关卡叙事模板（Must have）

**竞品 DNA 对齐**：答题必须有后果 + 错误不能是惩罚 + 长线成长有空间

**用户故事**：
> As a 学生，I want 每次冒险遵循统一的六段式叙事流程（入口→到达→冲突→解题→胜利→钩子），
> So that 每次体验都像一集动画片，有开头有结尾有悬念，我期待明天的冒险。

**验收标准**：
```
Scenario 1: 完整六段式流程
  Given 决策引擎已确定本关知识点和难度
    And RAG 约束包已组装完成
  When LLM 生成关卡内容
  Then 输出必须包含以下六个段落（每段有明确的叙事功能）：
    ① 入口段：传送门启动 → 今日目的地预告 → 博士抛出悬念
    ② 到达段：载具变形着陆 → 场景感官描写（≥3 种感官） → 引导式提问
    ③ 冲突段：数学概念以 NPC 形态登场 → 冲突包装（战争/失踪/失衡/诅咒/谜题）
    ④ 解题段：文学线+数学线双线推进 ≥ 2 轮交互
    ⑤ 胜利段：冲突解决 → 博士情感肯定 → 知识总结 → 获得奖励
    ⑥ 钩子段：新线索出现 → 悬念尾句 → 明日预告
  And 每段不超过 200 中文字（保持注意力）
  And 全文不使用"请计算"、"正确答案是"、"你做错了"、"再试一次"
```

```
Scenario 2: 禁止表述检查
  Given LLM 生成完成
  When 后置校验模块检查文本
  Then 如果检测到禁止表述（"请计算"/"正确答案是"/"你做错了"/"再试一次"）
    重生成相关段落直至通过
```

#### REQ-NRT-002：感官描写多样性（Must have）

**竞品 DNA 对齐**：ST Math "零语言教数学" — 大量动画演示减少文字依赖

**用户故事**：
> As a 学生，I want 每次关卡场景有新鲜的多感官描写，
> So that 我不会觉得场景"和上次一样"，保持探索的新鲜感。

**验收标准**：
```
Scenario 1: 感官描写覆盖
  Given 已生成 N 个历史关卡的场景描写标签
  When 生成新关卡场景
  Then 新场景必须包含 ≥ 3 种感官（视/听/触/嗅）
  And 感官描写标签不与最近 5 次历史记录重复
```

#### REQ-NRT-003：问题链自然引出（Should have）

**竞品 DNA 对齐**：Khanmigo "苏格拉底问答法" — 绝不直接给答案

**用户故事**：
> As a 学生，I want 数学问题从剧情中自然浮现（NPC 提问/环境线索/角色困境），
> So that 我不觉得"突然跳到做题"，而是为了推动剧情主动思考。

**验收标准**：
```
Scenario 1: 引导式提问
  Given 学生在解题段遇到数学障碍
  When 学生犹豫 > 10 秒或输入错误答案
  Then 博士/NPC 以引导式提问回应（如"你觉得这一步发生了什么？"）
  And 不直接给出正确答案
  And 不重复相同的引导语（最近 3 次引导语不重复）
```

---

#### REQ-NRT-004：疑案包装（Must have）

**竞品 DNA 对齐**：李毓佩"疑案代替应用题" — 每道题必须有案件包装

**用户故事**：
> As a 学生，I want 每道数学题被包装成一个"案件"或"谜题"，
> So that 我不是在"做题"，而是在"破案"——我是数学侦探。

**验收标准**：
```
Scenario 1: 题目案件包装
  Given 一道数学题需呈现给学生
  When 关卡剧情包装该题目
  Then 题目必须嵌入以下包装之一：
    - 案件侦破（"分数王国的蛋糕被偷了，3/4和1/2谁的嫌疑更大？"）
    - 冲突调解（"真分数和假分数争执谁更重要，你来评评理"）
    - 谜题解锁（"只有解开这个几何密码，才能进入下一层塔"）
    - 诅咒破解（"零国王中了诅咒，只有完成这些等式才能解除"）
  And 题目数值保留数学正确性
```

### 4.3 REQ-ADP：自适应难度引擎

#### ⭐ REQ-ADP-001：难度自适应算法（Must have）

**竞品 DNA 对齐**：DreamBox "分析过程而非结果" + Duolingo Birdbrain 双层解耦

**用户故事**：
> As a 系统，I want 基于学生历史答题数据和遗忘曲线动态调节难度，
> So that 学生始终处于"甜蜜点"（P(正确) ∈ [0.5, 0.7]），既不沮丧也不无聊。

**验收标准**：
```
Scenario 1: 难度计算
  Given 学生的知识点掌握度矩阵
    And 遗忘曲线衰退系数
    And 最近 10 次答题的情绪状态（连续错误→降难度）
    And 冒险进度因子（不跨越未解锁地图）
  When 计算下一关目标难度
  Then 难度值 = f(掌握度×0.4, 遗忘×0.2, 情绪×0.2, 进度×0.2)
  And P(正确) 预测值落在 [0.5, 0.7] 区间
```

```
Scenario 2: 错误后调整（Zearn 模式）
  Given 学生在当前知识点连续答错 2 次
  When 系统调整策略
  Then 不出更简单的回退题目
  And 而是提供同级别的"换个角度"题目（不同题型/不同包装/不同数值）
  And 最多 3 次同级别尝试后，触发博士介入（引导式提示）
```

```
Scenario 3: 过程分析数据采集（DreamBox 模式）
  Given 学生在答题交互中
  When 系统记录答题过程
  Then 必须采集以下数据点：
    - 单题耗时（秒）
    - 尝试次数（提交了几次答案）
    - 步骤间停顿位置（如果分步答题）
    - 求助使用次数（点击提示/跳过按钮）
  And 这些数据参与下次难度计算
```

#### REQ-ADP-002：IRT 难度参数模型（Should have — v2.1）

**用户故事**：
> As a 系统管理员，I want 基于 IRT（项目反应理论）参数化题库，
> So that 每道题的区分度、难度、猜测系数可量化，难度匹配更精确。

**验收标准**：
```
Scenario 1: IRT 三参数模型
  Given 一道题已有 ≥ 50 次答题记录
  When 计算 IRT 参数
  Then 输出：a(区分度), b(难度), c(猜测系数)
  And 参数用于后续难度匹配决策
```

> **决策说明**：MVP 阶段使用正确率滑动窗口 + 遗忘曲线计算（v2.0），在积累足够答题数据后（≥ 10,000 次）升级为完整 IRT/BKT 模型（v2.1）。

### 4.4 REQ-NPC：NPC 角色系统

#### ⭐ REQ-NPC-001：数学概念拟人化 NPC（Must have）

**竞品 DNA 对齐**：角色灵魂共鸣 — Numberblocks 数字人格化 + 李毓佩"零国王/分数王子" + Prodigy 宠物收集

**用户故事**：
> As a 学生，I want 数学概念是有性格、有故事的角色（NPC），
> So that 我不是"学习分数"，而是"和分数王子交朋友"，角色让我每天想回来。

**验收标准**：
```
Scenario 1: 概念-NPC 映射
  Given 数学概念分类体系（数/形/量/关系）
  When 系统初始化 NPC 角色池
  Then 每个核心数学概念对应一个 NPC，包含：
    - 名称（如"零国王"/"分数王子"/"三角形骑士"/"小数精灵"）
    - 性格特征（≥ 3 个形容词）
    - 视觉描述（外观、标志物）
    - 口头禅/说话风格
    - 与其他 NPC 的关系（盟友/对手/家人）
```

### 4.6 REQ-KG：知识图谱增强检索

#### ⭐ REQ-KG-001：KG 增强 RAG 检索（Must have）

```
    - 向量检索：语义相似的 Top-5 知识点
    - KG 前置查询：K 的所有前置依赖 → 确认学生已掌握
    - KG 后续查询：K 的后续扩展知识点 → 为下一关铺垫
    - KG 关联查询：K 的关联易错点 → 针对性强化
  And 组装的"关卡约束包"不包含未掌握的前置依赖知识
```

| REQ-NFR-P05 | 并发用户数 | 支持 500 并发（含 LLM 请求排队 + 关卡预生成池） | 压力测试 |

| REQ-NFR-P08 | RTO（恢复时间目标） | ≤ 4h | 恢复演练 |

> **性能设计说明**：P95/P99 选择基于业务特性——LLM 生成使用 P95（长尾由 API 外部抖动导致，不可控），RAG 检索和答题反馈使用 P99（内部可控）。重生成限制为 **1 次**（非 3 次），校验失败段落降级为模板替换。500 并发通过"关卡预生成池"缓冲 LLM API 峰值。

## 7. 关键决策记录

| # | 决策 | 选择 | 依据 |

| D6 | 叙事策略 | 六段式统一模板 + 参数池驱动 | 神奇校车/树屋/李毓佩/旅行四大作品 DNA 融合 |

> **D10 说明**：前端框架选择（Vue3 / React / Flutter）取决于团队技能、目标平台（Web-only vs 跨平台）、PWA 支持度等因素，留待阶段 2（架构设计）由架构师角色主导决策。
