## 1. 引言

### 1.1 目的

本文档旨在明确在线教育平台的核心功能需求与非功能需求，为系统设计、开发、测试及验收提供依据。平台支持课程发布、直播教学、作业提交与批改等核心业务流程。

### 1.2 文档约定

- 需求编号格式：`FR-XXX`（功能需求）、`NFR-XXX`（非功能需求）、`UC-XXX`（用例）、`API-XXX`（接口定义）
- 优先级定义：P1（必须实现）、P2（重要实现）、P3（可选实现）
- 所有接口遵循 RESTful 风格，请求/响应体为 JSON 格式，时间采用 ISO 8601 标准
- 文件上传使用 `multipart/form-data`，大小限制：作业附件 ≤10MB，课程视频 ≤500MB
- API 基本路径：`https://api.example.com/v1`
- 认证方式：Bearer Token（JWT），Header 中携带 `Authorization: Bearer <token>`

## 2. 综合描述

### 2.1 产品前景

本在线教育平台面向K12及成人职业教育场景，旨在提供一站式在线教学解决方案。平台通过课程管理、直播教学、作业批改等核心功能，连接教师与学生，实现高效、便捷的远程教学体验。初期目标用户规模为10万注册用户，并发直播观看人数不超过5000人。

| 直播教学   | 创建直播间、推流、互动、录制回放                   | P1     |
| 作业管理   | 布置作业、提交作业、批改作业、成绩反馈             | P1     |
| 用户管理   | 注册登录、角色管理、教师资质审核                   | P1     |
| 通知系统   | 站内信、邮件、短信、App推送                        | P2     |
| 数据统计   | 教师看板、管理员看板                               | P2     |
| 权限控制   | 角色权限、数据隔离                                 | P1     |

### 2.3 用户特征

| 用户角色 | 特征描述                                                     | 使用场景     |
|----------|--------------------------------------------------------------|--------------|
| 学生     | 具备基本网络操作能力，需观看直播、提交作业、查看成绩         | 日常学习     |
| 教师     | 需通过资质审核，可创建课程、直播教学、批改作业               | 教学管理     |
| 管理员   | 负责平台运营，审核课程和教师资质，查看统计数据               | 平台管理     |

### 2.4 运行环境

| 环境类型     | 要求                                                           |
|--------------|----------------------------------------------------------------|
| 浏览器       | Chrome、Firefox、Safari、Edge 最新两个版本                     |
| 移动端       | 响应式设计，支持主流移动浏览器                                 |
| 网络         | 支持 HTTPS 加密传输，直播需稳定网络                           |

### 2.5 限制

- 直播功能依赖第三方CDN或云服务
- 自动批改仅限选择题、判断题
- 文件上传大小限制：作业附件 ≤10MB，课程视频 ≤500MB
- 系统可用性要求：99.9%（全年停机 ≤8.76小时）

### 2.6 假设与依赖

- 用户具备基本网络操作能力
- 直播功能依赖第三方CDN或云服务
- 作业批改支持人工批改和自动批改（客观题）
- 平台初期用户规模为10万注册用户，并发直播观看人数不超过5000人

## 3. 外部接口需求

### 3.1 用户界面

- 支持主流浏览器（Chrome、Firefox、Safari、Edge 最新两个版本）
- 支持移动端自适应（响应式设计）
- 直播播放器支持自动切换清晰度（流畅/高清/超清）
- 所有操作需有明确的成功/失败反馈提示
- 关键操作（如删除课程、提交作业）需二次确认

### 3.2 硬件接口

- 服务器：支持微服务架构，各模块可独立部署升级
- 存储：对象存储（MinIO / 阿里云 OSS）用于文件存储
- 网络：支持 HTTPS 加密传输，直播流需防盗链

### 3.3 软件接口

#### 3.3.1 认证与鉴权

所有业务API（除用户注册/登录外）均需携带有效的 JWT Token。Token 由认证服务签发，有效期 24 小时，支持刷新（Refresh Token 有效期 7 天）。请求头格式：

```
Authorization: Bearer <access_token>
```

#### 3.3.2 REST API 设计规范

- 资源命名使用复数名词
- 版本号置于 URL 路径：`/v1`
- 分页参数：`page`（页码，从1开始）、`limit`（每页条数，默认20，最大100）
- 排序参数：`sort`（字段名），`order`（`asc` 或 `desc`）
- 成功响应格式：

```json
{
  "code": 0,
  "message": "success",
  "data": { ... }
}
```

- 错误响应格式：

```json
{
  "code": 4XX,
  "message": "错误描述",
  "details": {} 
}
```

- 常见 HTTP 状态码及业务错误码（见附录 7.3）

#### 3.3.4 文件上传

- 端点：`POST /v1/files/upload`
- 请求：`multipart/form-data`，字段 `file`
- 响应：返回文件访问URL和文件标识符
- 限制：作业附件 ≤10MB，课程视频 ≤500MB（服务端校验）
- 上传的文件需通过内容安全检测（病毒扫描、敏感信息检测），合格后方可使用

#### 3.3.5 WebSocket 接口

- 直播聊天室：`wss://api.example.com/v1/live-rooms/{roomId}/ws?token=<jwt>`
- 心跳间隔：30秒，三次无心跳由服务端主动断连
- 消息协议：JSON，包含 `type`（如 `chat`, `hand_raise`, `teacher_action`）及 `data` 字段

#### 3.3.6 消息队列

- 异步事件主题：
  - `course.created` — 课程创建通知管理员审核
  - `live.starting` — 直播开始前15分钟通知已报名学生
  - `submission.submitted` — 学生提交作业通知教师
  - `submission.graded` — 作业已批改通知学生
  - `audit.*` — 审核相关事件
- 保证至少一次投递，消费端需实现幂等

### 3.4 通信接口

- 所有数据传输使用 HTTPS 加密
- 直播流使用 WebRTC / RTMP / HLS 协议，播放地址需包含动态加密的 token，有效期 2 小时，防止盗链
- 实时通信使用 WebSocket 协议，连接时需校验 JWT
- 消息队列使用 Kafka 或 RabbitMQ 实现异步事件传递

### 4.2 用例描述

#### 用例1：教师发布课程

**用例编号**：UC-001  
**参与者**：教师  
**前置条件**：教师已登录，且账号通过资质审核  
**后置条件**：课程创建成功，状态为“待审核”或“已上架”（若无需审核）

**基本流**：
1. 教师点击“创建课程”按钮。
2. 系统显示课程信息表单（标题、简介、封面、分类、价格）。
3. 教师填写必填项（标题、简介、分类），可选填封面和价格。
4. 教师点击“保存草稿”或“提交审核”。
5. 系统校验必填项是否完整，并校验输入合法性（标题长度1-100字符，简介1-2000字符，分类来自预定义分类表）。
6. 系统保存课程信息，生成课程ID。
7. 若选择“提交审核”，系统将课程状态设为“待审核”，并通知管理员。
8. 系统显示课程创建成功提示，并跳转至课程详情页。

**备选流**：
- **2a**：教师未填写必填项，点击提交时系统提示“请填写标题/简介/分类”，并高亮缺失字段。
- **5a**：输入内容不符合规范（如超长、包含非法字符），系统提示具体错误信息，并要求修正。
- **6a**：系统保存失败（如数据库异常），提示“创建失败，请稍后重试”，并记录错误日志；支持用户再次提交，表单数据保留。
- **7a**：若平台配置为“无需审核”，则课程直接设为“已上架”状态。
- **7b**：通知管理员失败（消息队列不可用），系统应记录失败日志，并采取补偿机制（如定时任务扫描待审核课程再次通知），但课程创建操作不应回滚。

#### 用例2：学生提交作业

**用例编号**：UC-002  
**参与者**：学生  
**前置条件**：学生已登录，且已报名该课程；教师已布置作业且未过截止时间  
**后置条件**：作业提交成功，状态变为“待批改”

**基本流**：
1. 学生进入课程详情页，点击“作业”标签。
2. 系统显示作业列表（含作业标题、截止时间、状态）。
3. 学生点击未提交的作业，进入作业详情页。
4. 系统显示作业要求（题目、附件、分值说明）。
5. 学生根据题型作答：
   - 客观题：直接选择答案。
   - 主观题：在文本框输入文字，或上传文件（支持PDF、Word、图片）。
6. 学生点击“提交作业”按钮。
7. 系统校验：
   - 是否超过截止时间（服务端二次校验，防止客户端篡改时间）。
   - 客观题是否全部作答。
   - 文件大小是否≤10MB，文件类型是否在允许白名单内。
   - 防重复提交：判断 60 秒内同一作业是否存在“提交中”的同学生ID提交记录。
8. 系统保存作业答案，状态更新为“待批改”。
9. 系统显示“提交成功”提示，并异步通知教师有新作业待批改（消息队列）。
10. 若作业含有客观题，系统在提交后立即进行自动批改（异步），提前更新客观题得分，标记待教师复核。

**备选流**：
- **3a**：作业已过截止时间，系统提示“作业已截止，无法提交”，并禁用提交按钮；即使客户端绕过，服务端也返回错误码 4000。
- **7a**：文件超过10MB或类型不允许，系统提示“文件大小不符合要求，请压缩后上传”或“不支持的文件类型”。
- **7b**：客观题未全部作答，系统提示“请完成所有选择题后再提交”。
- **7c**：检测到重复提交（60秒内相同作业ID和用户ID的提交），提示“您已提交，请勿重复操作”。
- **8a**：系统保存失败（如数据库异常或文件存储失败），提示“提交失败，请稍后重试”，并保留已填写内容。重试时采用幂等键避免重复创建提交记录。
- **9a**：教师通知发送失败不影响提交结果，后续可通过站内信列表查看。

#### 用例3：教师批改作业

**用例编号**：UC-003  
**参与者**：教师  
**前置条件**：教师已登录，且该课程下有待批改的作业  
**后置条件**：作业批改完成，学生可查看分数和评语

**基本流**：
1. 教师进入课程管理后台，点击“作业批改”菜单。
2. 系统显示待批改作业列表（学生姓名、提交时间、作业标题），支持分页和筛选。
3. 教师点击某份作业，进入批改页面。
4. 系统展示学生提交的答案（客观题自动显示系统判分，主观题显示原文及附件）。
5. 教师逐题批改：
   - 客观题：系统已自动评分，教师可复核并修改（如果误判）。
   - 主观题：教师输入分数（0~题目满分）和评语。
6. 教师点击“提交批改”按钮。
7. 系统校验所有题目均有评分，且总分不超过作业总分。
8. 系统计算总分，保存批改结果，确保原子性。
9. 系统更新作业状态为“已批改”，并异步通知学生查看结果。
10. 系统返回批改列表或下一份待批改作业。

**备选流**：
- **3a**：教师选择“打回重做”，系统要求学生重新提交，作业状态变为“待重做”，并通知学生，同时携带教师评语。
- **5a**：教师输入的分数超出范围（如大于满分），或为负数，系统提示“分数无效，请输入0~{满分}之间的值”，阻止提交。
- **6a**：提交时若因网络原因后台已处理但前端未收到响应，教师再次提交时，服务端应识别重复批改操作（基于提交ID和批改指纹），返回已存在的批改结果，保证幂等。
- **8a**：系统保存失败，提示“批改失败，请稍后重试”，已输入内容不丢失，可重新提交。
- **9a**：通知学生失败同样不影响批改结果，提供补偿机制。

#### 用例4：学生参加直播课程

**用例编号**：UC-004  
**参与者**：学生  
**前置条件**：学生已登录，且已报名该课程；直播已开始或即将开始  
**后置条件**：学生进入直播间，可观看并参与互动

**基本流**：
1. 学生进入课程详情页，点击“直播”标签。
2. 系统显示直播列表（含直播主题、时间、状态：未开始/直播中/已结束）。
3. 学生点击“进入直播间”按钮（仅当直播状态为“直播中”或“未开始”可点击）。
4. 系统校验：
   - 学生是否已报名该课程。
   - 直播是否在有效时间窗口内（开始时间前5分钟允许进入等待，结束后5分钟仍可观看）。
   - 当前观看人数是否已达上限（设定5000），若已达上限，采用排队机制或拒绝。
5. 系统生成带有防盗链参数的播放URL（2小时有效期），加载直播播放器。
6. 学生观看直播，界面包含：
   - 视频画面（教师摄像头/屏幕共享）。
   - 文字聊天区（可发送消息，需防XSS）。
   - 举手按钮（申请发言或提问）。
7. 学生可点击“举手”按钮，教师端收到请求，可选择同意/拒绝；同意后建立 WebRTC 连麦。
8. 直播结束后，系统自动关闭播放器，提示“直播已结束”。
9. 系统异步生成回放，完成后通知已报名学生。

**备选流**：
- **3a**：直播状态为“未开始”，系统显示倒计时，按钮文案变为“等待直播开始”，并定时刷新状态。
- **3b**：直播状态为“已结束”，系统显示“观看回放”按钮（若已录制且转码完成），或“回放生成中，请稍后”。
- **4a**：学生未报名该课程，系统提示“请先报名课程”，并跳转至报名页面。
- **4b**：人数已达上限，系统提示“当前观看人数已满，请稍后再试或观看回放”，并自动订阅排队通知。
- **5a**：防盗链 token 生成失败，提示“进入失败，请刷新重试”。
- **6a**：网络不稳定导致播放卡顿，播放器自动尝试降低清晰度或重连，并显示缓冲提示。
- **7a**：举手后长时间无教师响应（超过2分钟），系统自动取消举手并提示学生。
- **7b**：连麦过程中网络断开，系统重试建立连接，失败后降级为纯观看模式。

### 5.2 安全需求

| 编号 | 需求描述 |
|------|----------|
| NFR-006 | 用户密码需加密存储（bcrypt cost ≥ 10 或 Argon2id），禁止明文传输及存储 |
| NFR-007 | 所有数据传输使用HTTPS加密（TLS 1.2 或以上），禁用不安全加密套件 |
| NFR-008 | 直播流需防盗链，播放地址包含限时有效签名 token（HMAC-SHA256），过期后需重新获取 |
| NFR-009 | 防止SQL注入、XSS、CSRF等常见Web攻击：所有输入验证服务端二次校验；输出编码；实施CSRF Token 或 SameSite Cookie；API 网关限流 |
| NFR-010 | 用户敏感信息（手机号、邮箱）需脱敏显示，日志中禁止记录密码、Token 等敏感数据 |
| NFR-011 | JWT Token 有效期 24 小时，Refresh Token 有效期 7 天且只能使用一次，刷新后旧 Refresh Token 失效 |
| NFR-012 | API 接口实现基于角色的访问控制 (RBAC) 和资源级权限 (数据隔离)，每次请求校验 |
| NFR-013 | 文件上传限制白名单扩展名（pdf, doc, docx, ppt, pptx, jpg, png, mp4...)，服务端校验文件魔数，防止脚本上传 |
| NFR-014 | 所有关键操作（课程删除、作业提交、批改、审核）需记录审计日志，保留至少 180 天 |
| NFR-015 | 敏感接口（如登录、注册、忘记密码）必须实施验证码（图形或短信）保护，防止暴力破解及撞库 |
| NFR-016 | 使用速率限制（API Rate Limiting），同一用户对敏感接口的请求频率限制为：登录 5次/分钟/IP，注册 3次/小时/IP；作业提交 10次/分钟/用户 |
