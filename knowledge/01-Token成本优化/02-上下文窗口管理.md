# 上下文窗口管理

## 概述

Claude Code 的上下文窗口上限为 200K tokens。当上下文接近或超过该限制时，模型会触发上下文压缩机制。合理的上下文窗口管理策略可以避免关键信息丢失，同时控制 token 消耗。本规范涵盖压缩触发条件、压缩命令使用、对话管理策略及信息密度原则。

## 核心规则

### 🔴 必须遵守 (MUST)

1. **三级阈值实时监控（每次回复前自检）**
   | 使用率 | Token 量 | 动作 | 优先级 |
   |--------|----------|------|:--:|
   | ≥ 75% | ~150K | 输出警告 + 下个逻辑断点 `/compact` | P1 |
   | ≥ 100% | ~200K | **立即** `/compact`，不等逻辑断点 | P0 |
   | ≥ 200% | ~400K | 保存恢复点到 `.claude/activeContext.md` + **强制** `/compact` | P0 |

2. **主动压缩，不依赖 autocompact**
   - autocompact 触发时上下文已严重超限，压缩质量最差
   - 在逻辑断点处（每阶段完成、每批次智能体返回后）主动 `/compact`
   - 压缩黄金窗口：50-100 轮，内容少时压缩质量高且全价 token 少

3. **压缩前必写 `.claude/activeContext.md`**
   - 保存项目状态（当前阶段、文件路径、待办）
   - 保存关键恢复信息（错误消息、架构决策、测试状态）
   - 配置 `PreCompact` Hook 自动提醒

4. **上下文 ≥ 200K 后禁止启动新智能体**
   - 子代理继承当前上下文，可能溢出
   - 先 compact 再派遣

### 🟡 强烈建议 (SHOULD)

1. **滑动窗口策略**
   - 保留最近的 N 轮完整对话
   - 较早的对话压缩为摘要
   - 摘要格式应包含：决策记录、代码变更摘要、待办事项

2. **信息密度评估**
   - 每轮对话前权衡：这条信息对后续任务的贡献度
   - 低密度信息（如确认性回复、中间状态输出）优先压缩

3. **关键信息标记**
   - 使用固定标记格式标注关键决策
   - 示例格式：`## DECISION: ...`、`## TODO: ...`

### 🟢 可选建议 (MAY)

1. **自动压缩策略**
   - 配置 token 计数自动触发压缩的阈值
   - 设置压缩前确认钩子

2. **会话存档**
   - 长任务完成后将完整会话存档
   - 建立可搜索的决策日志

## 正确示例

```python
# PreCompact Hook 配置 — .claude/settings.json
"""
{
  "hooks": {
    "PreCompact": {
      "command": "python scripts/preserve_context.py",
      "timeout": 10000,
      "description": "在压缩前抢救关键信息"
    }
  }
}
"""

# preserve_context.py — 抢救关键信息的示例脚本
import json
import re

# 从当前上下文中提取关键信息
def extract_key_information(conversation_text: str) -> dict:
    decisions = re.findall(r'## DECISION:\s*(.+?)(?:\n|$)', conversation_text)
    todos = re.findall(r'## TODO:\s*(.+?)(?:\n|$)', conversation_text)
    files_changed = re.findall(r'File: (.+?)(?:\n|$)', conversation_text)
    
    summary = {
        "decisions": decisions,
        "todos": todos,
        "files_changed": files_changed,
        "timestamp": "2026-05-30T12:00:00Z",
        "conversation_turn": estimate_turn_count(conversation_text)
    }
    
    # 写入受保护的关键信息文件
    with open(".claude/key_context.json", "w") as f:
        json.dump(summary, f, indent=2, ensure_ascii=False)
    
    print(f"[PreCompact] 已保存 {len(decisions)} 个决策, {len(todos)} 个待办")
    return summary

def estimate_turn_count(text: str) -> int:
    return len(re.findall(r'\*\*Assistant\*\*', text))


# 对话管理策略示例
class ConversationManager:
    """
    每 50 轮自动触发压缩，并保留关键信息摘要
    """
    def __init__(self, max_turns_before_compact: int = 50):
        self.turn_count = 0
        self.max_turns = max_turns_before_compact
        self.key_decisions: list[str] = []
        self.compact_summaries: list[str] = []
    
    def record_turn(self, decision: str | None = None):
        self.turn_count += 1
        if decision:
            self.key_decisions.append(f"## DECISION: {decision}")
        
        if self.turn_count >= self.max_turns:
            self.trigger_compact()
    
    def trigger_compact(self):
        summary = self._generate_summary()
        self.compact_summaries.append(summary)
        print(f"[Session] 触发压缩，当前摘要长度: {len(summary)} 字符")
        print(f"[Session] 压缩命令: /compact")
        self.turn_count = 0
    
    def _generate_summary(self) -> str:
        return (
            "--- 会话摘要 ---\n"
            + "\n".join(self.key_decisions[-20:]) + "\n"
            + f"--- 共 {len(self.key_decisions)} 个决策记录 ---"
        )
```

## 错误示例

```python
# 错误 1：从不压缩上下文
class BadConversationManager:
    """从不压缩，直到上下文超限报错"""
    def __init__(self):
        self.history = []
    
    def add_turn(self, turn: str):
        self.history.append(turn)
        # 没有压缩逻辑，上下文无限增长
        # 当 total_tokens > 200K 时，API 会报错
        # 且模型注意力质量会严重下降


# 错误 2：压缩时不保留关键信息
def bad_compact():
    """
    直接 /compact 没有前置抢救步骤
    → 压缩后丢失了关键的架构决策和 API 设计
    → 后续对话需要重复讨论已决定的内容
    """

# 错误 3：压缩频率过低
"""
情景：一个任务进行了 200 轮对话，只在结束时压缩一次
问题：
- 中间 150 轮时上下文已超限，模型开始"遗忘"早期内容
- attention 质量下降，回答出现矛盾
- 浪费大量 token 在低价值内容上
"""

# 错误 4：将工具返回的完整文件内容留在对话中
def bad_tool_output_handling():
    """
    每次搜索返回完整文件内容（300行）
    10次搜索 → 3000行代码留在上下文中
    → 上下文快速膨胀
    → 应当只保留关键片段和摘要
    """
    pass  # 工具输出应摘要化
```

## 工具链配置

```python
# Token 计数工具 — 用于在压缩前评估上下文大小
import tiktoken

def count_tokens(text: str, model: str = "claude-sonnet-4-20250514") -> int:
    """估算文本的 token 数量"""
    try:
        enc = tiktoken.encoding_for_model(model)
    except KeyError:
        enc = tiktoken.get_encoding("cl100k_base")
    return len(enc.encode(text))

def should_compact(conversation: str, threshold: int = 180_000) -> bool:
    """判断是否需要触发压缩"""
    token_count = count_tokens(conversation)
    print(f"[TokenCounter] 当前: {token_count} / {threshold} tokens")
    return token_count > threshold
```

## 进阶专题

### Anthropic 官方：5 种会话管理策略

来自官方博客 *Using Claude Code: session management and 1M context* (2026.04)：

| 策略 | 命令 | 何时用 | Token 代价 |
|------|------|------|:--:|
| **Continue** | 直接继续 | 同一任务，上下文仍相关 | 累积增长 |
| **Rewind** | `/rewind` / 双击 Esc | Claude 走错路了 | 丢弃失败尝试 |
| **Compact** | `/compact <引导>` | 会话被过时调试撑满 | 有损压缩 |
| **Clear** | `/clear` | 开始全新任务 | 完全重置 |
| **Subagents** | 子代理 | 只要结论不要过程 | 独立上下文 |

### 上下文腐烂（Context Rot）

Anthropic 官方承认：**上下文越长，模型越蠢**。1M 上下文下，约 **300K-400K tokens** 开始明显退化。注意力被稀释到更多 token 上，旧内容干扰当前任务。

### 压缩的陷阱

> "自动压缩触发时，恰恰是上下文最长、模型表现最差的时候——**用最差的状态去做最关键的总结**。"

**对策**：
1. **主动压缩**：在逻辑断点处手动 `/compact`，不要等 autocompact
2. **引导压缩**：附带保留指令 —— 见下方 CLAUDE.md 配置
3. **早压缩**：50 轮前主动触发，内容少时压缩质量高且全价 token 少

### CLAUDE.md 压缩引导指令

```markdown
# Compact Instructions
When compacting, always preserve:
- All modified file paths with line numbers
- Current test results (pass/fail with file names)
- The active task plan and remaining TODO items
- Error messages and stack traces from the current debug session
- Key architectural decisions made in this session
```

### Rewind 是最重要的习惯

与其说"这个不行，换 X 试试"→ **rewind 到失败前**，带着新认知重新下指令。保留有用的文件读取和调试成果，丢弃失败的尝试路径。

### Subagents 的上下文隔离价值

每个子代理拥有**独立 200K 上下文窗口**。判断标准：**"我需要过程还是只需要结论？"** 三个并行子代理 ≈ 600K 有效工作内存。

### 关键命令速查

| 命令 | 功能 | 时机 |
|------|------|------|
| `/context` | 显示 token 使用明细 | 经常检查 |
| `/cost` | 显示累计 token 和费用 | 监控支出 |
| `/compact` | 手动压缩 + 保留指令 | 逻辑断点处 |
| `/clear` | 完全清除上下文 | 任务完全改变时 |
| `/rewind` (Esc+Esc) | 回退到之前消息 | 纠正错误路径 |

### 窗口分配明细

```
200K 总窗口
├── 系统提示 + 内置工具     ~20K   (固定，ToolSearch 后 ~5K)
├── MCP 工具定义            ~2-51K (取决于活跃 MCP 数)
├── CLAUDE.md               ~0.3-2K
├── 自动压缩缓冲区           ~33K
└── 实际可用于对话           ~114K  (~57%)
```

**性能退化**在约 150K (75%) 开始，远早于 200K 硬限制。
