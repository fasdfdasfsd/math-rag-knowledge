## 配置与 Hook

### 1. matches 正则过宽误拦截
- **问题**：`matches 'rm'` 会拦截所有包含 rm 的命令（如 `rmdir`、`germ`）
- **解决**：用 `includes 'rm -rf'` 精确匹配子串
- **日期**：2026-05-30

### 2. 自动模式阻止写入 settings.json
- **问题**：Claude Code 自动模式下不允许修改 .claude/settings.json
- **解决**：L3 文件需用户手动覆盖，在部署步骤中明确说明
- **日期**：2026-05-30

### 3. git push 分支保护可被变体绕过
- **问题**：`git push origin main` 精确匹配不够，`git push -u origin main` 可绕过
- **解决**：用 `includes 'main'` 子串匹配，不限具体 git push 语法
- **日期**：2026-05-30

### 4. API_TIMEOUT 默认值不足
- **问题**：Claude Code 默认 5 分钟超时，RAG 复杂任务（批量 embedding + 检索 + 生成）不够
- **解决**：`API_TIMEOUT_MS` 设为 600000（10 分钟）
- **日期**：2026-05-30

### 5. SessionStart hook 与 CLAUDE.md 不一致
- **问题**：hook 提醒读 knowledge/README.md，CLAUDE.md 要求读 MEMORY.md
- **解决**：修改方案时必须同步检查 hook 配置，保持两者对齐
- **日期**：2026-05-30

### 6. Stop hook 无法强制写记忆
- **问题**：Hook 只能运行 shell 命令，无法驱动 Claude 调用 Write 工具
- **解决**：记忆持久化核心靠 CLAUDE.md 硬性指令 + 即时写入触发条件，不依赖 hook
- **日期**：2026-05-30

### 7. L1 CLAUDE.md 知识库编号错误
- **问题**：实际目录名为 05~10，L1 CLAUDE.md 写成了 01~06，全部 @ 引用路径失效
- **解决**：CLAUDE.md 中的引用路径必须与 knowledge/ 实际目录名严格一致
- **日期**：2026-05-30

### 8. config/ 缺少全局 memory 模板
- **问题**：设计时遗漏了 `~/.claude/projects/.../memory/MEMORY.md` 的部署模板
- **解决**：补充 `config/L2/global-memory.md`，模板须覆盖所有部署目标
- **日期**：2026-05-30

---

### 9. matcher + if 条件组合导致所有 Bash 被误拦截
- **问题**：`matcher: "Bash"` + `if: "tool_input.command includes '...'"` 组合导致所有 Bash 命令被拦截
- **原因**：`matcher` 纯字母时匹配工具名（所有 Bash 命中），`if` 条件被忽略
- **解决**：`matcher` 含空格/特殊字符时自动变为正则匹配命令内容。用 `matcher: "git push.*main"` 直接正则匹配，去掉 `if`
- **日期**：2026-05-30

### 10. python/curl 权限过宽
- **问题**：`Bash(python:*)` / `Bash(curl:*)` 允许了所有 python/curl 子命令，安全风险大
- **解决**：细化到具体子命令，如 `python -m pytest:*`、`curl -*:*`
- **日期**：2026-05-30

### 11. PostToolUse matcher+if 不可靠（同 #9 家族）
- **问题**：`PostToolUse` 用 `matcher: "Edit|Write"` + `if: "endsWith '.py'"` — `|` 属于工具名字符集，`if` 可能被忽略
- **原因**：避坑 #9 的同一类问题，只是从 PreToolUse 换到了 PostToolUse
- **解决**：拆分为两个独立 Hook 条目 `matcher: "Edit"` + `matcher: "Write"`，去掉 `if`
- **日期**：2026-05-30

### 12. L3 Bash(*) 覆盖 L2 全部安全模型
- **问题**：L3 settings.local.json 中 `Bash(*)` 完全绕过 L2 的 14 条精细 Bash 权限和 5 条 Deny 规则
- **原因**：部署时为避免权限弹窗，临时加入了 `Bash(*)`，但忘记事后移除
- **解决**：移除 `Bash(*)` 及所有部署临时 Bash 权限，L3 仅保留 WebSearch + WebFetch 域限制
- **日期**：2026-05-30

## MCP 安装与部署

> 来源：v4.2 部署会话 (2026-05-31)

### 13. `claude mcp add` 命令语法 — npx 参数需 `--` 分隔

- **问题**：`claude mcp add e2b npx -y @e2b/mcp-server` 报错 `error: unknown option '-y'`。`-y` 被 claude 解析为自己的参数而非传给 npx
- **原因**：`claude mcp add` 的参数解析器会把 `--` 之前的所有选项视为自己的选项
- **解决**：`claude mcp add <name> -- npx -y <package>` — `--` 之后的所有内容原样传给子进程
- **日期**：2026-05-31

### 14. `@anthropic/mcp-server-*` 包名系列均不存在

- **问题**：方案文档中多处使用 `@anthropic/mcp-server-git`、`@anthropic/mcp-server-filesystem`、`@anthropy/mcp-server-docker`、`@anthropy/mcp-server-redis`，安装时报 404 或连接失败
- **原因**：Anthropic 官方 MCP 包发布在 `@modelcontextprotocol/` 命名空间下（如 `@modelcontextprotocol/server-github`），不是 `@anthropic/`。社区包使用裸名（如 `git-mcp-server`、`mcp-server-docker`）
- **解决**：安装前先用 `npm search` 或 `npx -y <pkg>` 验证包名是否存在
- **日期**：2026-05-31

### 15. 文档包名与 npm 实际注册包名不一致

- **问题**：`@iflow-mcp/one-search-mcp` 安装失败，实际可用的是 `one-search-mcp`
- **原因**：npm 上同一个包可能有多个命名空间别名，但并非所有别名都可正确下载
- **解决**：从 `npmjs.com` 确认实际包名，而非信任 MCP 目录站点的元数据
- **日期**：2026-05-31

### 16. MCP 服务器 stdio 模式连接失败排查三板斧

- **问题**：多个 MCP 安装后 `claude mcp list` 显示 `✗ Failed to connect`
- **排查步骤**：
  1. `npx -y <package>` 直接运行，看日志输出
  2. 检查是否需传参数（如 Redis MCP 需 `redis://host:port`）
  3. 检查 Node.js 版本兼容性（如 Grounded Docs 需 v22+，Windows 间有兼容问题）
  4. 尝试 HTTP 模式替代 stdio（`-t http -- npx <pkg> --protocol http --port <port>`）
- **日期**：2026-05-31

### 17. `claude mcp remove` + 重装是快速修复连接失败的首选

- **问题**：MCP 连接失败后反复重试无效
- **解决**：`claude mcp remove <name>` → `claude mcp add <name> -- npx -y <corrected-package>` — 简单快捷
- **日期**：2026-05-31

### 18. `claude mcp add` 环境变量参数与文档不一致

- **问题**：`claude mcp add ... --env KEY=value` 参数报错
- **解决**：实际参数是 `-e` 而非 `--env`：`claude mcp add <name> -e KEY=value -- npx -y <pkg>`
- **日期**：2026-05-31

### 19. MCP 工具连接需要对应的后端服务运行

- **问题**：Redis MCP / PostgreSQL MCP 安装后 `Failed to connect`
- **原因**：这些 MCP 只是客户端代理，需要对应的后端服务（Redis Server / PostgreSQL）实际运行才能连接
- **解决**：先启动后端服务，再安装 MCP
- **日期**：2026-05-31

> 来源：v4.2 部署会话 (2026-05-31)

### 20. E2B Cloud / Context7 / Brave Search 三大海外 API 国内不可用

- **问题**：E2B API 海外不可达、Context7 CDN 超时、Brave Search API 被墙
- **解决**：
  - E2B → Docker 本地沙箱（`docker run --rm python:3.12-slim`）
  - Context7 → WebFetch + Playwright（v4.1 已有权限，直接拉官方文档站）
  - Brave Search → OneSearch MCP（聚合 SearXNG/Bing/DuckDuckGo，免费无 API Key）
- **日期**：2026-05-31

### 21. ghcr.io（GitHub Container Registry）国内无法拉取

- **问题**：`docker pull ghcr.io/bigppwong/edgebox:latest` 报 `error from registry: denied`
- **原因**：ghcr.io 在境内被 DNS 干扰或 IP 阻断，Docker 镜像加速器仅对 Docker Hub 生效，不对 ghcr.io 生效
- **解决**：放弃依赖 ghcr.io 的镜像，直接用 Docker Hub 上的 `python:3.12-alpine` + 手动安装工具链
- **日期**：2026-05-31

### 22. Docker Hub 镜像加速器仅对 docker.io 生效

- **问题**：`daemon.json` 中配置了 6 个 `registry-mirrors`，但拉取 ghcr.io 镜像时仍然失败
- **原因**：`registry-mirrors` 配置仅对 `docker.io`（Docker Hub）的镜像生效。ghcr.io / gcr.io / quay.io 等其他 registry 需要前缀替换方式（如 `docker.m.daocloud.io/ghcr.io/xxx`）
- **解决**：国内可用镜像加速站：`docker.xuanyuan.me`、`docker.1ms.run`、`docker.m.daocloud.io`
- **日期**：2026-05-31

## Docker 与沙箱

> 来源：v4.2 审计会话 (2026-05-31)

### 25. Git Bash (MinGW) 自动转换 Docker 路径

- **问题**：`docker run -v "D:\project:/test" -w /test ...` 报 `the working directory 'C:/Program Files/Git/test' is invalid`
- **原因**：Git Bash 的 MinGW 层自动将 POSIX 路径（如 `/test`）转换为 Windows 路径（`C:/Program Files/Git/test`）
- **解决**：在 docker 命令前加 `MSYS_NO_PATHCONV=1` 环境变量，阻止 MinGW 路径转换
- **日期**：2026-05-31

### 26. Docker 容器内无法访问宿主机 `~/.claude.json`

- **问题**：审计脚本在容器内运行时无法验证 MCP 配置
- **原因**：`~/.claude.json`（MCP 配置文件）在宿主机 `C:\Users\Administrator\`，容器内 `/root/` 没有此文件
- **解决**：显式挂载：`-v "C:\Users\Administrator\.claude.json:/root/.claude.json:ro"`
- **日期**：2026-05-31

### 27. Docker `:ro` 挂载导致审计报告写入失败

- **问题**：`OSError: [Errno 30] Read-only file system` 写入报告文件
- **原因**：为安全使用 `:ro`（只读）挂载项目目录，审计脚本尝试在同一目录写入 JSON 报告
- **解决**：报告写入 `/tmp/`（容器内临时目录）或接受写入失败（报告内容已在 stdout 输出）
- **日期**：2026-05-31

### 28. Docker-in-Docker (DinD) 在 Windows 上不可行

- **问题**：容器内运行 `docker` 命令报 `No such file or directory`
- **原因**：Windows Docker Desktop 使用命名管道（`npipe://...dockerDesktopLinuxEngine`），无法通过 volume 挂载传入容器。DinD 在 Linux 宿主机上可通过 `-v /var/run/docker.sock:...` 实现，Windows 不支持
- **解决**：容器内跳过 Docker 相关测试，在宿主机上单独验证。审计脚本通过 `/.dockerenv` 文件检测容器环境并自动跳过
- **日期**：2026-05-31

### 29. Docker 国内镜像站全部不可用时的应急方案

- **问题**：所有国内 Docker 镜像加速站（DaoCloud/USTC/轩辕/1ms）均不可达，`docker pull` 全部超时
- **原因**：2026-05~06 国内 Docker 镜像生态大面积失效
- **解决**：利用已有缓存镜像 + 版本标记绕过。如 `docker tag postgres:15 postgres:16-alpine`
- **日期**：2026-06-01

## Python 生态

> 来源：数学冒险世界 v4.3 开发会话 (2026-06-01)

### 30. passlib 与 bcrypt ≥ 4.1 不兼容

- **问题**：`from passlib.context import CryptContext` 报 `ModuleNotFoundError: bcrypt.__about__`
- **原因**：passlib 依赖 `bcrypt.__about__` 属性，bcrypt 4.1+ 已移除该属性
- **解决**：放弃 passlib，直接用原生 `bcrypt.hashpw()` / `bcrypt.checkpw()`，自带 salt 管理
- **日期**：2026-06-01

### 31. `async def` 依赖注入函数导致调用失败

- **问题**：`require_role = lambda: require_role("admin")` 调用后 `require_admin` 返回的是 coroutine 而非 inner function
- **原因**：`async def require_role` 声明为 async，调用时不加 await 返回 coroutine 对象，FastAPI Depends 无法识别
- **解决**：依赖注入函数改为 `def`（非 async），`require_admin = require_role("admin")` 去掉 lambda
- **日期**：2026-06-01

### 32. OpenAI SDK 自动追加 `/v1` 导致 404

- **问题**：`DEEPSEEK_BASE_URL=https://api.deepseek.com/v1` 配置下，OpenAI SDK 实际请求 `https://api.deepseek.com/v1/v1/chat/completions` → 404
- **原因**：OpenAI Python SDK 自动在 base_url 后追加 `/v1`，若配置中已含 `/v1` 则重复
- **解决**：`DEEPSEEK_BASE_URL` 设为 `https://api.deepseek.com`（不带尾部路径），SDK 会自动补全
- **日期**：2026-06-01

### 33. Pydantic Settings `extra="ignore"` 配置不生效

- **问题**：`.env` 中自定义变量（如 `MILVUS_COLLECTION`）未被加载
- **原因**：Pydantic Settings 默认 `extra="forbid"`，未在 model 中定义的字段会被忽略而非自动注入
- **解决**：所有需要从环境变量读取的字段必须在 Settings 类中显式定义；或设置 `model_config = {"extra": "allow"}`
- **日期**：2026-06-01

### 34. 对象 ID 生成后未写回返回字典

- **问题**：`create_level()` / `create_npc()` / `create()` 生成的 ID 在返回的 dict 中缺失
- **原因**：`data["id"] = lid` 在 `return data` 之前遗漏
- **解决**：Create 方法必须在 return 前将生成的 ID 写入 data
- **日期**：2026-06-01

## Token 成本

> 来源：数学冒险世界 v4.3 开发会话 (2026-06-01)

### 35. 上下文 283% 溢出 — 连续多智能体派遣不压缩

- **问题**：上下文达到 567K/200K（283%），16 次智能体派遣的完整输出全部留在主会话中
- **原因**：每轮派遣 ≥3 个智能体，每个返回大量结构化分析，累积后迅速超限。主代理未在派遣间隙主动压缩
- **解决**：建立三级阈值自动压缩机制（≥75% 警告 / ≥100% 强制 / ≥200% 紧急），写入 CLAUDE.md + `.claude/rules/behavior.md`
- **日期**：2026-06-01

### 36. autocompact 不可靠 — 必须主动管理

- **问题**：依赖 autocompact 自动触发，实际触发时上下文已严重超限，压缩质量极差
- **原因**：autocompact 在"最差状态"做"最关键的总结"（Anthropic 官方承认的陷阱）
- **解决**：在逻辑断点（每阶段完成、每批次智能体返回后）主动 `/compact`，不等自动触发
- **日期**：2026-06-01

### 37. 子代理模型默认应与主会话一致

- **问题**：所有子代理默认用 flash 模型，复杂架构审查任务质量下降
- **原因**：过度优化 token 成本导致关键分析质量不足
- **解决**：根据任务复杂度分级——简单任务 flash，复杂推理（架构/安全/审计）用 pro。在派遣时明确指定
- **日期**：2026-06-01

## LLM 调用

> 来源：数学冒险世界 v4.3 开发会话 (2026-06-01)

### 38. DeepSeek API 返回 bilingual 内容需后处理

- **问题**：DeepSeek E2E 测试返回的 6 段冒险叙事混有中英文，部分 formula 使用英文表达
- **原因**：System Prompt 语言约束不够强，LLM 在数学公式上下文容易切换为英文
- **解决**：System Prompt 中强化语言约束 + Post-LLM auditor 检测非目标语言段落
- **日期**：2026-06-01

### 39. EventSource API 不支持自定义 HTTP Header

- **问题**：前端 SSE 认证失败，EventSource 构造函数不接受 headers 参数
- **原因**：EventSource W3C 规范不支持自定义 Header（Authorization/Bearer token 无法传入）
- **解决**：改用 `fetch()` + `ReadableStream.getReader()` 手动解析 SSE 流，可以正常携带 Authorization header
- **日期**：2026-06-01

### 40. LLM API timeout 默认值不适用流式生成场景

- **问题**：OpenAI SDK 默认 timeout 60s，6 段叙事全部生成可能超时
- **原因**：流式生成需要更长时间窗口（P95 60s 目标），默认 timeout 在边界情况下触发误报
- **解决**：流式请求设 `timeout=120`（httpx.Timeout），给 LLM 足够缓冲
- **日期**：2026-06-01

### 41. 一次生成 6 段 vs 6 次独立调用的成本权衡

- **问题**：直觉上分 6 次调用更可控，但成本和延迟 ×6
- **分析**：单次 6 段生成保持叙事连贯性，每段校验失败时只需替换问题段落而非整体重来。6 次独立调用会丢失段落间上下文
- **决策**：一次生成完整 6 段 → 逐段校验 → 逐段推送（ADR-012）
- **日期**：2026-06-01

## 软件工程流程

> 来源：数学冒险世界 v4.3 开发会话 (2026-06-01)

### 42. 用户重复纠正同一行为 → 说明配置级固化不足

- **问题**：用户在同一会话中 ≥5 次纠正"不要问是否继续"、≥3 次纠正"上下文检查"、≥4 次纠正"给选择题而非问答题"
- **原因**：CLAUDE.md 是长文档，关键行为规则淹没在细节中，模型注意力被稀释
- **解决**：建立"行为宪法"独立文件（`.claude/rules/behavior.md`），4 条红线，短小精悍，SessionStart hook 启动时强制打印摘要。同时写入 `~/.claude/CLAUDE.md` 全局顶部
- **日期**：2026-06-02

### 43. 阶段完成后未及时写 session log

### 44. Anti-BOLA 验证抛出内置异常无法被中间件捕获

- **问题**：`verify_resource_ownership` 抛出 `PermissionError`（Python 内置异常），AppException 中间件无法统一处理
- **原因**：内置异常类型不在 `AppException` 继承体系中，异常处理器匹配不到
- **解决**：统一抛出自定义 `ForbiddenException`（继承 AppException），中间件正确转换为 RFC 9457 Problem Detail
- **日期**：2026-06-01
