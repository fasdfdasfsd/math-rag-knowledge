# 架构设计规范

## 概述

架构设计决定系统的骨架和质量属性。本规范涵盖 C4 模型分层表达、ADR 决策记录、SOLID 原则实践、微服务与单体架构决策树、架构评审清单及技术债务管理，确保架构设计可沟通、可追溯、可持续演进。

---

## 核心规则

### MUST（必须遵守）

1. **MUST - 使用 C4 模型表达架构**
   - 至少绘制 Context 和 Container 两层图
   - Component 和 Code 层可根据复杂度选择性绘制
   - 使用统一符号标准（如 Structurizr DSL 或 PlantUML C4）

2. **MUST - 所有重大架构决策须记录 ADR**
   - ADR 文件存储在项目仓库 `docs/adr/` 目录下
   - 每个 ADR 文件以序号和标题命名：`0001-use-postgresql.md`

3. **MUST - 核心模块遵循 SOLID 原则**
   - 单⼀职责、开闭原则、里氏替换、接口隔离、依赖反转

4. **MUST - 架构建模与代码实现保持一致**
   - 架构图需随代码演进同步更新，避免架构腐化

### SHOULD（应该遵守）

1. **SHOULD - 微服务拆分前先确认边界**
   - 基于业务领域（DDD Bounded Context）而非技术维度拆分

2. **SHOULD - 定义明确的架构质量属性**
   - 性能、可用性、安全性、可维护性、可扩展性、可测试性

3. **SHOULD - 建立架构守护（ArchGuard）机制**
   - 使用自动化工具检查架构约束（如层间依赖方向）

4. **SHOULD - 每季度做架构健康检查**
   - 识别架构腐化迹象和技术债务

### MAY（可以遵守）

1. **MAY - 使用事件风暴（Event Storming）做领域建模**
2. **MAY - 采用演进式架构（Evolutionary Architecture）**
3. **MAY - 引入架构适应度函数（Architecture Fitness Functions）**

## 流程与检查清单

### C4 模型层级说明

| 层级 | 名称 | 受众 | 内容 |
|------|------|------|------|
| Level 1 | System Context | 所有相关方 | 系统与用户、外部系统的关系 |
| Level 2 | Container | 开发团队 | 应用、服务、数据库、消息队列等 |
| Level 3 | Component | 开发团队 | 容器内部的模块划分与依赖 |
| Level 4 | Code | 开发者 | 类图、核心接口、关键模式 |


### 微服务 vs 单体决策树

```
系统需要拆分？
├─ 团队 < 10 人 AND 业务复杂度低 → 单体架构（Modular Monolith）
├─ 团队 > 10 人 OR 独立部署需求高 → 考虑微服务
│   ├─ 领域边界清晰？→ 按 Bounded Context 拆分
│   ├─ 领域边界模糊？→ 先做 Modular Monolith，逐步拆分
│   └─ 数据一致性要求高？→ 考虑 Saga 模式
└─ 总原则：不要为了微服务而微服务
```

### SOLID 原则速查

| 原则 | 含义 | 反例 |
|------|------|------|
| S - 单一职责 | 一个类只有一个变更理由 | God Class 做所有事 |
| O - 开闭原则 | 对扩展开放，对修改关闭 | 修改既有代码增加功能 |
| L - 里氏替换 | 子类可替换父类使用 | 子类抛出异常破坏父类合约 |
| I - 接口隔离 | 客户端不应依赖不需要的接口 | Fat Interface 包含无用方法 |
| D - 依赖反转 | 依赖抽象而非具体实现 | 高层模块直接依赖低层实现 |

### 架构评审 Checklist

| 维度 | 检查项 |
|------|--------|
| 模块化 | 模块间耦合度是否可控？依赖方向是否合理？ |
| 扩展性 | 新增功能是否需要修改大量现有代码？ |
| 数据流 | 数据流向是否清晰？是否存在循环依赖？ |
| 安全性 | 是否有安全边界？敏感数据是否隔离？ |
| 可测试性 | 核心逻辑能否脱离基础设施测试？ |
| 性能 | 是否存在明显的瓶颈点？缓存策略是否合理？ |
| 可用性 | 是否有单点故障？容错和降级策略？ |
| 一致性 | 架构图与代码实现是否一致？ADR 是否覆盖决策？ |

### 技术债务管理

| 债务类型 | 示例 | 处理策略 |
|----------|------|----------|
| 有意的主动债务 | 为赶上线暂不做优化 | 记录在 Tech Debt Backlog，设定还款期限 |
| 有意的被动债务 | 明知有坑但无力修复 | 加 TODO 注释，分配 20% 时间偿债 |
| 意外的债务 | 设计失误导致的耦合 | 架构评审尽早发现，及时重构 |
| 过时的债务 | 使用了废弃的库 | 纳入升级计划，设定 EOL |

**债务管理流程**：识别 → 记录（含利息估算）→ 分类 → 纳入迭代规划 → 偿债 → 验证。
