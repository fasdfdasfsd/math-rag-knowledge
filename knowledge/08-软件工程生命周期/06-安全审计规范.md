# 安全审计规范

## 概述

安全审计是软件工程生命周期中不可缺失的一环，贯穿设计、开发、测试、部署和运维全过程。本规范基于 STRIDE 威胁建模方法，涵盖 SAST 和 DAST 工具使用、渗透测试流程、安全审计检查清单及 CVSS 漏洞管理流程，建立纵深防御体系。

---

## 核心规则

### MUST（必须遵守）

1. **MUST - 每个架构设计阶段须做威胁建模**
   - 使用 STRIDE 方法对每个系统组件进行威胁分析
   - 威胁建模结果记录在 ADR 或安全文档中

2. **MUST - CI/CD Pipeline 集成 SAST 扫描**
   - 每次提交代码时自动执行静态安全分析
   - SAST 扫描发现的 Critical/High 级别问题阻止合并

3. **MUST - 上线前执行 DAST 扫描**
   - 对 staging 环境运行动态安全测试
   - DAST 应覆盖所有公开 Endpoint

4. **MUST - 所有漏洞按 CVSS 评分分级管理**
   - Critical（9.0-10.0）：24 小时内修复
   - High（7.0-8.9）：72 小时内修复
   - Medium（4.0-6.9）：7 天内修复
   - Low（0.1-3.9）：下一次迭代修复

### SHOULD（应该遵守）

1. **SHOULD - 每年至少一次渗透测试**
   - 由独立安全团队或第三方执行

2. **SHOULD - 使用依赖扫描工具检测第三方库漏洞**
   - npm audit / pip-audit / OWASP Dependency-Check / Snyk

3. **SHOULD - 建立安全响应流程**
   - 明确漏洞上报、分类、处置、公开的 SOP

4. **SHOULD - 敏感数据使用密钥管理服务（KMS）**
   - 禁止在配置文件中硬编码密钥、Token 或密码

### MAY（可以遵守）

1. **MAY - 建立漏洞赏金计划（Bug Bounty）**
2. **MAY - 使用 IAST（交互式应用安全测试）增强检测**
3. **MAY - 参与安全合规认证（SOC2 / ISO 27001）**

## 流程与检查清单

### STRIDE 威胁建模

| 威胁类型 | 含义 | 对应安全属性 | 常见攻击 |
|----------|------|-------------|----------|
| Spoofing（欺骗） | 冒充他人身份 | 认证 | Token 伪造、Session Hijacking |
| Tampering（篡改） | 修改数据或代码 | 完整性 | 数据篡改、请求参数注入 |
| Repudiation（抵赖） | 否认操作行为 | 不可否认性 | 缺少审计日志 |
| Information Disclosure（信息泄露） | 泄露敏感信息 | 机密性 | XSS、路径遍历、SQL 注入 |
| Denial of Service（拒绝服务） | 使服务不可用 | 可用性 | DDoS、资源耗尽攻击 |
| Elevation of Privilege（权限提升） | 越权操作 | 授权 | 越权访问、SQL 注入提权 |

**威胁建模步骤**：
1. 绘制数据流图（DFD）
2. 识别每个数据流和存储的 STRIDE 威胁
3. 评估威胁风险等级
4. 制定缓解措施
5. 跟踪缓解措施的落实

### 安全审计 Checklist

```markdown
## 安全审计检查清单

### 认证与授权
- [ ] 所有 API 端点是否需要身份验证？
- [ ] 密码策略是否符合最低要求（长度、复杂度、散列算法）？
- [ ] 是否有多因素认证（MFA）选项？
- [ ] Token 过期和刷新机制是否正确实现？
- [ ] RBAC 权限模型是否正确执行最小权限原则？

### 数据安全
- [ ] 敏感数据在传输和存储中是否加密（TLS / AES）？
- [ ] 数据库连接字符串和密钥是否使用环境变量或密钥管理服务？
- [ ] 日志中是否脱敏敏感信息（身份证、手机号、密码）？
- [ ] 是否对用户输入进行校验和输出编码（防止 XSS）？

### 通信安全
- [ ] 是否强制 HTTPS（HSTS）？
- [ ] API 是否实现速率限制（Rate Limiting）？
- [ ] CORS 配置是否严格（只允许必要 Origin）？

### 基础设施
- [ ] 服务器和容器是否最小化安装（减少攻击面）？
- [ ] 是否定期更新和打安全补丁？
- [ ] 网络是否有分段和访问控制（防火墙 / Security Group）？

### 日志与监控
- [ ] 是否记录完整的安全事件日志？
- [ ] 是否有异常登录告警？
- [ ] 日志存储是否防篡改？
```

### 漏洞管理流程

```
发现漏洞（内部/外部/自动扫描）
    ↓
确认漏洞（安全团队验证）
    ↓
CVSS 评分分级
    ↓
指派修复责任人
    ↓
开发修复 → Code Review（安全 Reviewer） → QA 测试
    ↓
部署修复到生产环境
    ↓
修复验证（重新扫描确认）
    ↓
事后复盘（Root Cause Analysis）
```

### CVSS v3.1 评分速查

| 评分范围 | 等级 | 修复时限 | 示例 |
|----------|------|----------|------|
| 9.0-10.0 | Critical | 24h | 远程代码执行（RCE） |
| 7.0-8.9 | High | 72h | SQL 注入、越权访问 |
| 4.0-6.9 | Medium | 7d | XSS、信息泄露 |
| 0.1-3.9 | Low | Next Sprint | 无实际利用场景的信息泄露 |
