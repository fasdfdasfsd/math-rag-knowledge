# 向量数据库规范

## 概述

本文档定义项目中向量数据库的选型、设计和使用规范，涵盖主流向量数据库对比、pgvector 作为首选的配置与使用、索引类型选择（IVFFlat vs HNSW）、向量维度选择与精度权衡以及批量操作优化策略。适用于 RAG（检索增强生成）、语义搜索、推荐系统等需要向量相似性搜索的场景。

## 核心规则

### MUST（强制遵守）

#### 向量数据库选型对比

| 特性 | pgvector | Pinecone | Milvus | Weaviate | Qdrant |
|------|----------|----------|--------|----------|--------|
| 部署方式 | 自托管（PostgreSQL 扩展） | 托管 SaaS | 自托管 / Zilliz Cloud | 自托管 / 云 | 自托管 / 云 |
| 与主数据一致性 | 原生一致（同一 PostgreSQL） | 需额外同步 | 需额外同步 | 需额外同步 | 需额外同步 |
| 索引类型 | IVFFlat, HNSW | 专有 | IVF, HNSW, DiskANN | HNSW | HNSW, Epsilon |
| 过滤支持 | WHERE + 索引 | 元数据过滤 | 标量过滤 | 元数据过滤 | 过滤 + 超集 |
| 事务支持 | 完全 ACID | 最终一致 | 无 | 无 | 无 |
| 适合场景 | 中小规模 RAG (< 1000 万向量) | 托管部署快速启动 | 大规模（10 亿级） | 多模态搜索 | 高精度搜索 |
| 运维成本 | 低（复用 PostgreSQL） | 最低（全托管） | 高（K8s 集群） | 中 | 中 |

**选型建议：**

- **首选 pgvector**：当项目中已使用 PostgreSQL 且向量数据量 < 1000 万条时。优势在于数据一致性（向量与业务数据在同一数据库）、零额外运维成本。
- **Pinecone**：需要全托管、快速交付、不愿投入运维的场景。
- **Milvus / Qdrant**：向量数据量超过 1000 万条、需要专门的向量数据库集群。
- **Weaviate**：需要混合搜索（向量 + 全文搜索）的开箱支持。

本规范以 **pgvector** 为主要参考，其余向量数据库的接入应引用各自的官方驱动。

#### pgvector 使用规范

##### 启用扩展与建表

```sql
-- 启用 pgvector 扩展
CREATE EXTENSION IF NOT EXISTS vector;

-- 文档嵌入表
CREATE TABLE document_embeddings (
    id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    document_id       UUID NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
    chunk_index       INTEGER NOT NULL,
    chunk_text        TEXT NOT NULL,
    chunk_text_hash   VARCHAR(64) NOT NULL,          -- SHA-256，用于去重检测
    embedding         VECTOR(1536) NOT NULL,          -- OpenAI text-embedding-3-small
    token_count       INTEGER NOT NULL DEFAULT 0,
    metadata          JSONB NOT NULL DEFAULT '{}',
    created_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (document_id, chunk_index)
);

-- 查询嵌入表（用于检索）
CREATE TABLE query_embeddings (
    id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    query_text        TEXT NOT NULL,
    embedding         VECTOR(1536) NOT NULL,
    query_type        VARCHAR(50) NOT NULL DEFAULT 'user',  -- 'user', 'test'
    response_text     TEXT,
    latency_ms        INTEGER,
    created_at        TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

##### 向量维度选择

| 嵌入模型 | 维度 | 适用场景 | 精度 |
|----------|------|----------|------|
| OpenAI text-embedding-3-small | 512 / 1536 | 通用 RAG（推荐 1536） | 高 |
| OpenAI text-embedding-3-large | 256 / 1024 / 3072 | 高精度语义搜索 | 非常高 |
| BAAI/bge-large-zh-v1.5 | 1024 | 中文场景 | 高 |
| sentence-transformers/all-MiniLM-L6-v2 | 384 | 轻量、快速 | 中 |
| Cohere embed-english-v3.0 | 1024 | 英文文本 | 高 |
| gte-large-en-v1.5 | 1024 | 英文通用 | 高 |

- 优先使用 OpenAI `text-embedding-3-small` (1536 维)，性价比最高。
- 如果对精度要求极高且预算充足，使用 `text-embedding-3-large` (3072 维)。
- 如果向量数量超千万且对精度要求不极端，考虑降维至 512~768 维。
- 维度越高，存储开销和查询延迟越大，HNSW 索引受维度影响小于 IVFFlat。

#### 索引类型选择

##### IVFFlat（Inverted File with Flat）

| 特性 | 说明 |
|------|------|
| 构建速度 | 快（粗聚类 + 存储） |
| 查询速度 | 快（取决于 lists 参数） |
| 精度 | 中（受 probes 参数控制） |
| 内存占用 | 低（索引小） |
| 适用场景 | 精确度要求一般，插入后不频繁更新 |

```sql
-- IVFFlat 索引
-- lists = sqrt(行数) 约估值，例如 100 万行使用 1000 lists
CREATE INDEX idx_doc_embeddings_ivfflat
    ON document_embeddings
    USING ivfflat (embedding vector_cosine_ops)
    WITH (lists = 100);

-- 查询时设置 probes（探索列表数量）
-- probes 越大越精确，越慢；推荐 lists/10 ~ lists
SET ivfflat.probes = 20;
```

##### HNSW（Hierarchical Navigable Small World）

| 特性 | 说明 |
|------|------|
| 构建速度 | 慢（需要更多计算） |
| 查询速度 | 最快 |
| 精度 | 最高 |
| 内存占用 | 高（索引大） |
| 适用场景 | 高精度要求，查询频次远高于插入/更新 |

```sql
-- HNSW 索引（推荐用于生产环境）
-- ef_construction: 构建时动态列表大小（越大质量越高，默认 200）
-- m: 每个节点的连接数（默认 16，范围 2-100）

-- 余弦相似度索引
CREATE INDEX idx_doc_embeddings_hnsw_cosine
    ON document_embeddings
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 200);

-- 内积索引（适用于归一化向量的余弦等价）
CREATE INDEX idx_doc_embeddings_hnsw_ip
    ON document_embeddings
    USING hnsw (embedding vector_ip_ops)
    WITH (m = 16, ef_construction = 200);

-- L2 距离索引（较少使用）
CREATE INDEX idx_doc_embeddings_hnsw_l2
    ON document_embeddings
    USING hnsw (embedding vector_l2_ops)
    WITH (m = 16, ef_construction = 200);

-- 查询时设置 ef_search（搜索时动态列表大小，越大越精确越慢）
SET hnsw.ef_search = 100;
```

**选型建议：**

| 场景 | 推荐索引 | 参数 |
|------|----------|------|
| 原型开发 / 小数据量 (< 10 万) | IVFFlat | lists=10, probes=1 |
| 生产 RAG (< 500 万) | HNSW | m=16, ef_construction=200, ef_search=100 |
| 大规模数据 (> 500 万) | IVFFlat 或 HNSW | HNSW 精度更优但内存占用高 |
| 超大规模 (> 5000 万) | 考虑 Milvus / Qdrant | - |

### SHOULD（建议遵守）

#### 相似度计算函数

```sql
-- pgvector 支持的相似度计算
-- 1. 余弦相似度（推荐，文本嵌入通用）
SELECT * FROM document_embeddings
ORDER BY embedding <=> $query_vector    -- 余弦距离（1 - cosine_similarity）
LIMIT 20;

-- 2. 内积（向量已归一化时等价于余弦）
SELECT * FROM document_embeddings
ORDER BY embedding <#> $query_vector     -- 负内积（pgvector 用 <#> 表示负内积）
LIMIT 20;

-- 3. L2 欧氏距离（图像/音频嵌入更常用）
SELECT * FROM document_embeddings
ORDER BY embedding <-> $query_vector     -- L2 距离
LIMIT 20;
```

- 文本嵌入推荐使用余弦相似度（`<=>`）。
- 向量插入 pgvector 后不需要显式归一化，`<=>` 操作符已处理。

#### 过滤 + 相似度搜索

- 使用 `WHERE` + `ORDER BY` + `LIMIT` 实现带元数据过滤的向量搜索。
- 确保过滤条件字段有合适索引，避免先全表扫描再排序。

```sql
-- 带过滤的向量搜索
SET hnsw.ef_search = 100;

SELECT
    de.id,
    de.chunk_text,
    de.document_id,
    de.chunk_index,
    1 - (de.embedding <=> $query_vector) AS similarity
FROM document_embeddings de
INNER JOIN documents d ON d.id = de.document_id
WHERE d.status = 'published'
  AND d.collection_id = $collection_id
  AND de.token_count >= 50           -- 过滤太短的 chunk
ORDER BY de.embedding <=> $query_vector
LIMIT 10;
```

#### 批量操作优化

```python
import asyncpg
import json
from typing import Any


async def batch_insert_embeddings(
    pool: asyncpg.Pool,
    embeddings: list[dict[str, Any]],
    batch_size: int = 100,
) -> int:
    """Batch insert embeddings using COPY protocol for maximum throughput.

    Args:
        pool: Database connection pool.
        embeddings: List of embedding records.
        batch_size: Number of records per batch.

    Returns:
        Total number of records inserted.
    """
    inserted = 0
    for i in range(0, len(embeddings), batch_size):
        batch = embeddings[i:i + batch_size]

        # Use COPY for high-performance bulk insert
        # pgvector supports vector type in COPY
        records = []
        for emb in batch:
            vector_str = "[" + ",".join(str(v) for v in emb["embedding"]) + "]"
            records.append((
                emb["document_id"],
                emb["chunk_index"],
                emb["chunk_text"],
                emb.get("chunk_text_hash", ""),
                vector_str,
                emb.get("token_count", 0),
                json.dumps(emb.get("metadata", {})),
            ))

        # Use executemany for batch insert
        await pool.executemany(
            """
            INSERT INTO document_embeddings
                (document_id, chunk_index, chunk_text, chunk_text_hash,
                 embedding, token_count, metadata)
            VALUES ($1, $2, $3, $4, $5::vector, $6, $7::jsonb)
            ON CONFLICT (document_id, chunk_index) DO NOTHING
            """,
            records,
        )
        inserted += len(batch)
    return inserted


async def search_similar_chunks(
    pool: asyncpg.Pool,
    query_vector: list[float],
    collection_id: str,
    top_k: int = 10,
    similarity_threshold: float = 0.7,
) -> list[dict]:
    """Search for similar document chunks with metadata filtering."""
    vector_str = "[" + ",".join(str(v) for v in query_vector) + "]"

    rows = await pool.fetch(
        """
        SELECT
            de.id,
            de.chunk_text,
            de.document_id,
            de.chunk_index,
            de.metadata,
            1 - (de.embedding <=> $1::vector) AS similarity
        FROM document_embeddings de
        INNER JOIN documents d ON d.id = de.document_id
        WHERE d.collection_id = $2
          AND d.status = 'published'
          AND 1 - (de.embedding <=> $1::vector) >= $3
        ORDER BY de.embedding <=> $1::vector
        LIMIT $4
        """,
        vector_str,
        collection_id,
        similarity_threshold,
        top_k,
    )

    return [dict(row) for row in rows]
```

### MAY（可选的推荐）

- 使用 `Halfvec`（pgvector 0.7.0+）将全精度向量降为半精度浮点，节省存储空间约 50%，精度损失很小。
- 考虑向量量化：使用 `pgvector` 的 `vector_l2_ops`、`vector_ip_ops` 等操作符类。
- 使用分片分区策略：对超大向量集合按 `collection_id` 分区。
- 使用多向量检索策略（MMR / 重排序）提高 RAG 检索质量。
- 结合 BM25 或全文搜索实现混合检索（hybrid search）提升召回率。

```sql
-- 混合检索示例（向量相似度 + BM25 全文检索）
SELECT
    de.id,
    de.chunk_text,
    de.document_id,
    -- 融合得分（向量得分 + 文本得分）
    (1 - (de.embedding <=> $query_vector)) * 0.7
    + ts_rank(d.search_vector, plainto_tsquery('english', $query_text)) * 0.3 AS combined_score
FROM document_embeddings de
INNER JOIN documents d ON d.id = de.document_id
WHERE d.status = 'published'
  AND d.search_vector @@ plainto_tsquery('english', $query_text)
ORDER BY combined_score DESC
LIMIT 20;
```

## 正确示例

```python
import asyncio
import asyncpg
from typing import Any

from openai import AsyncOpenAI


class VectorSearchService:
    """Vector search service using pgvector and OpenAI embeddings.

    Handles embedding generation, storage, and similarity search.
    """

    def __init__(
        self,
        dsn: str,
        openai_api_key: str,
        embedding_model: str = "text-embedding-3-small",
        embedding_dimensions: int = 1536,
    ):
        self.dsn = dsn
        self.openai = AsyncOpenAI(api_key=openai_api_key)
        self.embedding_model = embedding_model
        self.embedding_dimensions = embedding_dimensions
        self._pool: asyncpg.Pool | None = None

    async def initialize(self) -> None:
        """Initialize connection pool and ensure HNSW index exists."""
        self._pool = await asyncpg.create_pool(
            self.dsn, min_size=5, max_size=20,
        )
        async with self._pool.acquire() as conn:
            # Enable extension
            await conn.execute("CREATE EXTENSION IF NOT EXISTS vector")
            # Ensure HNSW index exists
            await conn.execute("""
                CREATE INDEX IF NOT EXISTS idx_embeddings_hnsw
                ON document_embeddings
                USING hnsw (embedding vector_cosine_ops)
                WITH (m = 16, ef_construction = 200)
            """)

    async def embed_texts(self, texts: list[str]) -> list[list[float]]:
        """Generate embeddings for a list of texts."""
        response = await self.openai.embeddings.create(
            model=self.embedding_model,
            input=texts,
            dimensions=self.embedding_dimensions,
        )
        return [item.embedding for item in response.data]

    async def index_document_chunks(
        self,
        document_id: str,
        chunks: list[dict[str, Any]],
    ) -> int:
        """Generate embeddings and index document chunks."""
        texts = [chunk["text"] for chunk in chunks]
        embeddings = await self.embed_texts(texts)

        records = []
        for chunk, embedding in zip(chunks, embeddings):
            records.append({
                "document_id": document_id,
                "chunk_index": chunk["index"],
                "chunk_text": chunk["text"],
                "chunk_text_hash": chunk.get("text_hash", ""),
                "embedding": embedding,
                "token_count": chunk.get("token_count", 0),
                "metadata": chunk.get("metadata", {}),
            })

        return await batch_insert_embeddings(self._pool, records)

    async def search(
        self,
        query: str,
        collection_id: str,
        top_k: int = 10,
        similarity_threshold: float = 0.7,
    ) -> list[dict]:
        """Search for similar chunks to the query text."""
        query_embedding = (await self.embed_texts([query]))[0]
        return await search_similar_chunks(
            self._pool,
            query_embedding,
            collection_id,
            top_k,
            similarity_threshold,
        )

    async def close(self) -> None:
        if self._pool:
            await self._pool.close()
```

## 错误示例

```python
# 错误 1：不使用索引直接搜索
# 表有 100 万行，没有建索引时查询
SELECT * FROM document_embeddings
ORDER BY embedding <=> $query_vector
LIMIT 10;
# → 全表扫描，每个向量都要计算距离，耗时可能数秒

# 错误 2：索引类型与距离函数不匹配
# 建索引使用余弦（cosine_ops），查询却用 L2
CREATE INDEX ON document_embeddings USING hnsw (embedding vector_cosine_ops);
SELECT * FROM document_embeddings
ORDER BY embedding <-> $query_vector   -- L2 距离
LIMIT 10;
# 索引不会被使用，改为索引列上使用对应操作符

# 错误 3：lists 参数选择严重错误
-- 1000 万行数据使用 lists=10（过少，每个 list 太大）
CREATE INDEX idx_ivfflat ON document_embeddings
    USING ivfflat (embedding vector_cosine_ops) WITH (lists = 10);
-- 正确：lists ≈ sqrt(n) = sqrt(10_000_000) ≈ 3162
-- 但 pgvector 限制 lists ≤ 4000，所以取 3000~4000

# 错误 4：未设置 probes/ef_search
-- 使用 IVFFlat 但未设置 probes，默认 probes=1，精度很差
-- 使用 HNSW 但未设置 ef_search，默认 ef_search=40，可适当提高

# 错误 5：返回过多结果 + 后过滤
-- 先取 10000 条再过滤
SELECT * FROM (
    SELECT * FROM document_embeddings
    ORDER BY embedding <=> $query_vector
    LIMIT 10000
) sub
WHERE sub.token_count > 100;
-- 应该先过滤再搜索

# 错误 6：逐条插入向量
for chunk in chunks:
    embedding = await get_embedding(chunk["text"])
    await conn.execute(
        "INSERT INTO document_embeddings (...) VALUES ($1, $2, ...)",
        chunk["doc_id"], chunk["index"], chunk["text"], embedding,
    )
# 1000 个 chunk → 1000 次网络往返
# 正确：批量插入（batch_insert_embeddings）

# 错误 7：向量维度不匹配
# 建表定义 VECTOR(1536)，插入 3072 维嵌入
INSERT INTO document_embeddings (embedding) VALUES ('[0.1, 0.2, ... 3072个值]');
# → 错误：vector of type vector(1536) but expression is of type vector(3072)

# 错误 8：混合检索得分融合不合理
# 向量得分和 BM25 得分量纲不同，需要归一化
# 随机权重（0.5 + 0.5）可能导致检索质量下降
# 正确做法：进行 min-max 归一化后加权融合

# 错误 9：不处理 embedding API 失败
embedding = (await openai.embeddings.create(...)).data[0].embedding
# 忽略 API 限速、超时、空响应等错误处理

# 错误 10：在非向量列上使用向量索引
SELECT * FROM document_embeddings
ORDER BY embedding::text <=> '[0.1, 0.2, ...]'  -- 类型转换导致索引失效
LIMIT 10;
```

### 嵌入缓存策略

```python
import hashlib
import json
from typing import Any

import redis.asyncio as aioredis


class EmbeddingCache:
    """Cache embedding results to reduce API calls and cost."""

    def __init__(self, redis: aioredis.Redis, ttl: int = 86400):
        self.redis = redis
        self.ttl = ttl

    def _make_key(self, text: str, model: str) -> str:
        text_hash = hashlib.sha256(text.encode()).hexdigest()[:16]
        return f"emb:{model}:{text_hash}"

    async def get(self, text: str, model: str) -> list[float] | None:
        key = self._make_key(text, model)
        cached = await self.redis.get(key)
        return json.loads(cached) if cached else None

    async def set(self, text: str, model: str, embedding: list[float]) -> None:
        key = self._make_key(text, model)
        await self.redis.setex(key, self.ttl, json.dumps(embedding))
```
