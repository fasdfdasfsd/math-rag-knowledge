# 知识审查协议

## 一、审查周期

| 事项 | 周期 | 负责人 | 下次审查 |
|------|------|--------|----------|
| knowledge/ 规范文件审查 | 每季度 | 项目成员 | 2026-08-30 |
| .claude/rules/ 规则审查 | 每季度 | 项目成员 | 2026-08-30 |
| CLAUDE.md 审查 | 每季度 | 项目成员 | 2026-08-30 |
| 避坑汇总归档审查 | 每半年 | 项目成员 | 2026-11-30 |

## 二、审查检查清单

### 2.1 过期检测

对每个 `knowledge/` 和 `.claude/rules/` 文件检查：

- [ ] `updated` 字段：是否超过 12 个月未更新？
- [ ] 内容准确性：规则是否仍然适用？约束是否仍然正确？
- [ ] 技术栈匹配：引用的工具/框架版本是否过时？

**处理**：
- 过时但仍有参考价值 → 加 `status: deprecated` 和 `replaced_by: <新文件路径>`
- 完全无效 → 移到 `knowledge/_archived/` 目录

### 2.2 冲突检测

- [ ] 同一规则是否在多个文件中出现且不一致？
- [ ] CLAUDE.md 与 `.claude/rules/` 是否有矛盾指令？
- [ ] 规范文件中的约束与实际 Hook/CI 配置是否一致？

**处理**：
- 以 `.claude/rules/` 为准（最接近执行点）
- 删除重复条目，保留唯一来源
- 同步更新规则-强制映射表

### 2.3 覆盖度检测

- [ ] 规则-强制映射表中是否有新增的 `❌ advisory only` 项？
- [ ] 最近 3 个月产生的避坑是否都已归入避坑汇总？
- [ ] 是否存在「应该写但没写」的规范缺口？

**处理**：
- 新规则配强制执行手段
- 未归类避坑立即归类
- 规范缺口列入下季度计划

### 2.4 闭环率检测

- [ ] 阶段一→二：SessionStart 是否正常提醒？
- [ ] 阶段二→三：未归类避坑是否在 30 天内处理？
- [ ] 阶段三→四：验证过的避坑是否提炼为规范？
- [ ] 阶段四→五：更新后的规范是否同步了映射表？

## 三、废弃标注格式

```yaml
---
category: <原分类>
type: spec
status: deprecated          # 废弃标记
deprecated_date: YYYY-MM-DD # 废弃日期
replaced_by: <替代文件路径>  # 必填
reason: <废弃原因>          # 必填
---
```
