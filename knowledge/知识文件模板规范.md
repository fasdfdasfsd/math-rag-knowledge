# 知识文件模板规范

## 一、文件分类

每个 `knowledge/` 下的 `.md` 文件属于三类之一：

| type | 含义 | 示例 |
|------|------|------|
| `spec` | 规范：硬性规则，不可违反 | Python 编码规范、安全红线 |
| `convention` | 约定：团队推荐实践 | API 命名约定、分支策略 |
| `pitfall` | 避坑：经验教训记录 | 某配置参数导致的线上事故 |

---

## 二、模板 A：规范文件（spec）

```yaml
---
category: <所属分类>
type: spec
priority: must | recommended | optional
version: 1.0
updated: YYYY-MM-DD
applies_to:               # 可选：适用范围
  - "src/**/*.py"
enforced_by:              # 必填：强制执行方式（至少一项）
  - hook: <Hook名称>
  - linter: <工具名>
  - ci: <CI步骤>
  - advisory              # 无强制手段时标注
---
```

**规则写作要求**：
- 每条规则必须是**可证伪的**陈述句（"所有函数必须有类型注解" ✅；"写好代码" ❌）
- 使用 `（NON-NEGOTIABLE）` 标注不可协商的规则
- ❌ 禁止写法："建议..." "考虑..." — 要么是规则，要么删除

---

## 三、模板 B：约定文件（convention）

```yaml
---
category: <所属分类>
type: convention
priority: recommended | optional
updated: YYYY-MM-DD
rationale: |              # 必填：为什么这样约定（Claude 无法从代码推断）
  <一句话理由>
---
```

**约定写作要求**：
- 必须包含 `rationale` 字段 — 这是 Claude 判断何时破例的关键依据
- 如有例外情况，在正文中明确说明

---

## 四、模板 C：避坑文件（pitfall）

```yaml
---
category: <所属分类>
type: pitfall
date: YYYY-MM-DD
symptoms: <一句话症状>
root_cause: <根因>
resolution: <解决方案>
---
```

**避坑写作要求**：
- `symptoms` 便于 Claude 遇到相同症状时检索匹配
- `root_cause` 帮助 Claude 理解本质原因而非表面现象
- 验证过的避坑应提炼为 spec 规则并加入对应规范文件
