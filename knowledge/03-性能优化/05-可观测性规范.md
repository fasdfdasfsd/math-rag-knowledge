# 可观测性规范

## 一、结构化日志

### 格式要求
- 所有日志使用 structlog，JSON 格式输出
- 每行日志必须包含：`timestamp`, `level`, `event`, `logger`
- 禁止 `print()` 和纯文本日志

### 必须包含的字段
- `correlation_id`：跨服务追踪 ID
- `user_id`：操作者（如有）
- `duration_ms`：操作耗时（如有）

### 禁止记录
- 密码、Token、API Key
- 完整的用户 PII（姓名、身份证、手机号需脱敏）
- 信用卡号、银行账号

## 二、指标（Metrics）

### RED 方法
- **Rate**：每秒请求数
- **Error**：错误率（HTTP 5xx + 业务异常）
- **Duration**：P50/P95/P99 延迟

### 必须监控的端点
- 所有 API 端点
- 数据库查询（延迟 + 错误率）
- 外部 API 调用（延迟 + 成功率）
- 后台任务（执行时长 + 失败次数）

## 三、分布式追踪

- 使用 OpenTelemetry 自动插桩
- span 包含 `service.name`, `operation`, `attributes`
- 错误 span 记录 `exception.type`, `exception.message`
- 跨服务传递 W3C TraceContext

## 四、告警

### 告警分级
| 级别 | 条件 | 响应 |
|------|------|------|
| P0 紧急 | 用户不可用 | 5 分钟响应 |
| P1 严重 | 核心功能降级 | 15 分钟响应 |
| P2 警告 | 非核心异常 | 1 小时响应 |

### 告警规则
- 错误率 >1%（5 分钟窗口）
- P95 延迟 >2× baseline（15 分钟窗口）
- 磁盘/内存 >85%
- 禁止在循环或高频路径中触发告警
