# 软件构造与实现规范

## 一、编码规范

### 通用原则
- 函数短小（≤ 50 行），单一职责
- 类 ≤ 500 行，单一抽象层次
- 圈复杂度 ≤ 10
- 参数 ≤ 5 个（超过用数据类封装）

### 命名
- 变量：`snake_case`（描述性，避免缩写）
- 常量：`UPPER_SNAKE_CASE`
- 类名：`PascalCase`
- 布尔变量：`is_`, `has_`, `should_` 前缀

### 注释
- 注释解释 WHY，不是 WHAT
- 不要注释显而易见的代码
- TODO/FIXME/HACK 必须带 issue 链接或负责人

## 二、错误处理

### 异常层次
```
AppException（基类）
├── ValidationError
├── NotFoundError
├── AuthenticationError
├── AuthorizationError
└── ExternalServiceError
```

### 规则
- 禁止裸 `except:`
- 禁止 `except: pass`
- 在边界层捕获并转换为业务异常
- 重试策略：exponential backoff，最多 3 次

## 三、依赖管理

- 显式依赖注入（FastAPI Depends）
- 禁止循环依赖
- 第三方库用稳定版本（`>=x.y, <x+1.0`）
- 新增依赖前检查许可证和安全漏洞

## 四、代码复用

- DRY 三次原则：出现 3 次再抽象
- 组合优于继承
- 工具函数放 `utils/`，不放全局

## 五、复杂度管理

### 复杂度阈值
| 度量 | 阈值 | 工具 |
|------|------|------|
| 圈复杂度 | ≤ 10 | ruff (C901) |
| 认知复杂度 | ≤ 15 | SonarQube |
| 函数行数 | ≤ 50 | ruff (PLR0911) |
| 参数数量 | ≤ 5 | ruff (PLR0913) |

### 重构时机
- 函数超过 50 行
- 类超过 500 行
- 同一文件被多个 PR 频繁修改
- 测试需要大量 mock
