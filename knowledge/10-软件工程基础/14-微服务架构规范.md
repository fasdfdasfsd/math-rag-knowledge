# 微服务架构规范

## 一、服务拆分原则

### 何时拆分
- 独立的业务能力（Bounded Context）
- 独立的发布节奏
- 独立的扩展需求
- 独立的技术栈需求

### 何时不拆
- 强事务一致性需求
- 共享大量数据
- 功能简单且变化少
- 团队规模小（< 10 人）

## 二、服务通信

### 同步通信（REST / gRPC）
- 查询场景用 REST（GET）
- 命令场景用 POST / PUT / DELETE
- 内部服务间用 gRPC（性能敏感）

### 异步通信（消息队列）
- 领域事件用消息队列
- 至少一次投递保证
- 消费者幂等

### 禁止
- 共享数据库（每个服务独立 database）
- 同步调用链超过 3 层
- 分布式事务（用 Saga 替代）

## 三、数据一致性

### Saga 模式
- 编排型 Saga：协调器指挥各步骤
- 编排失败 → 补偿事务回滚
- 每个步骤记录操作日志

### 最终一致性
- 接受短暂的数据不一致
- 通过领域事件异步同步
- 提供对账机制

## 四、服务治理

### API 网关
- 统一认证、限流、日志
- 路由规则配置化管理
- 禁止在网关做业务逻辑

### 服务发现
- 服务注册 → 健康检查 → 自动摘除
- 客户端负载均衡

### 容错
- 熔断器：连续失败 N 次 → 熔断 → 半开 → 恢复
- 舱壁隔离：独立线程池
- 重试：exponential backoff + jitter，最多 3 次

## 五、可观测性

- 每个服务独立的 metrics + traces
- 分布式追踪（OpenTelemetry）
- 统一日志格式（structlog + correlation_id）
- 健康检查端点：`/health`（liveness）+ `/ready`（readiness）

## 六、部署

- 每个服务独立 CI/CD 流水线
- 容器化部署（Docker + K8s）
- 蓝绿部署或金丝雀发布
- 每个服务有回滚方案
