# 文档规范

## 概述

文档即代码（Docs as Code）——文档与源代码一样被视为一等公民，纳入版本管理、代码评审和 CI/CD 流程。本规范覆盖 README、API 文档、架构文档、注释原则及文档工具链。

---

## 核心要点

### README 模板

```
项目名称
========
  简洁的一句话描述项目定位与价值。

简介
-----
  2-3 段项目背景、核心功能、适用场景。

快速开始
--------
  # 前置依赖
  - Python 3.11+
  - PostgreSQL 14+

  # 安装
  git clone https://github.com/org/project.git
  cd project
  pip install -r requirements.txt

  # 运行
  cp .env.example .env
  python manage.py runserver

文档
-----
  链接到在线文档目录或 `/docs` 文件夹。

贡献指南
--------
  如何报告 Bug、提交 PR、代码规范链接。

许可证
------
  MIT License — 详见 LICENSE 文件。
```

**原则**：
- README 写完，新成员应能在 **15 分钟** 内完成环境搭建
- 保持简短，详细内容链接到 `/docs` 目录
- 随版本更新，README 中的示例代码需可执行

### API 文档（OpenAPI / Swagger）

**推荐工具链**：

```yaml
# openapi.yaml — 示例
openapi: 3.1.0
info:
  title: User Service API
  version: 1.0.0
  description: 用户管理微服务接口
paths:
  /users/{id}:
    get:
      summary: 获取用户信息
      parameters:
        - name: id
          in: path
          required: true
          schema:
            type: integer
      responses:
        '200':
          description: 用户信息
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/User'
```

**规范**：
1. API 文档与代码在同一仓库，随代码变更
2. Python 推荐 `FastAPI`（自动生成 OpenAPI）或 `drf-spectacular`（DRF）
3. 每次 CI 构建验证 OpenAPI spec 格式正确
4. 所有响应需定义 `200`、`4xx`、`5xx` 三种状态码

### 架构文档（C4 + ADR）

**C4 模型**（Context → Container → Component → Code）：

| 层级 | 视图 | 受众 | 说明 |
|------|------|------|------|
| Level 1 | 系统上下文图（System Context） | 所有人 | 系统整体定位与外部依赖 |
| Level 2 | 容器图（Container） | 技术团队 | 进程/服务划分 |
| Level 3 | 组件图（Component） | 开发团队 | 模块内部结构 |
| Level 4 | 代码图（Code） | 开发人员 | 类图/ER 图 |

**ADR（Architecture Decision Record）**：

```markdown
# ADR-001: 使用 PostgreSQL 作为主数据库

## 上下文
需要选择应用主数据库，候选：PostgreSQL、MySQL、MongoDB。

## 决策
采用 PostgreSQL 14+。

## 理由
- 团队已有 PostgreSQL 运维经验
- 需要复杂事务支持 + JSONB 半结构化能力
- 强一致性优于 MongoDB 最终一致性

## 后果
- 正向：ACID 事务保证，成熟生态
- 负向：水平扩展比 MongoDB 复杂，需额外中间件

## 状态
已接受 | 已废弃 | 提议中
```

**ADR 管理**：
- 每份 ADR 一个文件，存放在 `docs/adr/` 目录
- 文件名格式：`NNNN-title-with-dashes.md`
- 涵盖上下文、决策、理由、后果、状态

### 代码注释原则

**核心信条**：**Why > What**

```python
# 好的注释 — 解释为什么这么做
# 使用 PostgreSQL JSONB 而非单独表，因为权限结构为稀疏矩阵
# 90% 用户只有 1-2 个权限，JSONB 可减少 70% 的 JOIN
permissions = models.JSONField(default=dict)

# 差的注释 — 重复代码在做什么
# 从数据库中获取用户列表
users = User.objects.all()
```

**注释规范**：
- **公共接口**：必须写 docstring（参数、返回值、异常）
- **复杂逻辑**：写 Why 而非 What
- **临时修复**：加 `TODO:` 或 `FIXME:` 标签及日期
- **禁止注释**：`# 遍历列表` 这类显而易见的描述

### 文档即代码（Docs as Code）

**工具链**：

```
源代码仓库
├── README.md                # 项目首页
├── docs/
│   ├── adr/                 # 架构决策记录
│   ├── architecture.md      # C4 架构文档
│   ├── api/                 # OpenAPI 规范
│   │   └── openapi.yaml
│   └── guides/              # 用户/运维指南
├── CHANGELOG.md             # 变更日志
├── CONTRIBUTING.md          # 贡献指南
└── mkdocs.yml               # 文档站点配置
```

**原则**：
- Markdown 作为主要文档格式
- 文档纳入代码评审（Review Docs like Code）
- CI 中检查文档链接有效性（`markdown-link-check`）
- 使用 `MkDocs` / `Read the Docs` 生成文档站点
