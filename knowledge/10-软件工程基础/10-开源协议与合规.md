# 开源协议与合规

## 概述

开源协议（Open Source License）决定了代码的使用、修改和分发条件。选错协议或忽视依赖的许可证可能导致法律风险。本规范介绍主流开源协议、选择决策树、依赖检查工具以及开源项目发布 checklist。

---

## 核心要点

### 主流开源协议

| 协议 | 类型 | 允许商用 | 允许修改 | 需要披露 | 需同协议分发 | 说明 |
|------|------|----------|----------|----------|-------------|------|
| MIT | 宽松 | 是 | 是 | 是（保留版权声明） | 否 | 最宽松，广泛使用 |
| Apache 2.0 | 宽松 | 是 | 是 | 是（保留声明 + 修改说明） | 否 | 含专利授权条款 |
| BSD 3-Clause | 宽松 | 是 | 是 | 是（保留版权声明） | 否 | 与 MIT 类似，禁止背书 |
| LGPL | 弱著佐权 | 是 | 是 | 是 | 仅修改部分需 LGPL | 库项目常用 |
| MPL 2.0 | 弱著佐权 | 是 | 是 | 是（修改文件需公开） | 仅修改文件 | Mozilla 生态 |
| GPL 3.0 | 强著佐权 | 是 | 是 | 是 | 是（整个作品需 GPL） | 开源运动核心协议 |
| AGPL 3.0 | 强著佐权 | 是 | 是 | 是 | 是（网络使用也算分发） | 适用于 SaaS 场景 |

### 协议选择决策树

```
项目对外提供服务（SaaS）？  ──否──→ 商业分发二进制或嵌入？
    是                                      │
     ↓                                      ├──是──→ 希望保护生态？──是──→ GPL
选择 AGPL（防闭源）或 MIT/Apache（欢迎商用）    │                    └──否──→ MIT/Apache
                                           │
                                           └──否（纯内部使用）──→ 任意协议均可
                                                     │
                                                     ↓
                                             推荐 MIT 或 Apache 2.0
```

**通用建议**：
- **个人/小项目** → MIT（零束缚）
- **企业级/需要专利保护** → Apache 2.0
- **库/框架** → MIT 或 Apache 2.0
- **希望改造版必须开源** → GPL 3.0
- **云服务防闭源** → AGPL 3.0
- **项目属于已有基金会** → 遵循基金会要求（如 Apache 基金会项目需 Apache 2.0）

### 依赖许可证检查

**Python 依赖检查**：

```bash
# 安装 pip-licenses
pip install pip-licenses

# 查看所有依赖的许可证
pip-licenses --format=markdown

# 检查兼容性（拒绝未知和 GPL 类许可证）
pip-licenses --allow-only="MIT License;Apache Software License;BSD License;Python Software Foundation License"

# 输出到文件
pip-licenses --format=json --output-file=licenses.json

# 结合安全扫描
pip-audit
```

**许可兼容性矩阵**：

| 你的协议 | 可包含的依赖协议 |
|----------|-----------------|
| MIT | MIT / Apache 2.0 / BSD / LGPL / MPL / GPL（条件下） |
| Apache 2.0 | MIT / Apache 2.0 / BSD / LGPL / MPL |
| GPL 3.0 | MIT / Apache 2.0（GPLv3 compatible）/ BSD / GPL |
| AGPL | MIT / Apache 2.0（需确认）/ GPL / AGPL |

### 合规注意事项

| 风险场景 | 说明 | 应对措施 |
|----------|------|----------|
| GPL 传染性 | GPL 代码可能要求整个项目开源 | 动态链接豁免？隔离微服务？选择不同协议库 |
| 专利条款 | Apache 2.0 / GPL 3.0 含明示专利授权 | 确认企业专利组合无冲突 |
| 商标使用 | 协议允许代码使用但不允许商标 | 不将项目名与依赖商标混淆 |
| 署名要求 | 所有宽松协议要求保留版权声明 | README 或 NOTICE 文件中声明 |
| 出口管制 | 部分国家/地区的加密软件限制 | 确认合规（EAR / 中国密码法）|
| 许可证版本 | 部分协议标记"或更高版本"（GPL-2.0-or-later） | 明确指定版本或接受最新版 |

### 开源项目发布 Checklist

- [ ] 选择一个开源协议并创建 `LICENSE` 文件
- [ ] 在 `README.md` 顶部标注协议标识（Badge）
- [ ] 确保 `NOTICE` 或 `LICENSE` 文件包含所有依赖的版权声明
- [ ] 删除代码中所有硬编码的密钥、Token、密码
- [ ] 配置 `.gitignore` 排除 IDE 文件、构建产物、`__pycache__`
- [ ] 检查 `requirements.txt` 中所有依赖的许可证兼容性
- [ ] 配置 CI 门禁：`pip-licenses --allow-only=兼容协议列表`
- [ ] 添加 `CONTRIBUTING.md` 协议条款要求（如 DCO / CLA）
- [ ] 确保没有引用未授权的商标或专利名称
- [ ] 配置 `SECURITY.md` 说明漏洞报告流程

## 实际应用

### 协议选择速查

| 你的需求 | 推荐协议 |
|----------|----------|
| "让别人随便用，改，卖" | MIT |
| "要一个专利授权条款" | Apache 2.0 |
| "不可接受专利诉讼" | Apache 2.0（含终止条款）|
| "修改版必须开源" | GPL 3.0 |
| "云服务也必须开源" | AGPL 3.0 |
| "只想保护修改的文件" | MPL 2.0 / LGPL |
| "不确定，我只是个库" | MIT |
