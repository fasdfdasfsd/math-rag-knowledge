# 技术债务管理

## 一、债务分类

| 类型 | 示例 | 处理优先级 |
|------|------|-----------|
| 架构债务 | 选错框架、耦合过紧 | 🔴 高 |
| 设计债务 | 违反 SOLID、代码异味 | 🟡 中 |
| 测试债务 | 覆盖率不足、测试不稳定 | 🟡 中 |
| 文档债务 | 过时的文档、缺失的 API 文档 | 🟢 低 |
| 依赖债务 | 过期的库、已知漏洞 | 🔴 高（安全相关） |

## 二、债务识别

### 代码异味清单
- 过长函数（> 50 行）
- 过大的类（> 500 行）
- 重复代码（> 20 行相似）
- 过多参数（> 5 个）
- Feature Envy（方法过多使用其他类的数据）
- 发散式变化（一个类因不同原因被修改）

### 检测工具
- ruff（复杂度、代码异味）
- mypy --strict（类型债务）
- pytest --cov（测试债务）
- SonarQube（综合质量度量）

## 三、偿还策略

### 童子军规则
每次修改文件时，使其比原来更干净一点。

### 偿还节奏
- 每个 Sprint 预留 10-20% 时间偿还债务
- 安全债务 → 立即处理
- 架构债务 → 纳入季度规划

### 债务登记
```
| ID | 类型 | 位置 | 描述 | 影响 | 预估工作量 | 状态 |
|----|------|------|------|------|-----------|------|
| TD-001 | 设计 | src/services/user.py | God Class | 修改风险高 | 3d | Open |
```

## 四、预防措施

- 代码审查中标记潜在债务
- CI 质量门禁阻止新增债务
- 重构前先补测试
- 新功能设计时评估长期维护成本
