# 容器化与部署规范

## 概述

本规范定义 Docker 镜像构建、docker-compose 开发环境配置、Kubernetes 资源部署以及镜像安全扫描的最佳实践，确保容器化应用的构建效率、运行稳定性和安全性。

---

## 核心要点

### Dockerfile 最佳实践

**多阶段构建（Multi-stage Build）**：

```dockerfile
# Stage 1: 构建阶段
FROM python:3.11-slim AS builder
WORKDIR /app
COPY requirements.txt .
RUN pip install --user --no-cache-dirs -r requirements.txt

# Stage 2: 运行阶段
FROM python:3.11-slim
RUN groupadd -r appuser && useradd -r -g appuser appuser
WORKDIR /app
COPY --from=builder /root/.local /root/.local
COPY src/ .
USER appuser
EXPOSE 8000
CMD ["python", "main.py"]
```

**原则**：
- 使用 **最小基础镜像**：`python:3.11-slim` 而非 `python:3.11`
- 使用 **非 root 用户** 运行应用（创建专用用户）
- 利用 **Docker 缓存**：将 `COPY requirements.txt` 放在 `COPY src/` 之前
- 合并 `RUN` 命令减少层数，但不过度牺牲可读性
- **不要**在镜像中保存密钥、Token 或 SSH Key

**docker-compose 开发环境**：

```yaml
version: "3.9"
services:
  app:
    build:
      context: .
      dockerfile: Dockerfile.dev
    ports:
      - "8000:8000"
    volumes:
      - ./src:/app/src    # 热重载挂载
    environment:
      - DATABASE_URL=postgresql://user:pass@db:5432/app
      - DEBUG=true
    depends_on:
      db:
        condition: service_healthy

  db:
    image: postgres:14-alpine
    ports:
      - "5432:5432"
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready"]
      interval: 5s
      timeout: 3s

volumes:
  pgdata:
```

**开发环境容器要求**：
- 支持代码热重载（源码挂载 + watch mode）
- 包含测试数据库、消息队列等依赖
- docker-compose 一键启动

### Kubernetes 资源规范

**Deployment 配置示例**：

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: user-service
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1
      maxSurge: 1
  template:
    spec:
      containers:
        - name: app
          image: registry.example.com/user-service:v1.2.0
          resources:
            requests:
              cpu: 100m
              memory: 256Mi
            limits:
              cpu: 500m
              memory: 512Mi
          livenessProbe:
            httpGet:
              path: /healthz
              port: 8000
            initialDelaySeconds: 10
            periodSeconds: 10
          readinessProbe:
            httpGet:
              path: /ready
              port: 8000
            initialDelaySeconds: 5
            periodSeconds: 5
```

| 字段 | 建议值 | 说明 |
|------|--------|------|
| `resources.requests` | 实际使用的 80% | 确保调度公平 |
| `resources.limits` | requests 的 1.5-2 倍 | 防止内存泄漏拖垮节点 |
| `livenessProbe` | `/healthz` 端点 | 检查进程是否存活 |
| `readinessProbe` | `/ready` 端点 | 检查是否可以接收流量 |
| `replicas` | 生产 ≥ 2 | 保证 SLA |
| `rollingUpdate.maxUnavailable` | 1 | 最小不可用数 |
| `strategy` | RollingUpdate | 滚动更新避免停机 |

**Pod 反亲和**（高可用）：
```yaml
affinity:
  podAntiAffinity:
    preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 100
        podAffinityTerm:
          labelSelector:
            matchLabels:
              app: user-service
          topologyKey: kubernetes.io/hostname
```

### 镜像安全扫描

**扫描流程**：

```
CI Pipeline:
  Build Image → 镜像安全扫描 → 漏洞分析 → 通过→推送到 Registry
                                        ↓ 不通过
                                   阻断构建，通知安全团队
```

**工具推荐**：
| 工具 | 类型 | 特点 |
|------|------|------|
| Trivy | CLI/SaaS | 开源，全面，支持多语言依赖扫描 |
| Snyk | SaaS | SaaS 服务，深度漏洞库 |
| Docker Scout | CLI | Docker 官方，与 Docker Hub 集成 |
| Grype | CLI | 轻量开源，与 Syft 配合使用 |
| Harbor | Registry | 集成式镜像仓库 + 扫描 |

**门禁标准**：
- **禁止**：存在 Critical 或 High 级别漏洞的镜像进入生产
- **警告**：存在 Medium 漏洞，需记录并在下个迭代修复
- **例外**：无补丁的漏洞可申请豁免（提供风险缓释方案）

## 实际应用

### 镜像标签规范

```
格式：<项目名>/<服务名>:<版本号>[-<环境>]
示例：myapp/user-service:v1.2.0-staging
```

- 避免使用 `latest` 标签（不可追溯）
- 使用 Git commit SHA 或 SemVer 作为标签
