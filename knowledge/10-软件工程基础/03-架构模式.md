# 架构模式

## 概述

架构模式是软件系统的高层组织方式，决定了系统的模块划分、通信方式和演化能力。本文件覆盖从经典模式到现代架构，并提供模式选择决策树。

---

## 核心要点

### MVC / MVP / MVVM

| 模式 | 全称 | 数据流向 | 使用框架 | 适用场景 |
|------|------|----------|----------|----------|
| MVC | Model-View-Controller | View → Controller → Model | Django, Rails | 传统 Web 应用 |
| MVP | Model-View-Presenter | View ↔ Presenter → Model | 桌面应用 | 可测试性要求高的 UI |
| MVVM | Model-View-ViewModel | View ↔ ViewModel → Model (双向绑定) | WPF, Vue, React | 数据驱动 UI |

**核心原则**：
- Model 不依赖 View，避免 UI 逻辑污染业务逻辑
- Controller/Presenter/ViewModel 负责协调，应保持薄层
- 遇复杂 UI 逻辑优先考虑 MVVM，纯展示页面用 MVC 即可

### 分层架构（Layered Architecture）

```
表现层（Presentation Layer）    ← HTTP / CLI / gRPC 接口
    ↓
业务层（Business Layer）        ← 业务规则与领域逻辑
    ↓
持久层（Persistence Layer）     ← 数据存储与访问
```

- **严格分层**：上层只依赖相邻下层，跨层访问需通过中间层
- **松散分层**：允许上层访问非相邻下层（如 Controller 直接使用 Repository）
- 推荐：核心业务逻辑与框架代码解耦，避免在 Controller 中写业务逻辑

### 六边形架构（Hexagonal Architecture / Ports & Adapters）

```
        [适配器]              [适配器]
      (Web Controller)     (REST Client)
            ↓                    ↑
       ┌─────────────────────────┐
       │       应用/领域层        │
       │   (核心业务逻辑)         │
       └─────────────────────────┘
            ↑                    ↓
        [端口接口]            [端口接口]
      (Repository)          (Event Publisher)
            ↓                    ↑
      [数据库适配器]        [消息队列适配器]
```

- **核心思想**：应用核心（Domain）不依赖任何外部基础设施
- **端口（Port）**：领域层定义的接口
- **适配器（Adapter）**：实现端口的外部系统连接器
- 优势：测试可 mock 端口，无需启动数据库/网络

### CQRS（Command Query Responsibility Segregation）

```
Command Path:  POST /orders → Command Handler → Write DB (RDB)
Query Path:    GET /orders  → Query Handler  → Read DB (Read Model)
```

- **命令（Command）**：改变状态，不返回值（void），使用写模型
- **查询（Query）**：返回值，不改变状态，使用读模型
- 适用场景：读/写负载差异大、复杂查询与事务写入分离
- 不需单独数据库时可用**逻辑 CQRS**（同一数据库，不同接口模型）

### 事件驱动架构（Event-Driven Architecture）

| 组件 | 职责 |
|------|------|
| Event Producer | 产生事件并发布到消息总线 |
| Event Channel | 消息队列 / 事件总线（Kafka / RabbitMQ / SQS） |
| Event Consumer | 订阅并处理事件 |

**模式变体**：
- **事件通知**：简单通知，消费者自行拉取数据
- **事件携带状态传输**：事件中携带完整数据
- **事件溯源（Event Sourcing）**：以事件序列作为事实来源

### 微服务 vs 模块化单体

| 维度 | 微服务 | 模块化单体 |
|------|--------|-----------|
| 部署 | 独立部署 | 单进程部署 |
| 通信 | 网络调用（REST/gRPC） | 进程内方法调用 |
| 数据 | 每个服务独立数据库 | 共享数据库 |
| 复杂度 | 高（分布式陷阱） | 中 |
| 适合团队 | 5+ 个独立团队 | 1-3 个团队 |
| 演化成本 | 基础设施投入大 | 重构成本较低 |

**建议**：优先从**模块化单体**开始，当出现明确拆分需求（独立扩展、独立部署、团队自治）时再演进为微服务。

## 实际应用

### 模式选择决策树

```
需要复杂 UI 交互？ ──是──→ MVVM
    否
需要严格业务域隔离？ ──是──→ 六边形架构
    否
读写负载差异大？ ──是──→ 加入 CQRS
    否
需要异步解耦？ ──是──→ 事件驱动架构
    否
团队规模 > 5 个独立团队？ ──是──→ 微服务
    否
                     ──→ 分层架构 + 模块化单体
```
