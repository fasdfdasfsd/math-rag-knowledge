# 领域驱动设计（DDD）

## 一、战略设计

### 限界上下文（Bounded Context）
- 每个上下文拥有独立的数据和通用语言
- 上下文之间通过明确定义的接口通信
- 上下文映射：Partnership / Customer-Supplier / Conformist / ACL / Open Host

### 通用语言（Ubiquitous Language）
- 每个限界上下文定义自己的术语表
- 术语在代码、API、文档中保持一致
- 示例：用户上下文叫 `User`，订单上下文叫 `Buyer`

### 核心域识别
- Core Domain：竞争优势所在，最高投入
- Supporting Domain：支撑业务，可自建或购买
- Generic Domain：通用能力，优先使用现成方案

## 二、战术设计

### 聚合（Aggregate）
- 一个事务只修改一个聚合
- 聚合根保证内部不变量
- 小聚合优先（通过 ID 引用外部聚合，而非对象引用）

### 实体 vs 值对象
| 维度 | 实体 | 值对象 |
|------|------|--------|
| 标识 | 有唯一 ID | 无 ID，属性相等即相等 |
| 可变性 | 可变 | 不可变 |
| 示例 | User, Order | Money, Address, Email |

### 领域事件
- 跨上下文通信使用领域事件
- 事件命名：`{聚合}{动作}Event`（如 `OrderPlacedEvent`）
- 事件包含：事件 ID、时间戳、聚合 ID、事件数据

## 三、架构模式

### 六边形架构（端口与适配器）
```
Domain（领域逻辑，零框架依赖）
  ↑↓ 端口（接口定义）
Application（用例编排，事务管理）
  ↑↓ 适配器（具体实现）
Infrastructure（DB、HTTP、消息队列）
```

### CQRS（读写分离）
- 写模型：聚合 + 领域事件
- 读模型：优化的查询视图
- 适用场景：读写模式差异大、查询性能要求高

## 四、反模式

- ❌ 贫血模型：实体只有 getter/setter，无业务行为
- ❌ 跨上下文共享数据模型
- ❌ 巨型聚合（应拆分为小聚合）
- ❌ 领域层依赖框架或基础设施
