# 设计模式规范

## 概述

基于 GoF 23 种经典设计模式与企业集成模式，提供模式分类、适用场景及 Python 实现要点。设计模式不是万能药，滥用模式反而增加复杂度——应在代码中出现重复问题时引入模式，而非预先引入。

---

## 核心要点

### 创建型模式（5 种）

| 模式 | 意图 | Python 实现要点 | 适用场景 |
|------|------|----------------|----------|
| 单例（Singleton） | 确保类只有一个实例 | 使用 `__new__` 或模块级变量（Python 模块天然单例）| 配置管理、日志记录器、数据库连接池 |
| 工厂方法（Factory Method） | 定义创建对象的接口，让子类决定实例化哪个类 | 使用类方法或独立工厂函数 | 多态创建，客户端无需知道具体类 |
| 抽象工厂（Abstract Factory） | 创建相关对象族 | 组合多个工厂方法，抽象基类定义接口 | 跨平台 UI 组件、多个产品族 |
| 建造者（Builder） | 分步构建复杂对象 | 使用流式接口（Fluent Interface），返回 `self` | 复杂对象构造（如 HTTP 请求组装）|
| 原型（Prototype） | 通过复制现有对象创建新实例 | 使用 `copy.deepcopy` 注意深拷贝 | 创建成本高的对象（如大配置对象） |

**Python 注意**：`functools.lru_cache` 或模块单例比手写 Singleton 更 Pythonic；使用 `dataclass` 配合 `copy` 可实现轻量原型。

### 结构型模式（7 种）

| 模式 | 意图 | Python 实现要点 | 适用场景 |
|------|------|----------------|----------|
| 适配器（Adapter） | 让接口不兼容的类协作 | 类适配用继承，对象适配用组合 | 集成第三方库、新旧系统对接 |
| 桥接（Bridge） | 将抽象与实现解耦 | 抽象持有一组实现接口的引用 | 多维度变化的场景（通知方式 × 消息渠道）|
| 组合（Composite） | 将对象组织成树形结构 | 使用 ABC 定义组件接口，叶子和容器统一处理 | 树形菜单、文件系统、组织架构 |
| 装饰（Decorator） | 动态为对象添加职责 | Python 内置 `@decorator` 语法支持 | 日志、权限校验、缓存、数据校验 |
| 外观（Facade） | 为子系统提供统一接口 | 封装复杂子系统的简化 API | 简化第三方库调用、降低子系统耦合 |
| 享元（Flyweight） | 通过共享减少创建开销 | 使用对象池或 `__slots__` 减少内存 | 大量细粒度对象（如字处理字符）|
| 代理（Proxy） | 为对象提供替身以控制访问 | 实现与被代理类相同的接口 | 延迟加载、访问控制、远程代理 |

### 行为型模式（11 种）

**常用模式**：

| 模式 | 意图 | Python 实现 |
|------|------|-------------|
| 观察者（Observer） | 一对多依赖通知 | `Observable` + 回调列表 / `asyncio.Event` |
| 策略（Strategy） | 定义可互换的算法族 | 函数/类作为参数传入，或策略注册表 |
| 命令（Command） | 将请求封装为对象 | 可调用对象（`__call__`）配合队列 |
| 状态（State） | 对象内部状态改变行为 | 状态机模式，状态类实现统一接口 |
| 模板方法（Template Method） | 定义算法骨架，子类实现步骤 | 基类定义方法流程，子类覆写钩子方法 |

**其他模式**：迭代器（Iterator，Python 内置）、访问者（Visitor）、中介者（Mediator）、备忘录（Memento）、解释器（Interpreter）、职责链（Chain of Responsibility）。

### 企业集成模式（重点）

| 模式 | 意图 | Python 实现 |
|------|------|-------------|
| Repository | 封装数据访问逻辑，提供集合风格的接口 | 定义 `UserRepository` 类，封装 ORM 操作 |
| Unit of Work | 追踪对象变更，批量提交 | SQLAlchemy `Session` 即 Unit of Work 实现 |
| Specification | 将业务规则封装为可组合的对象 | 定义 `Specification` 基类，支持 `&`、`|` 运算 |
| Service Layer | 封装应用逻辑和事务边界 | 分层架构中的 Service 层 |
| Data Mapper | 对象与数据库表之间的双向映射 | SQLAlchemy ORM + `declarative_base` |
| Active Record | 对象同时负责数据和数据库操作 | Django ORM 的 Model 即 Active Record |

## 实际应用

### 模式选择原则

1. **先组合后继承**（Favor composition over inheritance）
2. **针对接口编程**（Program to an interface, not an implementation）
3. **发现变化后引入模式**，而非预先滥用
4. **Python 中函数是一等公民**——部分模式可用函数替代（如策略）

### Python 示例：策略模式

```python
from typing import Protocol

class PaymentStrategy(Protocol):
    def pay(self, amount: float) -> bool: ...

def alipay_pay(amount: float) -> bool:
    print(f"支付宝支付 {amount} 元")
    return True

def wechat_pay(amount: float) -> bool:
    print(f"微信支付 {amount} 元")
    return True

def execute_payment(strategy: PaymentStrategy, amount: float) -> None:
    strategy.pay(amount)
```
