# 日志与错误处理规范

## 概述

规范化的日志记录和错误处理是系统可观测性和稳定性的基础。本规范定义日志级别使用原则、结构化日志格式、敏感信息保护要求、异常处理模式以及 Python 实现示例。

---

## 核心要点

### 日志级别使用

| 级别 | 数值 | 使用场景 | 示例 |
|------|------|----------|------|
| DEBUG | 10 | 开发调试，定位问题 | SQL 执行语句、函数入参出参 |
| INFO | 20 | 正常业务流程记录 | 用户登录、订单创建、服务启停 |
| WARNING | 30 | 潜在问题，不影响运行 | API 响应慢（>1s）、配置缺失使用默认值 |
| ERROR | 40 | 功能异常，需人工介入 | 数据库连接失败、支付回调异常 |
| CRITICAL | 50 | 系统不可用，立即响应 | 磁盘写满、NPE 导致服务崩溃 |

**使用原则**：
- 生产环境默认日志级别 **INFO**，调试时动态调整为 DEBUG
- ERROR 级日志应触发告警，CRITICAL 级日志应触发紧急响应
- 不要在 `for` 循环体内使用 DEBUG 以上级别日志（影响性能）
- 异常处理中避免 `logger.error` + `raise` 双层记录

### 结构化日志（JSON 格式）

**推荐格式**：

```json
{
  "timestamp": "2026-05-30T10:30:00.123Z",
  "level": "INFO",
  "logger": "app.user.service",
  "message": "User login successful",
  "trace_id": "abc-123-def-456",
  "span_id": "span-001",
  "user_id": "u_10086",
  "duration_ms": 45,
  "extra": {
    "source_ip": "10.0.1.2",
    "login_type": "password"
  }
}
```

**核心字段**：
- `timestamp` — ISO 8601 格式，含时区
- `level` — 日志级别
- `logger` — 记录器名称（模块路径）
- `message` — 人类可读的描述
- `trace_id` — 链路追踪 ID（跨服务传递）
- `span_id` — 当前调用单元 ID

### 日志内容规范

**禁止记录的信息**（脱敏后才能输出）：
- 密码 / Token / API Key / 密钥
- 身份证号、银行卡号、手机号
- 个人生物识别信息
- 完整的数据库连接字符串
- 会话 Cookie 全文

**脱敏示例**：
```python
# 手机号脱敏
phone = "138****5678"
# 身份证脱敏
id_card = "110101********1234"
# Token 脱敏
token = "sk-****abc"
```

### 异常处理

**异常链（Exception Chaining）**：

```python
try:
    resp = requests.get(url, timeout=5)
    resp.raise_for_status()
except requests.RequestException as exc:
    raise ServiceCallError("第三方支付服务不可用") from exc
```

**自定义异常**：

```python
class AppError(Exception):
    """应用基础异常"""

class PaymentError(AppError):
    """支付相关异常"""

class ValidationError(AppError):
    """参数校验异常"""

class NotFoundError(AppError):
    """资源不存在"""
```

**重试策略**：

| 场景 | 策略 | 条件 |
|------|------|------|
| 网络超时 | 指数退避重试（3 次）| 仅重试可恢复异常 |
| 数据库锁冲突 | 重试 3 次，间隔 100ms | 非死锁且小于竞争时间 |
| 服务限流 | 重试 1 次，等待 Retry-After | 响应码 429 |

**不重试的场景**：
- 400 Bad Request（请求参数错误）
- 401 Unauthorized / 403 Forbidden（权限问题）
- 404 Not Found（资源不存在）
- 422 Unprocessable Entity（数据校验失败）

### Python structlog 配置示例

```python
import structlog
import logging

structlog.configure(
    processors=[
        structlog.stdlib.filter_by_level,
        structlog.stdlib.add_log_level,
        structlog.stdlib.PositionalArgumentsFormatter(),
        structlog.processors.TimeStamper(fmt="iso"),
        structlog.processors.StackInfoRenderer(),
        structlog.processors.format_exc_info,
        structlog.processors.UnicodeDecoder(),
        structlog.dev.ConsoleRenderer() if DEBUG
        else structlog.processors.JSONRenderer(),
    ],
    wrapper_class=structlog.stdlib.BoundLogger,
    context_class=dict,
    logger_factory=structlog.stdlib.LoggerFactory(),
    cache_logger_on_first_use=True,
)

logger = structlog.get_logger("app.user")
logger.info("user.login", user_id="u_10086", ip="10.0.1.2")
```

## 实际应用

### 日志配置规范（生产环境）

```yaml
# 日志配置
handlers:
  - type: rotating_file
    filename: /var/log/app/app.log
    max_size: 100MB
    backup_count: 10
    format: json
  - type: stdout
    format: json  # 容器化环境使用 stdout

# 敏感词过滤
sensitive_fields:
  - password
  - token
  - authorization
  - cookie
  - phone
```
