# 版本管理规范

## 概述

版本管理规范覆盖语义化版本控制（SemVer 2.0）、Changelog 维护、Git 分支策略以及发布与回滚流程。统一的版本管理保证团队成员对发布状态有一致理解，降低发布风险。

---

## 核心要点

### SemVer 2.0（语义化版本控制）

**格式**：`主版本号.次版本号.修订号 [-预发布号] [+构建元数据]`

| 段位 | 递增条件 | 示例 |
|------|----------|------|
| 主版本（MAJOR） | 不兼容的 API 变更 | `2.0.0` |
| 次版本（MINOR） | 向下兼容的功能新增 | `1.3.0` |
| 修订号（PATCH） | 向下兼容的问题修复 | `1.0.1` |

**预发布标识**（可选）：
- `-alpha.1`：内部测试版
- `-beta.2`：公测版
- `-rc.3`：发布候选版

**规则**：
1. 版本号不包含前导零（`01.2.3` 无效）
2. 1.0.0 之后才进入正式版
3. 预发布版本优先级低于正式版（`1.0.0-alpha < 1.0.0`）
4. 一旦发布，版本号内容不可修改

**Python 工具**：
```bash
# 查看当前版本
python -m setuptools_scm
# 使用 bumpversion / bump-my-version 自动递增
pip install bump-my-version
bump-my-version bump minor
```

### Changelog 格式（Keep a Changelog）

**文件位置**：项目根目录 `CHANGELOG.md`

```markdown
# Changelog

## [1.2.0] - 2026-05-30

### Added
- 用户权限管理模块（#123）

### Changed
- 优化订单查询响应时间（#119）

### Deprecated
- 旧版 /v1/api 接口计划下月移除

### Removed
- 已废弃的 LegacyPayment 类

### Fixed
- 修复并发场景下库存超卖问题（#125）

### Security
- 修复 SQL 注入漏洞（CVE-2026-xxxx）
```

**分类说明**：
- `Added`：新功能
- `Changed`：现有功能变更
- `Deprecated`：即将移除的功能
- `Removed`：已移除的功能
- `Fixed`：Bug 修复
- `Security`：安全漏洞修复

### Git 分支策略

| 分支 | 命名规范 | 来源 | 合并到 | 说明 |
|------|----------|------|--------|------|
| main | `main` / `master` | — | — | 生产分支，受保护 |
| develop | `develop` | — | main | 集成分支（可选）|
| feature | `feature/xxx` | develop / main | develop / main | 功能开发 |
| release | `release/x.x.x` | develop | main + develop | 发布准备 |
| hotfix | `hotfix/xxx` | main | main + develop | 紧急修复 |

**推荐策略**：

- **小型团队（< 10 人）** — GitHub Flow：`main` + `feature` 分支，无需 `develop`
- **中型团队（10-30 人）** — Git Flow：`main` + `develop` + `release` + `feature`
- **大型/成熟团队** — 主干开发（Trunk-Based）：每日向主干合并，配合 Feature Toggle

### 发布流程

```
1. 从 develop（或 main）建立 release/x.x.x 分支
2. release 分支上仅修复 Bug，不添加新功能
3. 版本号从 x.x.x-snapshot 变为 x.x.x-rc.1
4. 测试通过后，生成 CHANGELOG + Git Tag
5. 合并到 main 和 develop，打上 vx.x.x tag
6. 触发 CI/CD 构建部署到生产环境
```

### 版本回滚

**原则**：能回退功能（Feature Toggle）优于回退版本

**回退流程**：
```bash
# 1. 确认回退目标版本
git log --oneline -10

# 2. 回滚到指定 tag
git revert <commit-hash> | git reset --hard <tag>
# 优先使用 revert 而非 reset（保留历史）

# 3. 推送并部署
git push origin main
```

- **A/B 回滚**：金丝雀发布 → 发现问题 → 切换流量到旧版本
- **数据库回滚**：回滚版本需附带数据库回滚脚本
- **回滚后复盘**：5 Why 分析 + 改进 CI/CD 门禁
